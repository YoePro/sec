Registers and Memory-Mapped Hardware

A register type describes a fixed-width bit layout. It is intended for memory-mapped hardware, embedded systems, device controllers, communication protocols, sensors and other code where individual bits have defined meanings. 

type MotorControl register[8] {
    Speed: bit[4]
    Enabled: bit
    _: bit[3]
}

The compiler validates the complete layout and generates the masks, shifts and hardware accesses required to read and update individual fields.

Rule: A register declaration describes representation. An @address binding connects that representation to external hardware storage.

Register Declaration

A register is declared with type Name register[N]:

type Protocol register[8] {
    Speed: bit[4]
    Enabled: bit
    _: bit[3]
}

The value inside register[...] is the total width in bits. It must be a positive compile-time integer. 

type Status register[16] {
    Ready: bit
    Error: bit
    Code: bit[6]
    _: bit[8]
}
Rule: The combined width of all fields must exactly equal the declared register width.

Bit Fields

bit declares a one-bit field:

Ready: bit

It has the same width as:

Ready: bit[1]

A multi-bit field uses bit[N]:

Mode: bit[3]
Counter: bit[12]

The width must be a positive compile-time integer. Zero-width and negative-width fields are invalid. 

Value: bit[0]
Value: bit[-1]
Invalid: Every bit field must occupy at least one bit.

A bit field is not an independent integer variable. Its value is extracted from or inserted into the containing register by the compiler.

Reserved Bits

The identifier _ declares unnamed or reserved bits:

type Status register[8] {
    Ready: bit
    Error: bit
    _: bit[6]
}

Reserved bits contribute to the layout but do not create an accessible field.

let value := status._
Invalid: Reserved fields cannot be read or written.

Several reserved regions may appear in one declaration:

type Control register[16] {
    Enabled: bit
    _: bit[3]
    Mode: bit[4]
    _: bit[8]
}

Named-field updates preserve reserved bits unless a later register rule explicitly defines special write behavior.

Layout Validation

The field widths must occupy the complete register.

type InvalidProtocol register[8] {
    Speed: bit[4]
    Enabled: bit
    Mode: bit[2]
}

The fields occupy only seven bits, so the declaration is invalid.

Field Width
Speed4 bits
Enabled1 bit
Mode2 bits
Total7 bits

A valid eight-bit declaration could reserve the remaining bit:

type Protocol register[8] {
    Speed: bit[4]
    Enabled: bit
    Mode: bit[2]
    _: bit
}

Fields may not overflow the declared width either.

Addressed Register Instances

A register instance may be bound to a fixed hardware address with @address: 

@address(0x40021000)
let motorStatus: MotorStatus

The binding refers to existing external storage. It does not allocate ordinary local or heap memory.

An addressed binding has these properties:

  • It has a fixed external address.
  • Every access is volatile.
  • The binding does not own the storage.
  • The target defines the physical address representation.
  • The compiler validates that the declared type is a register type.
Rule: @address requires an explicit register type and a compile-time integer address.

Volatile Semantics

Every @address binding is volatile. No separate volatile keyword is used.

@address(0x40021000)
let status: StatusRegister

let first := status.Ready
let second := status.Ready

The two reads are separate observable hardware accesses. The compiler may not assume they return the same value because hardware may change the register between them.

Writes are also observable. A write may not be removed merely because its result is not later read by Sec code.

Rule: Volatility describes external access behavior. It does not imply that Sec code may write.

Read-Only Addressed Registers

An addressed instance declared with let is read-only from Sec code:

@address(0x40021000)
let motorStatus: MotorStatus

let ready := motorStatus.Ready

The hardware may still change the value at any time.

motorStatus.Ready = true
Invalid: The binding is volatile and externally mutable, but Sec code may not write to it.

Read-Write Addressed Registers

An addressed instance declared with let mut may be read and written:

@address(0x40021000)
let mut motorControl: MotorControl

let enabled := motorControl.Enabled
motorControl.Enabled = true
Declaration Sec access Volatile
@address(...) let value: T Read-only Yes
@address(...) let mut value: T Read-write Yes

Initialization

An addressed binding refers to hardware that already exists. The declaration must not initialize it:

@address(0x40021000)
let mut control: ControlRegister := initialValue
Invalid: Addressed register bindings cannot have initializers.

Hardware writes must be explicit:

@address(0x40021000)
let mut control: ControlRegister

control.Enabled = true

Declaring the binding alone performs no write.

Reading Fields

Named fields are read through ordinary member access:

let ready := motorStatus.Ready
let mode := motorStatus.Mode

The compiler extracts the selected bits and produces the field value. For addressed storage, the extraction begins with a volatile register read.

Reserved fields cannot be read.

Writing Fields

A field may be written when its containing storage is mutable:

motorControl.Enabled = true
motorControl.Mode = 3

The assigned value must be compatible with the field and must fit within its declared width.

A write through a read-only addressed binding is rejected at compile time.

Field Value Ranges

An unsigned bit[N] field stores values from zero through 2^N - 1.

Field type Minimum Maximum
bit01
bit[2]03
bit[4]015
bit[8]0255
 
type MotorControl register[8] {
    Speed: bit[4]
    Enabled: bit
    _: bit[3]
}

motorControl.Speed = 19
Invalid: The value 19 does not fit in a four-bit field whose range is 0..15.

Constant assignments are checked at compile time.

Field Updates

Writing one named field updates only that field:

motorControl.Enabled = true

For an ordinary read-write hardware register, the compiler may implement this as:

  1. Volatile-read the complete register.
  2. Preserve unrelated and reserved bits.
  3. Replace the selected field bits.
  4. Volatile-write the complete register.

The programmer does not need to manually create masks, shifts or read-modify-write sequences.

Rule: Ordinary named-field updates preserve every unrelated bit.

Hardware behaviors such as write-one-to-clear, write-only fields and atomic set registers require additional metadata and are not part of the basic register model.

Units on Register Fields

A multi-bit field may carry a unit:

type MotorProtocol register[8] {
    Speed: bit[4]<rpm>
    Enabled: bit
    _: bit[3]
}

The bit width describes storage. The unit describes meaning.

let speed: rpm := motorProtocol.Speed

An incompatible unit is invalid:

let distance: m := motorProtocol.Speed
Rule: Unit compatibility remains active when register fields are read, assigned or used in expressions.

Scaling, offsets, signed encodings and device-specific conversion formulas require separate declarations.

Ordinary Register Values

A register type may also be used for a normal in-memory value:

let snapshot: MotorStatus := ReadStatus()

The value is not volatile merely because its type is a register type.

Rule: Volatility belongs to the storage access, not to every value of the register type.

Compare:

let snapshot: MotorStatus := ReadStatus()

@address(0x40021001)
let liveStatus: MotorStatus

snapshot is an ordinary value. liveStatus is volatile external storage.

Address Validation

The address supplied to @address must be known at compile time:

@address(0x40021000)
let status: StatusRegister

The selected target determines which address forms and ranges are valid.

A microcontroller target may validate the address against known memory-mapped regions. Another target may use a different hardware address model.

Unchecked dynamic address calculations belong to unsafe code rather than ordinary addressed bindings.

Address Aliasing

Two declarations that refer to the same physical storage may conflict:

@address(0x40021000)
let mut first: RegisterA

@address(0x40021000)
let mut second: RegisterB

Statically known overlapping declarations may be rejected when their layouts or access rules conflict. Explicit alias declarations may be introduced later for intentional hardware views.

Safety

A statically declared register and a validated @address binding provide a typed alternative to raw pointers and manual pointer arithmetic.

Normal addressed register access does not require RawPtr.

  • The compiler knows the complete register width.
  • The compiler knows every field position and width.
  • Read-only and read-write access are explicit.
  • Volatile behavior is automatic for addressed storage.
  • Out-of-range constant assignments are rejected.
  • Reserved bits are not exposed as fields.

Dynamic addresses, unchecked address ranges and unsupported target address forms may require an unsafe context.

Future Extensions

The basic register model may later be extended with:

  • signed bit fields
  • explicit field offsets
  • bit ordering and endianness
  • read-only fields inside mutable registers
  • write-only fields
  • write-one-to-clear and write-zero-to-clear fields
  • atomic set and clear aliases
  • hardware reset values
  • scaled and offset field encodings
  • register arrays
  • dynamic base addresses
  • explicit register aliasing
  • generated declarations from hardware descriptions

These extensions must preserve the core register rules.

Common Errors

Incomplete layout

type Status register[8] {
    Ready: bit
    Error: bit
}

Add reserved bits or change the declared width.

Zero-width field

Value: bit[0]

Every field must occupy at least one bit.

Accessing reserved bits

let reserved := status._

Reserved regions participate only in layout.

Writing through a read-only binding

@address(0x40021000)
let status: Status

status.Ready = true

Use let mut only when Sec code is allowed to write to the hardware register.

Initializing addressed storage

@address(0x40021000)
let mut control: Control := initial

Bind first, then perform an explicit field write.

Out-of-range assignment

control.Mode = 8

A three-bit field accepts values from zero through seven.

Using an ordinary type with @address

@address(0x40021000)
let value: int

@address requires an explicit register type.

Summary of Rules

  • A register type has a fixed compile-time width in bits.
  • All fields must exactly occupy the declared width.
  • bit occupies one bit.
  • bit[N] occupies a positive compile-time number of bits.
  • _ declares reserved bits that cannot be accessed.
  • Named fields must be unique.
  • A register field may carry a unit.
  • @address binds a register type to external hardware storage.
  • An addressed binding requires a compile-time integer address.
  • Every addressed register access is volatile.
  • let creates a read-only addressed binding.
  • let mut creates a read-write addressed binding.
  • Addressed bindings cannot have initializers.
  • Declaring an addressed register performs no hardware write.
  • Named-field updates preserve unrelated and reserved bits.
  • Constant field assignments must fit within the field width.
  • Ordinary non-addressed register values are not implicitly volatile.
  • The compiler handles masks, shifts and volatile accesses.
  • Dynamic or unchecked addressing may require unsafe code.
