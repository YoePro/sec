Representing currency and exact decimals without floating-point issues (such as 0.1 + 0.2 = 0.30000000000000004) is a fantastic addition to a modern language. Your structure, featuring a fixed base (int64) and an exponent/scale (uint8), aligns perfectly with how the BigDecimal or Decimal data type is typically designed. Since LLVM lacks a built-in "decimal type", we must lower it to primitive operations. In LLVM IR, your struct decimal will be represented as an anonymous or named literal struct: { i64, i8 }. Here is a walkthrough of how to implement the mathematical operations in your compiler and how to generate the corresponding LLVM IR.

------------------------------
## Sec implementation status

Current compiler decision:

* `decimal` is represented in LLVM as `%sec.decimal = type { i64, i8 }`.
* Field 0 is the signed integer value.
* Field 1 is the base-10 scale.
* Decimal literals are emitted directly as LLVM aggregate values using `insertvalue`.

Implemented in LLVM code generation:

* `let d: decimal := 3.14` emits `{ i64 314, i8 2 }`.
* `let d: decimal := 1` emits `{ i64 1, i8 0 }` when the target type is decimal.
* `let d := 2.5d` emits a decimal value.
* `decimal(5)` explicitly converts an integer value to decimal scale 0.
* Functions may return `decimal`.

Not implemented yet:

* Decimal arithmetic in LLVM.
* `_sec_decimal_add`, `_sec_decimal_sub`, `_sec_decimal_mul`, `_sec_decimal_div`.
* Decimal overflow checks via LLVM overflow intrinsics.
* Runtime contract failure path for decimal overflow, division by zero, or precision exhaustion.

Compiler rule:

Decimal arithmetic must be lowered to runtime helper calls rather than duplicating large control-flow-heavy IR at every operator site.

------------------------------
## The Mathematical Concept
When two decimal numbers meet, they must first be synchronized to the same scale before they can be added or subtracted. For multiplication and division, the scale changes mathematically.

   1. Addition / Subtraction: Align the decimal point of the number with the lower scale so that both numbers share the same scale. Then, simply add or subtract the number fields.
   2. Multiplication: Multiply number1 by number2. The new scale becomes scale1 + scale2.
   3. Division: This is the most difficult operation, as we must increase the precision of the numerator before dividing to avoid losing decimal places (e.g., 1 / 3).

------------------------------
## 1. LLVM Structure Definition
In your compiler's LLVM code generator, you define the type like this:

; Define the type for sec decimal
%struct.decimal = type { i64, i8 }

------------------------------
## 2. Addition (Example)
Let us assume we want to add 1.25 (scale 2) and 0.5 (scale 1).

* a = { number: 125, scale: 2 }
* b = { number: 5, scale: 1 }

The compiler must generate code that detects which number has the lower scale, calculates the difference, and multiplies that number's number field by $10^{\text{difference}}$.
In this case, b must be scaled up by $10^{(2-1)} = 10^1$.
5 × 10 = 50.
Now we can perform the addition: 125 + 50 = 175.
The resulting scale becomes 2 (which represents 1.75).

------------------------------
## Generated LLVM IR for Addition
Here is the raw LLVM IR code that your compiler needs to emit to perform this scaling and addition dynamically at runtime (when the scales are not known at compile-time):

define %struct.decimal @decimal_add(%struct.decimal %a, %struct.decimal %b) {
entry:
    ; Extract values from struct A
    %a.num = extractvalue %struct.decimal %a, 0
    %a.scale = extractvalue %struct.decimal %a, 1

    ; Extract values from struct B
    %b.num = extractvalue %struct.decimal %b, 0
    %b.scale = extractvalue %struct.decimal %b, 1

    ; Check which scale is larger
    %cmp = icmp ugt i8 %a.scale, %b.scale
    br i1 %cmp, label %scale_b, label %check_scale_a

scale_b:
    ; A has a larger scale, so we must scale up B
    %diff.b = sub i8 %a.scale, %b.scale
    
    ; Compute 10^diff.b (This can be a quick lookup table or a loop)
    %factor.b = call i64 @ipow10(i8 %diff.b)
    
    ; Multiply B's number to match A's scale
    %b.num.scaled = mul i64 %b.num, %factor.b
    
    ; Perform the addition
    %res.num.b = add i64 %a.num, %b.num.scaled
    
    br label %exit

check_scale_a:
    %cmp.a = icmp ugt i8 %b.scale, %a.scale
    br i1 %cmp.a, label %scale_a, label %equal_scales

scale_a:
    ; B has a larger scale, so we must scale up A
    %diff.a = sub i8 %b.scale, %a.scale
    %factor.a = call i64 @ipow10(i8 %diff.a)
    %a.num.scaled = mul i64 %a.num, %factor.a
    %res.num.a = add i64 %a.num.scaled, %b.num
    br label %exit

equal_scales:
    ; Scales are already equal, just add numbers
    %res.num.eq = add i64 %a.num, %b.num
    br label %exit

exit:
    ; Merge the results using PHI nodes depending on which branch we took
    %final.num = phi i64 [ %res.num.b, %scale_b ], [ %res.num.a, %scale_a ], [ %res.num.eq, %equal_scales ]
    %final.scale = phi i8 [ %a.scale, %scale_b ], [ %b.scale, %scale_a ], [ %a.scale, %equal_scales ]

    ; Insert values into a new return struct
    %tmp = insertvalue %struct.decimal undef, i64 %final.num, 0
    %res = insertvalue %struct.decimal %tmp, i8 %final.scale, 1
    ret %struct.decimal %res
}

; Helper function to compute 10^x efficiently
declare i64 @ipow10(i8)

------------------------------
## 3. Multiplication (Easier!)
For multiplication, we do not need to scale the numbers beforehand. We simply multiply the base numbers directly and add their scales together.
1.25 × 0.5 becomes 125 × 5 = 625. The new scale is 2 + 1 = 3. Result: 625 with a scale of 3 (0.625).
## Generated LLVM IR for Multiplication

define %struct.decimal @decimal_mul(%struct.decimal %a, %struct.decimal %b) {
entry:
    %a.num = extractvalue %struct.decimal %a, 0
    %a.scale = extractvalue %struct.decimal %a, 1
    %b.num = extractvalue %struct.decimal %b, 0
    %b.scale = extractvalue %struct.decimal %b, 1

    ; Multiply the core numbers
    %res.num = mul i64 %a.num, %b.num
    
    ; Add the scales
    %res.scale = add i8 %a.scale, %b.scale

    ; Pack into result struct
    %tmp = insertvalue %struct.decimal undef, i64 %res.num, 0
    %res = insertvalue %struct.decimal %tmp, i8 %res.scale, 1
    ret %struct.decimal %res
}

## 4. Strategic Tip for Your Compiler (Sec)
Instead of emitting this massive amount of LLVM IR instructions (with branches, conditions, and phi nodes) every single time a developer writes a + b in Sec, it is far more elegant to use Runtime Helper Functions.
Approach your compiler design like this:

   1. Write the functions _sec_decimal_add, _sec_decimal_sub, _sec_decimal_mul, and _sec_decimal_div in C or directly in Sec (if the language supports raw structs).
   2. Compile these into a small runtime library (rt.bc or libsec_rt.a).
   3. When your Sec compiler encounters c = a + b where the type is decimal, it simply emits a standard function call (LLVM call) to _sec_decimal_add.

This keeps your compiler's LLVM code generator extremely clean and easy to maintain!
------------------------------
## Don't Forget Contract Verification!
Since we are talking about Sec and contracts: remember to add overflow checks (llvm.sadd.with.overflow.i64) inside your decimal functions. If two large decimal numbers are added and the int64 base number overflows, you can immediately trigger a contract failure or a runtime error!

## 5. Division Algorithm & Precision Control
Division is fundamentally different from addition and multiplication because dividing two integers can result in infinite periodic decimals (e.g., $1 / 3 = 0.3333...$). To prevent infinite loops and manage memory, the compiler and runtime must enforce precision boundaries.
## Precision Strategies

   1. User-Defined Precision: If the developer explicitly requests a specific precision (e.g., 4 decimals), the compiler scales the numerator to exactly match that target scale before dividing. For $1 / 3$ with 4 decimals, the result is 0.3333 (Base: 3333, Scale: 4).
   2. Implicit Precision ("As many as fit"): If no precision is specified, the compiler falls back to a maximum safe scale. Since the base is an int64 (max value $\approx 9 \times 10^{18}$), the maximum safe scale for a dividend starting near 1 is 18. The runtime will scale the numerator up as much as possible without causing an integer overflow, perform a truncating integer division, and then strip any trailing zeros to normalize the scale.

------------------------------
## The Mathematical Step-by-Step
To compute $Q = A / B$ where $A = \{\text{num}_A, \text{scale}_A\}$ and $B = \{\text{num}_B, \text{scale}_B\}$:

   1. Calculate Target Scale:
   * If explicit: $\text{target\_scale} = \text{user\_defined\_scale}$.
      * If implicit: $\text{target\_scale} = \text{max\_safe\_scale} \pmod{18}$.
   2. Calculate Required Upscale Factor:
   To achieve the target scale after division, the numerator's internal value must be shifted before the integer division occurs. The formula for the scale alignment is:
   $$\text{shift} = \text{target\_scale} + \text{scale}_B - \text{scale}_A$$ 
   3. Scale the Numerator:
   $$\text{scaled\_num}_A = \text{num}_A \times 10^{\text{shift}}$$ 
   (Note: This step requires an overflow check! If shift is too large, it must cap at the max bits of i64).
   4. Execute Integer Division:
   $$\text{res\_num} = \text{scaled\_num}_A / \text{num}_B$$ 
   5. Normalize (Optional but Recommended):
   If the implicit mode was used, the result might have trailing zeros (e.g., 2.500000). Loop through the result: while $\text{res\_num} \pmod{10} == 0$ and $\text{target\_scale} > 0$, divide $\text{res\_num}$ by 10 and decrement $\text{target\_scale}$.

------------------------------
## Concrete Examples## Example A: User-Defined Precision (4 decimals)

* Operation: $1 / 3$ (User requests scale 4)
* Inputs:
* $A = \{ \text{num}: 1, \text{scale}: 0 \}$
   * $B = \{ \text{num}: 3, \text{scale}: 0 \}$
* Execution:
1. $\text{target\_scale} = 4$
   2. $\text{shift} = 4 + 0 - 0 = 4$
   3. $\text{scaled\_num}_A = 1 \times 10^4 = 10000$
   4. $\text{res\_num} = 10000 / 3 = 3333$
* Result: $\{ \text{num}: 3333, \text{scale}: 4 \}$ (Represents 0.3333)

## Example B: Implicit Precision ("As many as fit")

* Operation: $1 / 8$ (No precision specified, fallback to max safe scale, let's use 6 for a short example)
* Inputs:
* $A = \{ \text{num}: 1, \text{scale}: 0 \}$
   * $B = \{ \text{num}: 8, \text{scale}: 0 \}$
* Execution:
1. $\text{target\_scale} = 6$
   2. $\text{shift} = 6 + 0 - 0 = 6$
   3. $\text{scaled\_num}_A = 1 \times 10^6 = 1000000$
   4. $\text{res\_num} = 1000000 / 8 = 125000$
   5. Normalization: 125000 has trailing zeros.
   * $125000 / 10 = 12500$ (scale 5)
      * $12500 / 10 = 1250$ (scale 4)
      * $1250 / 10 = 125$ (scale 3)
      * $125 \pmod{10} \neq 0 \rightarrow$ Stop.
   * Result: $\{ \text{num}: 125, \text{scale}: 3 \}$ (Represents 0.125)

------------------------------

## 6. Integer Overflow Detection via LLVM Intrinsics
Because the Sec language enforces contract verification, safe arithmetic is mandatory. When handling decimals, scaling up (multiplication by powers of 10) and additions can easily exceed the boundaries of a signed 64-bit integer (i64 max: 9,223,372,036,854,775,807).
Instead of standard instructions like add or mul, the compiler runtime must use LLVM's overflow-aware intrinsics:

* @llvm.sadd.with.overflow.i64 (Signed Addition)
* @llvm.smul.with.overflow.i64 (Signed Multiplication)

These intrinsics return a two-element literal structure: { i64, i1 }, where the first element is the mathematical result (truncated), and the second element is a boolean flag (i1) which evaluates to true if an overflow occurred.
------------------------------
## LLVM IR Implementation Pattern (Addition Example)
When adding two scaled i64 numbers, we extract the overflow bit and use a conditional branch (br) to handle contract failures.

; Declare the intrinsic at the bottom of the file
declare { i64, i1 } @llvm.sadd.with.overflow.i64(i64, i64)

define %struct.decimal @safe_decimal_add_raw(i64 %a.num, i64 %b.num.scaled, i8 %scale) {
entry:
    ; Perform checked signed addition
    %res.struct = call { i64, i1 } @llvm.sadd.with.overflow.i64(i64 %a.num, i64 %b.num.scaled)
    
    ; Extract the mathematical result and the overflow bit
    %res.num = extractvalue { i64, i1 } %res.struct, 0
    %overflow = extractvalue { i64, i1 } %res.struct, 1
    
    ; If overflow is true, branch to panic/contract failure
    br i1 %overflow, label %overflow_panic, label %success

overflow_panic:
    ; Trigger Sec contract failure or runtime panic
    call void @sec_panic_contract_violation()
    unreachable

success:
    ; Pack into result struct and return
    %tmp = insertvalue %struct.decimal undef, i64 %res.num, 0
    %res = insertvalue %struct.decimal %tmp, i8 %scale, 1
    ret %struct.decimal %res
}

; Mock declaration of your language's panic handler
declare void @sec_panic_contract_violation()

------------------------------
## Multiplication Overflow Pattern
Multiplication escalates values much faster than addition. Every multiplication step in both the user operations and the inner scaling algorithms must be wrapped:

declare { i64, i1 } @llvm.smul.with.overflow.i64(i64, i64)

; Inside your multiplication or scaling logic:
%res.struct = call { i64, i1 } @llvm.smul.with.overflow.i64(i64 %base, i64 %factor)
%overflow = extractvalue { i64, i1 } %res.struct, 1
br i1 %overflow, label %overflow_panic, label %success

------------------------------
## Compiler Strategy Summary

   1. Never use naked add or mul on the .num fields inside your runtime functions.
   2. Always branch immediately on the extracted i1 overflow flag.
   3. Unreachable optimization: Marking the block after sec_panic_contract_violation() as unreachable tells LLVM that this path breaks execution, allowing the optimizer to generate highly efficient machine code for the success path.

## 7. Runtime Helper Function Implementation (_sec_decimal_add)
To avoid generating complex control flow (branches and PHI nodes) directly inside the compiler's code generator, we offload the logic to a static runtime library. The compiler simply emits an LLVM call instruction to _sec_decimal_add.
Below is the conceptual logic for the runtime function. This can be implemented in C (compiled to an LLVM bitcode .bc file via Clang and linked to the target binary) or generated once as an LLVM module.
## Core Logic (Go/Pseudo Implementation Blueprint)

// The runtime representation matching %struct.decimal = type { i64, i8 }type Decimal struct {
    Number int64
    Scale  uint8
}
// Global lookup table for fast 10^x scaling (prevents slow loops)var pow10 = [19]int64{
    1, 10, 100, 1000, 10000, 100000, 
    1000000, 10000000, 100000000, 1000000000, 
    10000000000, 100000000000, 1000000000000, 10000000000000, 
    100000000000000, 1000000000000000, 10000000000000000, 100000000000000000,
    1000000000000000000,
}
func _sec_decimal_add(a Decimal, b Decimal) Decimal {
    if a.Scale == b.Scale {
        // Safe addition with overflow check (maps to llvm.sadd.with.overflow)
        resNum, overflow := safeAdd(a.Number, b.Number)
        if overflow {
            panicContractViolation("Decimal addition overflow")
        }
        return Decimal{Number: resNum, Scale: a.Scale}
    }

    if a.Scale > b.Scale {
        diff := a.Scale - b.Scale
        if diff >= 19 { panicContractViolation("Scale difference exceeds max precision") }
        
        factor := pow10[diff]
        
        // Safe multiplication with overflow check (maps to llvm.smul.with.overflow)
        scaledB, overflowMul := safeMul(b.Number, factor)
        if overflowMul { panicContractViolation("Decimal scale-up overflow") }
        
        resNum, overflowAdd := safeAdd(a.Number, scaledB)
        if overflowAdd { panicContractViolation("Decimal addition overflow") }
        
        return Decimal{Number: resNum, Scale: a.Scale}
    } else {
        // b.Scale > a.Scale
        diff := b.Scale - a.Scale
        if diff >= 19 { panicContractViolation("Scale difference exceeds max precision") }
        
        factor := pow10[diff]
        
        scaledA, overflowMul := safeMul(a.Number, factor)
        if overflowMul { panicContractViolation("Decimal scale-up overflow") }
        
        resNum, overflowAdd := safeAdd(scaledA, b.Number)
        if overflowAdd { panicContractViolation("Decimal addition overflow") }
        
        return Decimal{Number: resNum, Scale: b.Scale}
    }
}

## How Your Go Compiler Emits the Call
When your AST walker in Go encounters a binary expression like ExprAdd where the types are Decimal, you do not generate block branching. Instead, you emit code that pushes the arguments and calls the runtime function:

// Conceptual snippet inside your Go compiler's LLVM IR generatorfunc (c *Compiler) CompileAddDecimal(lhs llvm.Value, rhs llvm.Value) llvm.Value {
    // 1. Look up the pre-declared runtime function in your LLVM module
    runtimeFunc := c.Module.NamedFunction("_sec_decimal_add")
    
    // 2. Prepare arguments (the two %struct.decimal values)
    args := []llvm.Value{lhs, rhs}
    
    // 3. Emit a simple LLVM Call instruction
    return c.Builder.CreateCall(runtimeFunc, args, "add_res")
}

## Benefits of This Architecture

* Testing: You can easily unit-test the _sec_decimal_add logic in isolation inside your runtime test suite.
* Diagnostics: If a contract violation happens, the runtime helper can print the exact numbers that caused the overflow, making Sec much friendlier to debug.
