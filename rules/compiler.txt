
Platform Selection and Conditional Compilation: #target and #if
===============================================================

Purpose
=======

Sec provides distinct constructs for platform targeting and feature flags.

The core model consists of:

    #target(os: string, arch: string)
    #if (compile_time_boolean_expression) { ... }

Goals:

- platform routing is handled cleanly at the file level,
- feature flags and build options are managed within the language,
- conditional blocks are parsed and type-checked even if not emitted,
- C-style preprocessor text substitution is avoided completely,
- the build system remains simple and driven by the source files.


#target
=======

`#target` is a file-level directive that specifies platform compatibility.

Examples:

    #target(os: "linux", arch: "amd64")
    import platform.linux.amd64 as sys

    #target(os: "linux", arch: "any")
    import platform.linux as linux

    #target(os: "windows", arch: "amd64")
    import platform.windows.amd64 as win32

Rules:

- `#target` must appear at the very top of a file, before any imports.
- It must take literal string arguments for `os` and `arch`.
- If `--target` is omitted, the compiler uses the host platform as the default target.
- Host default normalization:
    - runtime `linux` maps to `linux`
    - runtime `windows` maps to `windows`
    - runtime `darwin` maps to `macos`
    - runtime `amd64` maps to `amd64`
    - runtime `arm64` maps to `arm64`
    - runtime `arm` maps to `arm32`
- If the current compilation target does not match, the compiler drops the file.
- `arch: "any"` matches every architecture for the specified OS and is used
  for OS-common platform files.
- The rest of the dropped file is skipped during semantic analysis.
- If a package has no matching file for a target, it causes a compilation error.
- Current implementation note:
    - Single-file commands validate a parsed `#target` against the current target.
    - `arch: "any"` is accepted and matches the current architecture.
    - If the file does not match, compilation reports a target error instead of continuing into sema/codegen.
    - Multi-file package selection can later reuse the same target model to drop non-matching files before package analysis.
    - Source-level imports currently support selected `sec/stdlib/...` files and
      explicit `sec/platform/...` paths as a bootstrap mechanism.

Invalid:

    let mut x: int := 5
    #target(os: "linux", arch: "amd64") // must be at top

    #target(os: target_var, arch: "amd64") // must be literal

Expected errors:

    #target directive must appear before any code or declarations
    #target arguments must be compile-time string literals
    file target linux-arm64 does not match current target linux-amd64


#if
===

`#if` is a block-level construct for compile-time conditional code.

Examples:

    #if (compiler.target_os == "linux" && compiler.target_arch == "amd64") {
        let res = linux_amd64._sysCall3(linux_amd64._sysCall.sysWrite, fh, buff, len)
        return handle_linux_result(res)
    }

    #if (options.enable_debug_features) {
        log.Debug("Extended telemetry enabled")
    }

AST:

    CompileTimeIfStatement
        Condition Expression
        ThenBlock StatementBlock

Rules:

- `#if` must be followed by a compile-time evaluable boolean expression.
- The expression can use compiler constants or custom build options.
- The body of an `#if` block must be enclosed in braces `{ ... }`.
- The compiler parses and checks syntax for all branches regardless of the condition.
- Code generation (LLVM) only emits instructions for blocks where the condition is true.

Invalid:

    #if (runtime_variable == 10) { ... }
    #if (options.use_feature) print("missing braces")

Expected errors:

    #if condition must be evaluable at compile time
    #if block must be enclosed in braces

Compiler Command Line Interface: Sec CLI
========================================

Purpose
=======

The Sec compiler driver provides a unified CLI for developer tooling,
compilation stages, pipeline execution, and cross-compilation targeting.

The interface is split into clear development steps, progressing from
low-level pipeline inspection to high-level system builds.


Step 1: Compiler Pipeline Inspection (Debug Mode)
==================================================

For compiler development, debugging, and testing, specific subcommands
allow inspecting individual stages of the front-end and middle-end.

Usage:

    sec <lex|token|parse|ast|sema> <file.sec>
    sec emit-llvm <file.sec> -o <file.ll>

Rules:

- `lex` outputs the raw lexer character streams or errors.
- `token` outputs the structured token stream generated by the scanner.
- `parse` validates grammar and prints a positive confirmation or syntax error.
- `ast` prints a formatted tree representation of the Abstract Syntax Tree.
- `sema` runs semantic analysis (type checking, contract rules) and reports errors.
- `emit-llvm` generates raw LLVM IR text and requires the `-o` output flag.


Step 2: Executable Generation
=============================

The `build` subcommand automates the entire pipeline, forwarding the
generated LLVM IR through the compiler backend to produce a native binary.

Usage:

    sec build <file.sec> [options]

Rules:

- By default, `build` creates an executable named after the input file.
- The optional `-o <path>` flag overrides the default output binary name.
- It automatically invokes the backend linker (lld/clang) to bind the system runtime.

Example:

    sec build main.sec -o myprogram


Step 3: Immediate Execution
===========================

The `run` subcommand provides a fast development cycle by compiling and
immediately executing the program in memory or via a temporary location.

Usage:

    sec run <file.sec> [options] [-- [program_args...]]

Rules:

- It performs a complete compilation identical to `sec build`.
- It executes the resulting binary immediately after a successful build.
- Any arguments appended after a `--` separator are passed directly to the Sec program.


Step 4: Compilation Options and Target Management
=================================================

Sec supports cross-compilation and user-defined feature flags via explicit CLI options.
It adapts the transparent targeting model popularized by Zig and Go.

Usage flags:

    --target <os>-<arch>      Set explicit cross-compilation target
    -D <option>=<value>       Define a custom compile-time build option
    -O <level>                Set optimization level (0, 1, 2, 3, s, z)

Target options:

- If `--target` is omitted, the compiler defaults to the current host system.
- Valid `<os>` values: `linux`, `windows`, `macos`.
- Valid `<arch>` values: `amd64`, `arm64`, `arm` aka `arm32`.
- The target string maps directly to the internal `compiler.target_os` and `compiler.target_arch` flags checked by `#if` and `#target`.
Future OS options will be added and will include, BSDs, BAREMETAL, ESP32, some rtos etc and an array of arch's

Custom Build Options (-D):

- Maps directly to the `options` namespace available in compile-time `#if` checks.
- Boolean flags can be set without an explicit value (defaults to true).

Examples:

    sec build main.sec --target linux-amd64
    sec build main.sec -D enable_debug_features=true
    sec run main.sec -O 2 -- --port 8080

Expected CLI errors:

    Unknown subcommand: 'compile' (did you mean 'build'?)
    Missing output file flag '-o' for 'emit-llvm'
    Invalid target triple 'linux-x86'; expected '<os>-<arch>'
    Undefined option 'enable_debug_features' referenced in source files
