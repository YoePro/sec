Compiler, Targets, Platform Resolution and Compile-Time Selection
=================================================================

Purpose
=======

The Sec compiler automatically selects the appropriate platform
implementation for the requested compilation target.

Unlike C, Sec does not use textual preprocessing.

The model intentionally separates three concepts:

    Target selection      (compiler)
    Platform selection    (compiler)
    Feature selection     (developer)

This separation keeps platform routing deterministic, keeps the standard
library portable, and avoids C-style preprocessor substitution.


Compilation Targets
===================

Every compilation has a target consisting of:

    Operating System
    Architecture

Examples:

    linux-amd64
    linux-arm64
    linux-armv7
    windows-amd64
    macos-arm64
    baremetal-cortex-m4

If no target is specified, the compiler automatically selects the host platform.

Example:

    sec build main.sec

Host normalization:

    runtime "linux"   -> linux
    runtime "windows" -> windows
    runtime "darwin"  -> macos
    runtime "amd64"   -> amd64
    runtime "x86_64"  -> amd64
    runtime "arm64"   -> arm64
    runtime "arm"     -> arm32


Explicit Target Selection
=========================

Cross compilation is performed using:

    --target <os>-<arch>

Examples:

    sec build main.sec --target linux-arm64
    sec build main.sec --target windows-amd64

The target string maps to:

    compiler.target_os
    compiler.target_arch

These values are available to compile-time feature selection.


Target Registry
===============

All supported targets are registered inside the compiler.

Current implementation shape:

    type TargetDefinition struct {
        OS            string
        Arch          string
        LLVMTriple    string
        Status        TargetStatus
        CanParse      bool
        CanCheck      bool
        CanEmitLLVM   bool
        CanLink       bool
        CanRun        bool
    }

Each target contains:

    OS
    Architecture
    LLVM Target Triple
    Status
    Capability flags for parsing, semantic checking, LLVM emission, linking and running

Possible target states:

    TargetPlanned
    TargetExperimental
    TargetImplemented

Implemented target:

    linux-amd64

Experimental targets:

    linux-arm64
    linux-armv6
    linux-armv7
    macos-amd64
    macos-arm64

Planned targets:

    windows-amd64
    windows-arm64
    freebsd-amd64
    freebsd-arm64
    openbsd-amd64
    openbsd-arm64
    netbsd-amd64
    netbsd-arm64
    baremetal-cortex-m0
    baremetal-cortex-m3
    baremetal-cortex-m4
    baremetal-cortex-m7
    baremetal-riscv32
    freertos-cortex-m4
    freertos-cortex-m7
    rtems-any
    zephyr-any

Unknown targets produce:

    unknown compilation target 'linux-mips'

Targets without a requested capability produce:

    target windows-arm64 cannot emit LLVM yet
    target linux-arm64 cannot link yet

Future OS options may include BSDs, baremetal targets, ESP32 and RTOS targets.


Platform Resolution
===================

Platform routing is automatic.

For:

    linux-amd64

the compiler loads:

    sec/platform/linux/*.sec
    sec/platform/linux/amd64/*.sec

For:

    linux-arm64

the compiler loads:

    sec/platform/linux/*.sec
    sec/platform/linux/arm64/*.sec

Only files matching the selected target participate in the build.

The standard library should not import architecture-specific modules directly.
Instead it should import the virtual platform module:

    import "platform"

The compiler resolves this virtual module to the selected platform
implementation.

This keeps the standard library platform independent.

Bootstrap implementation note:

- Current source-level imports support selected `sec/stdlib/...` files and
  explicit `sec/platform/...` paths.
- This is a temporary bootstrap mechanism until package-based platform
  resolution is implemented.
- Existing explicit imports such as `platform/linux/amd64/raw_syscall` should
  be migrated toward the virtual `platform` module model.


#target
=======

`#target` is a file validation directive.

It documents which target a source file belongs to and allows the compiler to
verify that the file has been placed correctly.

Example:

    #target(os: "linux", arch: "amd64")

Rules:

- `#target` must appear before all imports and declarations.
- Arguments must be compile-time string literals.
- `#target` does not select the compilation target.
- `#target` does not participate in platform routing.
- `#target` is used only for validation.
- `arch: "any"` matches every architecture for the specified OS and is used
  for OS-common platform files.

Example:

    #target(os: "linux", arch: "any")

Invalid:

    let mut x: int := 5
    #target(os: "linux", arch: "amd64")

    #target(os: target_var, arch: "amd64")

Expected errors:

    #target directive must appear before any code or declarations
    #target arguments must be compile-time string literals
    file declares target linux-arm64 but is compiled as linux-amd64

Current implementation note:

- Single-file commands validate a parsed `#target` against the current target.
- `arch: "any"` is accepted and matches the current architecture.
- If the file does not match, compilation reports a target error instead of
  continuing into sema or codegen.
- Multi-file package selection should later use the same target model to drop
  non-matching files before package analysis.


#if
===

`#if` performs compile-time feature selection.

It is intended for optional functionality and build configuration.

Examples:

    #if (options.enable_sqlite) {
        ...
    }

    #if (options.enable_logging) {
        ...
    }

    #if (compiler.target_os == "linux" && compiler.target_arch == "amd64") {
        ...
    }

Rules:

- The condition must be evaluable at compile time.
- The body must be enclosed in braces.
- All branches are parsed and type checked.
- Only active branches generate LLVM IR.
- `#if` may use compiler constants and custom build options.

Typical compile-time values:

    compiler.target_os
    compiler.target_arch
    options.*

Invalid:

    #if (runtime_variable == 10) { ... }
    #if (options.use_feature) print("missing braces")

Expected errors:

    #if condition must be evaluable at compile time
    #if block must be enclosed in braces
    undefined option 'enable_debug_features'


Compiler Pipeline
=================

The compiler performs the following stages:

    Lexing
    Tokenization
    Parsing
    Semantic Analysis
    LLVM IR Generation
    Object Generation
    Linking

The frontend development commands expose these stages individually so compiler
work can be tested in small steps.


Compiler Commands
=================

Inspection commands:

    sec lex <file.sec>
    sec token <file.sec>
    sec parse <file.sec|dir|glob>...
    sec ast <file.sec|dir|glob>...
    sec sema <file.sec|dir|glob>...
    sec emit-llvm <file.sec> -o <file.ll>

Rules:

- `lex` outputs raw lexer findings.
- `token` outputs the structured token stream.
- `parse` validates grammar and prints the parsed source representation.
- `ast` prints a formatted tree representation of the AST.
- `sema` runs semantic analysis and reports errors and warnings.
- `emit-llvm` generates LLVM IR text and requires the `-o` output flag.
- `parse`, `ast` and `sema` accept multiple source inputs.
- A source input may be a `.sec` file, a directory or a shell/glob pattern.
- Directories are scanned recursively for `.sec` files.
- For target-aware commands such as `sema`, platform source paths are filtered
  before parsing so only files matching the selected target participate.
  Example: on `linux-amd64`, `sec/platform/linux/amd64/*.sec` is included and
  `sec/platform/linux/arm64/*.sec` is skipped.

Current implementation note:

- `lex`, `token`, `parse`, `ast`, `sema`, `emit-llvm` and `build` exist.
- `parse`, `ast` and `sema` support multiple input files, directories and
  quoted globs.
- `emit-llvm` supports `--target <os-arch>` and emits a target-specific LLVM
  triple.
- `build` supports `-o`, `--target <os-arch>`, `--keep-llvm` and
  `--clang <path>`.
- `build` currently emits LLVM IR and invokes `clang` directly.
- `run`, `-D` and optimization flags are specified but not implemented yet.


Build
=====

Usage:

    sec build <file.sec> [options]

Compilation performs:

    Detect target
        ->
    Resolve platform
        ->
    Compile package
        ->
    Generate LLVM IR
        ->
    Generate object files
        ->
    Link executable

If no output name is specified, the executable is named after the input source
file.

Example:

    sec build hello.sec

produces:

    hello

Custom output name:

    sec build hello.sec -o myprogram

Current implementation note:

- `build` currently emits LLVM and invokes `clang`.
- Without `--keep-llvm`, the LLVM file is temporary and removed after linking.
- With `--keep-llvm`, the LLVM file is written next to the output as
  `<output>.ll`.
- `--target <os-arch>` is used for source target validation, stdlib/platform
  selection, LLVM target triple generation and the clang `-target` argument.
- It validates that the program has `module main`.
- It validates that `fn main() int` or `fn main() void` exists.


Run
===

Usage:

    sec run <file.sec> [options] [-- program arguments]

`run` performs the same compilation pipeline as `build`.

After a successful build, the executable is started automatically.

Program arguments appearing after `--` are passed unchanged to the compiled
program.

Example:

    sec run server.sec -- --port 8080

Current implementation note:

- `run` is specified but not implemented yet.


Compiler Options
================

Output:

    -o <filename>

Optimization:

    -O0
    -O1
    -O2
    -O3
    -Os
    -Oz

Target selection:

    --target linux-amd64

Compile-time options:

    -D enable_sqlite=true
    -D enable_logging=true

Boolean `-D` options may be set without an explicit value and default to true.

Examples:

    sec build main.sec --target linux-amd64
    sec build main.sec -D enable_debug_features=true
    sec run main.sec -O2 -- --port 8080

Expected CLI errors:

    unknown subcommand 'compile'
    missing output file flag '-o' for 'emit-llvm'
    invalid target 'linux-x86'; expected '<os>-<arch>'
    undefined option 'enable_debug_features'


Platform Contract
=================

Every compilation target must provide the platform functionality required by
the runtime and the standard library.

The compiler validates that the selected platform implementation contains all
required functions before LLVM IR generation begins.

Missing functionality is reported as a compile-time error.

Example:

    platform linux-amd64 missing required function _rawSyscall3


Current Bootstrap Layout
========================

Current project layout:

    sec/platform/<os>/<arch>/
    sec/platform/<os>/
    sec/runtime/
    sec/stdlib/

Examples:

    sec/platform/linux/file.sec
    sec/platform/linux/amd64/raw_syscall.sec
    sec/platform/linux/amd64/syscall_numbers.sec
    sec/stdlib/io/write.linux.amd64.sec
    sec/stdlib/fmt/fmt.sec

The long-term platform model should prefer virtual platform imports and target
selection over hard-coded standard-library imports of architecture-specific
files.
