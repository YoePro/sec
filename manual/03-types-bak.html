Types

Primitive Types

Sec has a small set of primitive types.

Type Purpose
boolBoolean value, either true or false.
byteSingle byte value.
charCharacter value.
runeUnicode code point.
intSigned integer.
uintUnsigned integer.
floatFloating-point number.
decimalBuilt-in decimal number.
stringUTF-8 text.
bytesByte sequence.

decimal is a built-in type because financial, measurement, and business programs often need decimal behavior that should not be modeled with binary floating-point numbers.

Semantic Types

A Sec type may describe meaning, not only representation. This is a central part of the language. 

type CustomerID int
type ProductID int

Even if both types are based on int, they represent different concepts. Code should not accidentally pass a product ID where a customer ID is required. 

fn LoadCustomer(id: CustomerID) Result[Customer, CustomerError] {
}

This allows the compiler to catch domain mistakes that would otherwise be invisible if everything were represented as plain integers or strings.

Constrained Types

A type may add constraints to the underlying representation. Constraints describe what values are valid for that type. 

type Percent int range 0..100
type Port int range 1..65535

Possible built-in contracts include:

  • range
  • regex
  • minLength
  • maxLength
  • precision
  • scale
  • validate { ... }
Rule: Compile-time validation is performed whenever the value is known at compile time. Runtime values require explicit validation.

Conversion and Validation

Creating a constrained value from an unknown runtime value may fail. Such construction is fallible and therefore uses try. 

value := readNumber()
percent := try Percent(value)

If the compiler can prove that a literal is valid, construction can be accepted at compile time. If the literal is invalid, compilation fails. 

ok := Percent(90)      // valid
bad := Percent(150)    // compile-time error

Assignment between different semantic types is not automatic, even if the underlying representation is the same. 

type CustomerID int
type ProductID int

customer := CustomerID(10)
product := ProductID(10)

customer = product // type error

Default Values

Primitive types have default values.

Type Default
int0
uint0
float0.0
decimal0.0
boolfalse
string""
bytesempty byte sequence

A constrained type has a default value only if the underlying default value satisfies the constraint. 

type Percent int range 0..100
// Percent can default to 0.

type Port int range 1..65535
// Port cannot default to 0, because 0 is invalid.

Units of Measure

Units are written with angle brackets. The <> syntax is reserved for units of measure. 

type Meter decimal<m>
type Speed decimal<m/s>
type SEK decimal<SEK>

Values with incompatible units cannot be mixed by accident. 

type Money decimal<SEK>
type Speed decimal<m/s>

let mut amount: Money = 5.90
let mut speed: Speed = 50

amount += speed // unit error

Units are part of the type. They are not comments and not runtime metadata only. The compiler uses them to reject invalid operations.

Strings

Sec strings are UTF-8 text. Binary data should use bytes, not string. 

name := "Anna"
message := $"Hello {name}"

String interpolation uses the $"..." form. Expressions inside braces are evaluated and inserted into the resulting string.

Absence

Sec does not introduce nil. Absence is represented with Option[T]. 

type Option[T] union {
    Some(T)
    None
}

JSON null and SQL NULL map to None when the target field is optional. 

type CustomerRow struct {
    Name: Option[string],
    Port: Option[Port],
}
Rule: T always contains a valid value. Option[T] contains either Some(T) or None.

Any

Sec does not have a separate primitive any type. If a truly unknown runtime value is needed, it can be represented by a top-level empty interface. 

type Any interface {}

The preferred order is:

  1. Use generic type parameters for compile-time type variation.
  2. Use interfaces for behavior contracts.
  3. Use Any only for truly unknown runtime values.
