Properties

A property defines controlled behavior for reading or assigning a member-like value on a type.

Fields store data. Properties define behavior. Properties therefore belong inside an impl block. 

type Vehicle struct {
    _speed: Speed,
}

impl Vehicle {
    property TopSpeed: Speed {
        get {
            return _speed
        }

        try set value {
            _speed = value
            return Ok()
        }
    }
}
Rule: A field stores a value. A property controls how a value is read or assigned.

Syntax

A property declares a name, a type and one or more accessors:

property Name: Type {
    get {
        return value
    }

    set value {
        target = value
    }
}

A fallible setter uses try set:

property Name: Type {
    try set value {
        if invalid {
            return Err(Error.InvalidValue)
        }

        target = value
        return Ok()
    }
}

A property may contain:

  • one getter
  • one infallible setter
  • one fallible setter

A property must contain at least one accessor.

set and try set are mutually exclusive.

Placement

Properties are valid only inside ordinary impl blocks:

impl Vehicle {
    property TopSpeed: Speed {
        get {
            return _speed
        }
    }
}

The property becomes a member of the impl target type:

let current := vehicle.TopSpeed

Parsing keeps this as ordinary member access. Semantic analysis determines whether the member is a field, property, register field, enum value or another supported member form.

Invalid: Properties cannot be declared at module level or inside structs.

Property Identity

A property has:

  • a name
  • a declared type
  • an optional getter
  • an optional setter
  • a fallible-setter flag
  • an error type when the setter is fallible
  • a source location

Property names must be unique for the target type.

impl Vehicle {
    property TopSpeed: Speed {
        get {
            return _speed
        }
    }

    property TopSpeed: Speed {
        get {
            return _speed
        }
    }
}

This is invalid.

duplicate property "TopSpeed" in impl Vehicle

Getters

A getter produces the value returned by reading the property.

property CurrentSpeed: Speed {
    get {
        return _speed
    }
}

The returned value must be assignable to the property type.

property BadGet: Speed {
    get {
        return Money(5.90)
    }
}

This is invalid:

getter BadGet must return Speed, got Money

Constructor-style conversion is resolved before checking the return type:

Money(5.90)

has type Money, not plain decimal.

A getter must return on every continuing path.

property MissingGet: Speed {
    get {
    }
}

This is invalid:

getter MissingGet must return Speed

Setters

A setter defines the behavior of assigning to the property.

property CurrentSpeed: Speed {
    set value {
        _speed = value
    }
}

The setter must declare a value parameter.

property CurrentSpeed: Speed {
    set {
        _speed = value
    }
}

This is invalid:

setter for CurrentSpeed must declare value parameter

The setter value parameter has the declared property type.

property CurrentSpeed: Speed {
    set value {
        _speed = value
    }
}

Here, value has type Speed.

An infallible setter must not return an error:

property CurrentSpeed: Speed {
    set value {
        return Err(InvalidSpeed)
    }
}

This is invalid:

non-fallible setter CurrentSpeed cannot return Err

Fallible Setters

A fallible setter uses try set.

property TopSpeed: Speed {
    try set value {
        if value < Speed(0) {
            return Err(InvalidSpeed)
        }

        _speed = value
        return Ok()
    }
}

A fallible setter behaves like a hidden fallible operation behind assignment syntax.

Assignment therefore requires try:

try car.TopSpeed = currentSpeed {
    Err(error) => fmt.Println("could not set top speed")
}

Ordinary assignment is invalid:

car.TopSpeed = currentSpeed
assigning fallible property TopSpeed requires try

The success path is implicit. An Ok handler is not required.

Assignment

An infallible property uses ordinary assignment:

car.CurrentSpeed = speed

The assigned value must be assignable to the property type.

car.CurrentSpeed = Money(10)

This is invalid when CurrentSpeed has type Speed.

A fallible property uses try assignment:

try car.TopSpeed = speed {
    Err(error) => Handle(error)
}

The right-hand expression is evaluated before the setter runs.

The backend must preserve evaluation order, setter side effects and error behavior.

Compound Assignment

Compound assignment applies the corresponding operation and then invokes the setter.

car.CurrentSpeed += delta

Conceptually:

car.CurrentSpeed = car.CurrentSpeed + delta

A fallible property still requires try:

try car.TopSpeed += delta {
    Err(error) => Handle(error)
}

The getter and setter must be evaluated in the correct order.

Rule: Compound property assignment must not duplicate getter or setter evaluation.

Access Inside Property Bodies

Property bodies may access:

  • self, when available
  • fields on the impl target
  • other properties on the impl target
  • the setter value parameter
  • visible module symbols
  • enum values
  • constructors and conversions
impl Vehicle {
    property Current: Speed {
        get {
            return TopSpeed
        }
    }

    property TopSpeed: Speed {
        get {
            return _speed
        }
    }
}

This is valid because all property signatures are registered before property bodies are analyzed.

Forward references between properties are allowed.

Relationship to Contracts

A constrained assignment may use the same semantic model as a hidden fallible setter. 

type Percent int range 0..100

Assigning a runtime value to a constrained destination may require validation before the assignment succeeds.

The shared design principle is:

  • assignment that cannot fail uses ordinary assignment
  • assignment that can fail requires explicit try
Rule: Potential assignment failure must remain visible in source code.

Error Handling

Fallible property assignment participates in normal Sec error handling.

try car.TopSpeed = requestedSpeed {
    Err(error) => Log(error)
}

The setter error type determines the type bound by the error pattern.

The success path is implicit:

Ok(_) => continue

If the surrounding function propagates the error, normal try propagation rules apply.

Fallible property assignment is not a separate error system. It uses the same Result, Err and try model as other fallible operations.

Runtime and Lowering

A property is not stored as a field at runtime.

Property access lowers to explicit operations:

  • a getter operation for reads
  • a setter operation for infallible writes
  • a fallible setter operation for try writes

The compiler must preserve:

  • evaluation order
  • getter side effects
  • setter side effects
  • fallible setter errors
  • ownership and borrow behavior
  • cleanup and defer behavior

Semantic analysis resolves property behavior before backend lowering. The backend does not infer property semantics.

Rule: Properties have behavior but no independent runtime storage.

Summary of Rules

  • Properties belong inside ordinary impl blocks.
  • A property declares a name and a type.
  • A property must contain at least one accessor.
  • A property may contain one getter.
  • A property may contain either one setter or one fallible setter.
  • set and try set are mutually exclusive.
  • Property names must be unique on the target type.
  • A getter must return the property type on every continuing path.
  • A setter must declare a value parameter.
  • The setter parameter has the property type.
  • An infallible setter cannot return Err.
  • A fallible setter is declared with try set.
  • Assignment to a fallible property requires try.
  • The success path of try assignment is implicit.
  • Compound assignment follows the same fallibility rules.
  • Property bodies may refer to fields and other registered properties.
  • Forward references between properties are allowed.
  • Properties participate in normal Error Handling.
  • Properties do not create stored fields.
  • Property behavior is resolved before backend lowering.
