Implement first version of function declarations.

Syntax:

    fn name(param: Type, param: Type) ReturnType {
        statements
    }

Rules:
- Parameters use name: Type.
- Parameters are comma-separated.
- Trailing comma is allowed.
- Return type is required for now.
- void is used for no return value.
- Function body creates a new scope.
- Parameters are immutable local symbols by default.
- Duplicate parameter names are invalid.
- Return expression must match declared return type.
- A non-void function must return.
- A void function may use `return` without expression.
- Comparison operators return bool:
  - `==`
  - `!=`
  - `<`
  - `<=`
  - `>`
  - `>=`
- Logical `&&` and `||` require bool operands and return bool.
- Prefix `!` requires a bool operand and returns bool.
- Function calls use:

    Name(arg, arg)

- Function call argument count must match the declaration.
- Function call arguments must be assignable to the corresponding parameter type.
- Function call expression type is the declared return type.
- The parser creates a CallExpression for all `Name(...)`-shaped expressions.
  Semantic analysis resolves the callee as a function overload, type conversion,
  constructor or another callable entity.

Parser:
- Add FunctionStatement / FunctionDeclaration AST node.
- Parse name, parameters, return type, body.
- Body can initially support:
  - let declarations
  - assignments
  - return statements

AST:
FunctionDeclaration:
    Token
    Name *Identifier
    Parameters []Parameter
    ReturnType *TypeReference
    Body *BlockStatement

Parameter:
    Name *Identifier
    Type *TypeReference

BlockStatement:
    Statements []Statement

ReturnStatement:
    Token
    Value Expression // nil for `return`

Sema:
- Register functions before checking bodies.
- Function names may be overloaded in module scope.
- Overloads must have different parameter type signatures.
- A duplicate function with the same parameter type signature is invalid.
- Return type alone must not distinguish overloads. Call arguments must be
  sufficient to choose the function.
- Function call resolution chooses the overload whose parameters can accept the
  argument expressions.
- If no overload matches, report an argument error.
- If multiple overloads match, report an ambiguous call.
- Parameter types must exist.
- Return type must exist.
- Function body has child scope.
- Parameters are defined in function scope.
- Check return type compatibility.
- Check duplicate parameter names.

Tests:
Valid:

    fn add(a: int, b: int) int {
        return a + b
    }

    fn noop() void {
        return
    }

    let x := add(1, 2)
    let y: int := add(3, 4)

Invalid:

    fn bad(a: int, a: int) int {
        return a
    }

Expected:
    duplicate parameter "a"

    fn bad() int {
    }

Expected:
    function bad must return int

    fn bad() int {
        return true
    }

Expected:
    function bad must return int, got bool

---
Implement the first language-level support for Result, Ok, Err and try.

Goal
====

Do NOT implement full error handling yet.

This iteration should establish the language model and AST so later work can build on it without redesign.

Language model
==============

Result is a normal generic type:

    Result[ValueType, ErrorType]

The parser must not special-case Result as a type.
It is parsed as an ordinary generic TypeReference.

Ok and Err are language builtins.

Examples:

    return Ok(speed)

    return Err(IOError.InvalidValue)

A function returning Result[T,E] may return either:

    Ok(value)

or

    Err(error)

Parser
======

Implement parsing for:

    Ok(expr)

    Err(expr)

These should produce dedicated AST nodes:

    OkExpression
    ErrExpression

instead of generic CallExpression.

Reason:
The compiler will later give them special semantic meaning.

AST
===

Add:

    OkExpression
        Token
        Value Expression

    ErrExpression
        Token
        Value Expression

Parser should recognize:

    Ok(...)
    Err(...)

before generic CallExpression.

Result return checking
======================

For functions returning:

    Result[T,E]

allow:

    return Ok(expr)

where expr is assignable to T.

allow:

    return Err(expr)

where expr is assignable to E.

Reject:

    return Ok(errorValue)

    return Err(resultValue)

Reject plain returns:

    return value

inside Result-returning functions.

Sema
====

Implement only basic validation.

For:

    fn Foo() Result[int, IOError]

verify:

    Ok(...)
        matches Result value type.

    Err(...)
        matches Result error type.

Do NOT implement propagation.

Do NOT implement pattern matching.

Do NOT implement exhaustive analysis.

try
===

Add parser support only.

Syntax:

    try expression

Example:

    let speed := try CalculateSpeed()

Create:

    TryExpression
        Token
        Expression

Initial semantic behavior:

- expression type is Result[...]
- try expression type is the Result value type T from Result[T,E]
- otherwise report:

    try requires Result expression

No propagation or lowering yet.

Context rule:

A try expression is only valid inside a function that can propagate the same
error type, unless the error is handled locally by a later try block syntax.

Valid:

    fn Use() Result[Speed, IOError] {
        let speed := try CalculateSpeed()
        return Ok(speed)
    }

Invalid later:

    fn Use() Speed {
        return try CalculateSpeed()
    }

Expected later:

    cannot propagate IOError from function returning Speed

This propagation check does not need to be implemented in the first Result/try
step, but the AST and semantic model must keep the Result error type available
so the check can be added without changing parsing.

Tests
=====

Valid:

    fn Good() Result[int, IOError] {
        return Ok(123)
    }

    fn Fail() Result[int, IOError] {
        return Err(IOError.InvalidValue)
    }

Invalid:

    fn Bad() Result[int, IOError] {
        return Ok(IOError.InvalidValue)
    }

Expected:

    Ok value must be int, got IOError

Invalid:

    fn Bad() Result[int, IOError] {
        return Err(123)
    }

Expected:

    Err value must be IOError, got int

Invalid:

    fn Bad() Result[int, IOError] {
        return 123
    }

Expected:

    Result-returning function must return Ok(...) or Err(...)

Parser:

    let x := try Foo()

should parse successfully.

Sema:

    try 123

Expected:

    try requires Result expression

Overload resolution
===================

Function overload resolution must use parameter types only.

The return type is never used to select an overload.

Invalid:

    fn Convert(value: int) string
    fn Convert(value: int) bool

Reason:
Both functions have the same parameter signature.

Exact matches are preferred over conversions.

Example:

    fn Print(value: int) void
    fn Print(value: int64) void

    Print(10)

The literal should first be resolved using its natural/default type.
If both overloads remain valid with equal priority, the call is ambiguous.

Named types remain distinct during overload resolution.

Example:

    type Percent int range 0..100

    fn Set(value: int) void
    fn Set(value: Percent) void

    let p: Percent := 50

    Set(p)   // selects Percent overload
    Set(50)  // selects int overload unless explicitly converted

Result validation
=================

Result must have exactly two type arguments:

    Result[ValueType, ErrorType]

Invalid:

    Result[int]
    Result[int, IOError, string]

Both type arguments must resolve to known types.

Ok and Err must contain exactly one expression.

Invalid:

    Ok()
    Ok(1, 2)
    Err()
