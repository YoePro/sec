Structs

A struct defines a named data layout composed of typed fields. Structs contain data only. Behavior is added separately through an impl block. 

type Coordinate struct {
    x: Meter,
    y: Meter,
    z: Meter,
}
Rule: Sec does not use classes. Structs define data, impl blocks define behavior, and interfaces define behavior contracts.

Struct Declaration

A named struct is declared with type Name struct:

type User struct {
    id: int,
    name: string,
}

A struct declaration always introduces a named type.

Anonymous struct types are not part of the initial language model.

Rule: Stored data layout is defined only by type Name struct { ... }.

Struct Fields

Fields use typed identifier syntax:

name: Type

Fields are comma-separated. A trailing comma is allowed.

type Coordinate struct {
    x: Meter,
    y: Meter,
    z: Meter,
}

Single-line declarations use the same syntax:

type Coordinate struct { x: Meter, y: Meter, z: Meter }

Line breaks do not replace commas.

type Invalid struct {
    x: Meter
    y: Meter
}
Invalid: Struct field lists always require commas between fields.

Field names must be unique within the struct:

type Invalid struct {
    value: int,
    value: string,
}

Every field type must resolve to a known type.

type Invalid struct {
    value: MissingType,
}

Struct Construction

A struct value is created with a struct literal:

let point := Coordinate {
    x: Meter(10),
    y: Meter(20),
    z: Meter(30),
}

Field names make construction independent of declaration order:

let point := Coordinate {
    z: Meter(30),
    x: Meter(10),
    y: Meter(20),
}

Every supplied field must exist on the struct.

let point := Coordinate {
    x: Meter(10),
    y: Meter(20),
    unknown: Meter(30),
}

Duplicate fields are invalid:

let point := Coordinate {
    x: Meter(10),
    x: Meter(20),
    y: Meter(30),
    z: Meter(40),
}

Field values must be assignable to their declared field types.

let point := Coordinate {
    x: Second(10),
    y: Meter(20),
    z: Meter(30),
}
Rule: Struct construction preserves the exact semantic types of fields. Compatible storage representation is not enough.

Field Access

Fields are accessed with member syntax:

let horizontal := point.x
let vertical := point.y

The result has the field's declared type.

type Vehicle struct {
    _speed: Speed,
}

let speed: Speed := vehicle._speed

Accessing an unknown field is invalid:

let value := point.distance

Reserved register fields named _ are a register feature, not ordinary struct fields.

Field Assignment

Assignment to a struct field follows normal mutability rules.

let mut point := Coordinate {
    x: Meter(10),
    y: Meter(20),
    z: Meter(30),
}

point.x = Meter(15)

The containing value must be mutable.

let point := Coordinate {
    x: Meter(10),
    y: Meter(20),
    z: Meter(30),
}

point.x = Meter(15)
Invalid: Fields of an immutable struct value cannot be reassigned.

The assigned value must match the field type:

point.x = Second(15)

Initialization

An immutable struct variable must be initialized when declared:

let point: Coordinate

This is invalid.

A mutable struct variable may be declared without an initializer:

let mut point: Coordinate

The variable must be definitely assigned before it is read:

let mut point: Coordinate

point = Coordinate {
    x: Meter(10),
    y: Meter(20),
    z: Meter(30),
}

let x := point.x

Definite-assignment rules apply to struct values in the same way as other uninitialized mutable variables.

Whether a struct has an implicit default value depends on the defaultability rules of all its fields.

Nested Struct Types

A struct type may be declared inside the owning type's impl block:

type Vehicle struct {
    engine: Vehicle.Engine,
}

impl Vehicle {
    type Engine struct {
        power: Kilowatt,
    }
}

Outside the impl block, the full qualified name is required:

Vehicle.Engine

Inside impl Vehicle, the short name may be used:

Engine

The fully qualified form remains valid inside the impl block:

Vehicle.Engine

Nested types may be referenced before their declaration because Sec registers type names before resolving field types. 

type Vehicle struct {
    engine: Vehicle.Engine,
    fuel: Vehicle.FuelType,
}

impl Vehicle {
    type Engine struct {
        power: Kilowatt,
    }

    enum FuelType {
        petrol,
        diesel,
        electric,
    }
}
Rule: Nested named types still use type. Forms such as struct Engine { ... } are invalid.

Generic Structs

Struct declarations may introduce generic parameters:

type Pair[A, B] struct {
    first: A,
    second: B,
}

A concrete generic struct is constructed with all type arguments:

let pair := Pair[int, string] {
    first: 10,
    second: "ten",
}

Each concrete instantiation has its own field types and layout.

Pair[int, string]
Pair[Meter, Second]

These are different concrete struct types.

Generic structs follow all ordinary struct rules after substitution.

Struct Field Tags

A field may include metadata in a backtick-delimited tag after its type. 

type User struct {
    ID: int `json:"id" xml:"id" db:"user_id"`,
    Name: string `json:"name" xml:"name"`,
    Password: string `json:"-"`,
}

Tags use a Go-inspired key and quoted-value syntax:

key:"value" key:"value"

Multiple tags may be attached to one field.

Tag keys are not restricted by the compiler:

type Record struct {
    ID: int `json:"id" db:"record_id" search:"keyword"`,
}

This allows serializers, database tools and other systems to define their own metadata without changing the language.

Rule: Field tags are metadata. They do not change field type, mutability, layout or assignment rules.

Malformed tag syntax is invalid:

type Invalid struct {
    ID: int `json:id`,
}

Struct Type Identity

Every named struct is a distinct type.

type Point struct {
    x: int,
    y: int,
}

type Size struct {
    x: int,
    y: int,
}

Point and Size are different types even though their field layouts are identical. 

let point: Point := Size {
    x: 10,
    y: 20,
}

This assignment is invalid.

Rule: Sec uses nominal struct identity, not structural type equivalence.

Separation of Data and Behavior

A struct declares only stored data:

type Vehicle struct {
    _speed: Speed,
}

Behavior is declared separately:

impl Vehicle {
    fn Stop() void {
        _speed = Speed(0)
    }
}

Computed or controlled access is declared with properties:

impl Vehicle {
    property TopSpeed: Speed {
        get {
            return _speed
        }
    }
}

An impl block cannot declare stored fields or local variables:

impl Vehicle {
    let currentSpeed: Speed
}

This is invalid.

Design rule: Structs describe what a value stores. Impl blocks describe what the value can do.

Common Errors

Missing comma

type Coordinate struct {
    x: Meter
    y: Meter
}

Duplicate field

type Coordinate struct {
    x: Meter,
    x: Meter,
}

Unknown field type

type Coordinate struct {
    x: MissingType,
}

Unknown construction field

Coordinate {
    x: Meter(10),
    y: Meter(20),
    w: Meter(30),
}

Wrong field value type

Coordinate {
    x: Second(10),
    y: Meter(20),
    z: Meter(30),
}

Assignment through immutable value

let point := Coordinate {
    x: Meter(10),
    y: Meter(20),
    z: Meter(30),
}

point.x = Meter(15)

Uninitialized immutable value

let point: Coordinate

Invalid nested declaration syntax

impl Vehicle {
    struct Engine {
        power: Kilowatt,
    }
}

Stored data inside impl

impl Vehicle {
    let speed: Speed
}

Summary of Rules

  • A struct is declared with type Name struct.
  • Structs define stored data only.
  • Fields use name: Type syntax.
  • Fields are comma-separated.
  • A trailing comma is allowed.
  • Field names must be unique inside the struct.
  • Every field type must resolve to a known type.
  • Struct literals use named-field construction.
  • Unknown and duplicate construction fields are invalid.
  • Field values must match their declared semantic types.
  • Field access uses member syntax.
  • Field assignment requires a mutable containing value.
  • Immutable variables must be initialized.
  • Mutable uninitialized variables follow definite-assignment rules.
  • Struct type identity is nominal.
  • Generic structs are specialized with concrete type arguments.
  • Nested struct types belong to the owner type namespace.
  • Nested types use fully qualified names outside the impl block.
  • Field tags are optional metadata.
  • Tags do not affect type semantics.
  • Behavior belongs in an impl block.
  • Stored fields cannot be added in an impl block.
