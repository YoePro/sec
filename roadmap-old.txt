# SEC COMPILER TODO

## Current status

The main control-flow constructs have been implemented and tested:

* if
* for
* while
* switch
* match

Enum declarations have also been implemented and tested, including:

* implicit and explicit underlying integer types
* automatic numeric values
* explicit numeric values
* aliases
* iota
* enum namespaces
* nested enums
* explicit enum/integer conversions
* enum type checking
* match and switch interaction

defer is currently being implemented and tested.

The defer tests have revealed several parser, AST and semantic-analysis issues.
These should be fixed before moving on to another major language area.

The purpose of this file is to keep track of the recommended order for the
remaining parser, AST, semantic-analysis and later code-generation work.

======================================================================

1. Finish defer
   ======================================================================

Complete the implementation of defer before starting another major area.

Required behavior:

* defer uses block syntax only
* defer is function-scoped
* defer executes on normal function exit
* defer executes on explicit return
* defer executes on implicit void return
* defer executes before Result error propagation
* multiple defer blocks execute in LIFO order
* defer inside conditions is registered only when reached
* defer inside loops is registered once for every execution
* return expressions are evaluated before defer blocks execute
* the original return value or propagated error must be preserved
* variables are read when the defer block executes
* normal ownership and borrowing rules apply
* return, break, continue and fallthrough are invalid inside defer
* nested defer is invalid
* errors inside defer must be handled locally
* defer inside loops should produce a warning

Implementation work may affect:

* parser statement handling
* AST defer node
* semantic context for defer blocks
* control-flow analysis
* definite-assignment analysis
* return-path analysis
* Result propagation
* code generation for multiple exit paths
* future ownership and borrow tracking

Do not move on until defer_valid.sec passes and defer_invalid.sec produces the
expected errors and warnings without unrelated diagnostics.

* references may not outlive owners

Recommended implementation order
================================

The current recommended order is:

1. Finish defer.
2. Implement union declarations.
3. Build shared constant-expression evaluation.
4. Complete impl, methods and properties.
5. Complete arrays, slices and indexing.
6. Implement unsafe blocks and raw pointers.
7. Implement interfaces.
8. Continue generics beyond the implemented parser/sema subset.
9. Implement references, ownership and borrow checking.
10. Implement spread and variadic parameters.
11. Implement lambdas, local functions and closures.
12. Design and implement spawn and await.

Parser recovery, diagnostics, AST cleanup and semantic-analysis cleanup should
continue throughout all steps.

This order is not absolute.

It may be changed when implementation work reveals a dependency, but changes
should be deliberate and recorded here so that unfinished areas are not
forgotten.
