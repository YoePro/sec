=============================================================================================================================================================================================
Memory Model
=============================================================================================================================================================================================

The Sec memory model is based on deterministic ownership and lexical scopes.

Every value has a well-defined owner responsible for its lifetime. Memory is released automatically when ownership ends. The language does not require garbage collection.

Ownership is explicit, while lifetime management remains implicit. The compiler determines object lifetimes through static analysis rather than programmer-provided lifetime annotations.

Memory safety is achieved through:

- deterministic ownership
- lexical scopes
- compile-time lifetime analysis
- reference validation
- escape analysis
- ownership analysis
- restricted unsafe operations

The language is designed to prevent:

- use-after-free
- double free
- dangling references
- accidental aliasing of mutable data
- hidden heap allocations
- implicit ownership transfers

The compiler is responsible for verifying these properties during semantic analysis.

References never own memory. They provide temporary access to values owned elsewhere. Reference validity is determined by the compiler and may never exceed the lifetime of the referenced object.

Heap allocation is explicit. The compiler must never silently promote stack allocations to the heap.

Memory lifetime is determined solely by program structure. Scope exit, ownership transfer and deterministic destruction define when memory becomes invalid.

Unsafe code does not disable the memory model. Unsafe only permits operations that cannot be verified by the compiler while preserving all remaining semantic validation.

Foreign memory obtained through FFI follows the same ownership principles but may require explicit ownership annotations or wrapper types defined by the FFI boundary.

The memory model is identical across all supported targets. Platform-specific differences affect implementation details only and must not change language semantics.

Hosted and embedded targets share the same ownership rules. Platform profiles may restrict available allocation mechanisms but must not change ownership semantics.

Detailed rules for ownership, references, borrowing, raw pointers, lifetime analysis and FFI are specified in their respective rule documents.
