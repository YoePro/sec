Visibility

Proposal:

Name      public

_name     package/module

__name    private

Underscore visibility:

Names that do not start with `_` are public.

Names that start with a single `_` are visible inside the same module root.

Example:

module x
module x.y
module x.z

These all share the module root `x`.

Therefore `_name` declared in `x.y` may be used from `x.z`.

Names that start with `__` are visible only inside the exact module where they
are declared.

Therefore `__name` declared in `x.y` may be used from `x.y`, but not from `x`
or `x.z`.

Memory Model

Ownership model inspired by Rust, but intentionally simpler.

Goals

No GC by default.
No hidden allocation.
Safe by default.
Explicit unsafe.
Borrow checking.
Easier than Rust where possible.
No explicit lifetime annotations.

Core types

T            owned value
ref T        shared immutable reference
mut ref T    exclusive mutable reference
?ref T       optional reference

[]T          slice/view
Vec[T]       owned collection
Box[T]       owned heap allocation

unsafe *T    raw pointer

Core rules

Many ref T are allowed.

Only one mut ref T may exist at a time.

A mut ref may not coexist with any ref.

References may never outlive their owner.

No null in safe code.

No pointer arithmetic in safe code.
Borrow Checker

The borrow checker is block-oriented.

References are validated within lexical scopes.

Example

fn main() void {

    user := User{name: "Jonas"}

    {
        name := ref user.name
        println(name)
    }

    // name is no longer valid here
}

Returning a local reference is forbidden.

fn bad() ref User {

    user := User{name: "Jonas"}

    return ref user
}

Compile error.

Function Return Rule

No explicit lifetime annotations exist.

If a function returns a reference, that reference must clearly originate from the first reference parameter.

Allowed

fn Name(user: ref User) ref string {
    return ref user.name
}

Allowed

fn Item(list: ref List[Item], index: int) ref Item {
    return ref list[index]
}

Not allowed

fn Pick(a: ref User, b: ref User, useA: bool) ref User

because the compiler cannot determine a single originating lifetime.

Instead, return

an owned value,
an index,
a handle,
or redesign the API.
Design Principle

When ownership becomes ambiguous, the language prefers explicit ownership over increasingly complex borrow analysis.

The language should guide the programmer toward simple, safe APIs rather than making the borrow checker increasingly sophisticated.

Interfaces

Sec uses interfaces to describe behavior.

Interfaces are explicitly implemented.

Example

type Reader interface {
    fn read(buffer: []byte) Result[int, IOError]
}

Implementing an interface

type FileReader struct {
    handle: unsafe *void
}

impl Reader for FileReader {

    fn read(buffer: []byte) Result[int, IOError] {
        ...
    }
}

Adding normal methods

impl FileReader {

    fn close() Result[void, IOError] {
        ...
    }
}

Rules

Interfaces define behavior, never inheritance.
A type does not automatically implement an interface.
Every interface implementation must be explicit.
The compiler verifies that every required member exists.
Missing or incompatible members produce compile-time errors.
Separation of Data and Behavior

A fundamental design principle of Sec is that data and behavior are intentionally separated.

A struct describes what an object is.

An impl block describes what an object can do.

An interface describes what behavior other code may rely on.

Example

type FileReader struct {
    handle: unsafe *void
}

The structure contains only data.

Behavior is defined separately.

impl FileReader {

    fn close() Result[void, IOError] {
        ...
    }

    fn seek(offset: int) Result[void, IOError] {
        ...
    }
}

Contracts are also defined separately.

impl Reader for FileReader {

    fn read(buffer: []byte) Result[int, IOError] {
        ...
    }
}

This separation makes large code bases easier to understand.

When reading code, the programmer immediately knows whether they are looking at:

data,
behavior,
or externally visible contracts.

The goal is to minimize hidden coupling and make every responsibility explicit.

Why impl?

Sec intentionally uses the short keyword impl rather than longer forms such as implements.

Reasons:

It is concise and visually consistent with other core keywords (fn, mut, ref, try).
It is written relatively infrequently—typically once per type or interface implementation.
It clearly marks the beginning of an implementation block.
It separates implementation from declaration, reinforcing the language philosophy that types describe data, while implementation blocks describe behavior.

Modules and Project Layout

Sec distinguishes between language concepts and project concepts.

Each concept has exactly one responsibility.

module

A module is a language construct.

Every source file belongs to exactly one module.

Example

module internal.storage

or

module cmd.projA

A module defines the namespace visible to the compiler.

sec.module

sec.module defines the project.

Example

module myproject

sec 0.1

require github/foo/bar 1.2.0
require company/database 2.1.0

The project file contains:

Project name
Language version
External dependencies
Build configuration (future)
Package metadata (future)

It is not part of the language itself.

Project Layout

A recommended project layout is

myproject/

    sec.module

    cmd/
        projA/
            main.sec

        projB/
            main.sec

    internal/
        storage/
        network/
        parser/

    modules/
        crypto/
        http/
        database/
cmd

cmd contains executable applications.

Each directory represents one executable.

Example

cmd/projA
cmd/projB

Both contain

fn main() {
}

but belong to different modules.

Example

module cmd.projA
module cmd.projB

Unlike Go, Sec does not require every executable to belong to a generic main package.

The module name reflects the logical location of the executable.

internal

internal is a reserved language concept.

Modules inside internal are always private to the current project.

Imports are therefore intentionally short.

Example

import (
    "internal/storage"
    "internal/network"
)

The current project name is implicit.

If an external dependency also contains an internal directory, it must always be qualified.

Example

import (
    "github/foo/internal/storage"
)

This keeps local imports short while avoiding ambiguity.

modules

modules contains reusable project modules.

Example

import (
    "modules/database"
    "modules/http"
)

Modules may later be published independently.

Standard Library

Standard library modules are imported directly.

Example

import (
    "fmt"
    "json"
    "time"
)

The compiler resolves these automatically as standard library modules.

Import Philosophy

Imports should immediately reveal where code originates.

fmt                    Standard Library

internal/storage       Local private module

modules/database       Local reusable module

github/foo/bar         External dependency

No additional prefixes are required for local project code.

The origin of every imported module is immediately obvious.

Design Philosophy

The module system follows the same principles as the language itself.

module is a language concept.
sec.module defines the project.
internal defines project-private visibility.
modules contains reusable project code.
cmd defines executable entry points.

Each concept has exactly one responsibility.

Nothing serves two different purposes.

This separation makes both the compiler and the project structure predictable, scalable, and easy to understand.
