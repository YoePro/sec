Sec Manual / Impl Blocks

Impl Blocks

An impl block defines behavior for an existing named type. The type declaration defines the type itself, while the impl block defines operations associated with that type.

type Person struct {
    name: string,
    age: int,
}

impl Person {
}
Rule: A type defines data and identity. An impl block defines behavior.

Syntax

impl TypeName {
    members
}

The target must already be a known named type. An empty impl block is valid.

Allowed Targets

An impl block may target structs, enums, unions, registers, named primitive-derived types, units and generic named types.

impl Vehicle {
}

impl Color {
}

impl Result[T, E] {
}

impl Speed {
}

The target must exist and must be a named type.

impl MissingType {
}

Impl Members

An impl block may contain methods, properties, nested named types, nested enums, nested unions, nested units and constants when implemented.

impl Vehicle {
    type Engine struct {
        power: Kilowatt,
    }

    enum FuelType {
        petrol,
        diesel,
        electric,
    }

    property TopSpeed: Speed {
        get {
            return _topSpeed
        }
    }

    fn Stop(ref mut self) void {
        self._topSpeed = Speed(0)
    }
}

An impl block cannot contain stored fields or ordinary variable declarations.

impl Vehicle {
    let currentSpeed: Speed
}
Invalid: Stored data belongs in the type declaration, not in the impl block.

Self

Instance methods declare a receiver named self.

Shared receiver

impl Person {
    fn Print(ref self) void {
        fmt.Println(self.name)
    }
}

A shared receiver may read from the value but may not mutate it.

Mutable receiver

impl Person {
    fn SetAge(ref mut self, age: int) void {
        self.age = age
    }
}

A mutable receiver may modify the target value.

Static methods

impl Person {
    fn Create(name: string, age: int) Person {
        return Person {
            name: name,
            age: age,
        }
    }
}
Rule: self is available only when explicitly declared as a method receiver.

Methods

Methods are functions associated with the impl target.

impl Vehicle {
    fn Drive(ref mut self, speed: Speed) void {
        self._speed = speed
    }

    fn CurrentSpeed(ref self) Speed {
        return self._speed
    }
}
vehicle.Drive(Speed(50))
let speed := vehicle.CurrentSpeed()

Conceptually, methods are registered under qualified names such as Vehicle.Drive and Vehicle.CurrentSpeed.

All method signatures are registered before method bodies are analyzed, so forward references are allowed.

impl Vehicle {
    fn Start(ref mut self) void {
        self.Prepare()
    }

    fn Prepare(ref mut self) void {
    }
}

Additional method-level generic parameters are not part of the initial implementation.

Properties

Properties are controlled or computed members declared inside an impl block.

impl Vehicle {
    property TopSpeed: Speed {
        get {
            return _topSpeed
        }

        try set value {
            _topSpeed = value
            return Ok()
        }
    }
}

A property may contain get, set value or try set value. It must contain at least one getter or setter.

A getter must return a value assignable to the property type.

property Bad: Speed {
    get {
        return Money(5.90)
    }
}

A fallible property assignment requires try:

try car.TopSpeed = currentSpeed {
    Err(error) => fmt.Println("could not set top speed")
}

Nested Types

Associated nested declarations are scoped under the impl target type.

type Vehicle struct {
    engine: Vehicle.Engine,
    fuel: Vehicle.FuelType,
}

impl Vehicle {
    type Engine struct {
        power: Kilowatt,
    }

    enum FuelType {
        petrol,
        diesel,
        electric,
    }
}

Outside the impl block, nested types require qualified names such as Vehicle.Engine, Vehicle.FuelType and Vehicle.FuelType.petrol.

Inside impl Vehicle, short names such as Engine, FuelType and FuelType.petrol are available.

Rule: Nested declarations belong to the target type namespace.

Nested named types still use normal declaration syntax.

impl Vehicle {
    type Engine struct {
        power: Kilowatt,
    }
}

This form is invalid:

impl Vehicle {
    struct Engine {
        power: Kilowatt,
    }
}

Generic Impl Blocks

type Stack[T] struct {
    items: Vec[T],
}

impl Stack[T] {
    fn Push(ref mut self, value: T) void {
        self.items.Add(value)
    }

    fn Pop(ref mut self) Option[T] {
        return self.items.Pop()
    }
}

The generic parameters must match the target declaration. Generic impl members are specialized together with the concrete target type.

Interface Implementation

Interfaces use a separate explicit form:

impl Reader for FileReader {
    fn Read(ref mut self, buffer: []byte) Result[int, IOError] {
    }
}

This is not the ordinary impl block for FileReader. A type may have one ordinary impl block and separate explicit interface implementations.

Name Lookup

Impl members belong to the target type namespace.

Vehicle.Drive
Vehicle.TopSpeed
Vehicle.Engine
Vehicle.FuelType

All member names are registered before bodies are analyzed. This allows methods and properties to reference members declared later.

Impl Analysis Order

Pass 1: Register target and nested names

  • verify that the target exists
  • verify that the target is a named type
  • validate generic target arguments
  • register nested type names
  • reject duplicate nested type names

Pass 2: Resolve nested declarations

  • nested structs
  • nested enums
  • nested unions
  • nested units
  • nested aliases when supported

Pass 3: Register behavioral members

  • method signatures
  • property signatures
  • constants when implemented

Pass 4: Analyze member bodies

  • method bodies
  • property getters
  • property setters
  • fallible property setters
  • constant initializers when implemented
Rule: Member bodies are never analyzed before the complete member surface is known.

Restrictions

An impl block must not add stored fields, contain member-level variables, contain free-standing executable statements, redefine an existing member, change the target type definition or silently implement another type.

impl Vehicle {
    let speed: Speed
    speed = Speed(10)
}

A type may have only one ordinary impl block.

impl Vehicle {
}

impl Vehicle {
}
Design rule: One ordinary impl block keeps a type's behavior in one predictable namespace.

Visibility

Impl members follow normal visibility rules. Members and nested declarations may be public, module-visible or private.

A public member must not expose an inaccessible type in its public signature.

Runtime Model

impl is a compile-time construct. It does not create a runtime object, wrapper, namespace value or allocation.

Rule: impl has no runtime representation.

Summary of Rules

  • impl Type { ... } defines behavior for an existing named type.
  • A type may have only one ordinary impl block.
  • Interface implementations use impl Interface for Type.
  • Impl blocks may contain methods, properties and associated nested declarations.
  • Stored fields and free-standing statements are not allowed.
  • Instance methods explicitly declare ref self or ref mut self.
  • Static methods do not declare self.
  • Method and property signatures are registered before bodies are analyzed.
  • Forward references between members are allowed.
  • Nested declarations belong to the target type namespace.
  • Nested names must be qualified outside the matching impl block.
  • Generic impl parameters must match the target generic type.
  • Duplicate members and nested names are invalid.
  • impl has no runtime representation.
