Semantic rules: named types, contracts and units
================================================

Implementation status
---------------------

Implemented now:

- named range types and named unit types are real semantic types.
- sema reads all type declarations before checking let/assignment statements.
- named type registry stores name, underlying type, unit, range contracts and named=true.
- numeric literals can initialize named range/unit types in declaration context.
- variables are not implicitly converted into named range/unit types.
- explicit conversion syntax exists:

    Percent(x)

- compile-time known conversions into range-constrained types are checked at compile time.
- compound assignment between different named types is rejected with a type error:

    cannot add Speed to Money

- decimal unit dimensions are represented internally as:

    Dimension{Base: map[string]int}

- decimal division/multiplication can derive dimensions.
- if a declared decimal type exists for a derived dimension, sema can infer that type:

    Meter / Second -> Speed

- decimal literals infer as decimal by default:

    let x := 3.14

- explicit float declarations can still use decimal literals:

    let f: float64 := 3.14

- currency squared is rejected for now:

    Money * Money

- unit declaration shorthand is accepted:

    unit rpm physical
    unit request other

  The omitted storage type defaults to decimal.

- when an algebraic expression is checked against an expected named unit type,
  the expected type is used if the derived dimension matches. This avoids
  arbitrary alias selection.

- when no target type exists, matching named unit types are selected
  deterministically and unit declarations are considered after non-unit named
  types. Anonymous dimensions are preserved when no named type matches.

Not implemented yet:

- runtime contract checks for non-constant explicit conversions.
- strict rejection of raw literals in assignment after declaration.
- full unit algebra diagnostics and all invalid algebra cases beyond the currently implemented ones.
- advanced unit metadata in impl blocks:

    impl Unit {
        dimension: [...]
        scale: ...
        system: ...
    }

- scale-sensitive algebra and fixed unit conversions.
- exponent syntax in ordinary unit expressions such as decimal<m^2> or decimal<m²>.


1. Named type identity
----------------------

A user-defined type is a distinct type, even if it has the same storage type.

Example:

    type Percent int range 0..100
    type Age int range 0..130

Percent and Age are different types.

This must be rejected:

    let a: Age := 20
    let p: Percent := a

Reason:
Variables already have a type. They are never implicitly converted into another named type.


2. Untyped literals
-------------------

Untyped numeric literals may be assigned directly to a named type if they fit the underlying type and satisfy all contracts.

Valid:

    let p: Percent := 90
    let m: Money := 5.90
    let s: Speed := 50

Invalid:

    let p: Percent := 101
    let p: Percent := -1

Important:
The literal is checked against the target type before assignment.


3. Explicit conversion
----------------------

Conversion from another variable/expression into a named type must be explicit.

Valid:

    let x := 90
    let p: Percent := Percent(x)

Invalid:

    let x := 90
    let p: Percent := x

If x is compile-time known, Percent(x) may be checked at compile time.
If x is not compile-time known, Percent(x) must generate a runtime contract check.


4. Assignment
-------------

Assignment requires the assigned value to match the target variable type.

Valid:

    let mut p: Percent := 10
    p = Percent(20)

Invalid:

    let mut p: Percent := 10
    p = 20

Questionable but recommended invalid:
Do not allow raw literals in assignment to named types after declaration.

Reason:
Declaration context may shape an untyped literal, but assignment should preserve strict type identity.

So:

    let mut p: Percent := 10
    p = 20          // error
    p = Percent(20) // ok


5. Compound assignment
----------------------

Compound assignment is equivalent to assignment after applying the operator.

    p += x

means:

    p = p + x

Rules:
- p must be mutable.
- operator must be valid for p and x.
- result type must be assignable to p.
- contracts must still hold.

Example:

    type Percent int range 0..100

    let mut p: Percent := 0
    p += Percent(50)  // ok
    p += Percent(51)  // compile-time error if p is known to be 50

Recommended:
Do not allow:

    p += 50

Require:

    p += Percent(50)

unless we explicitly decide that compound assignment accepts untyped literals in target context.


6. Contracts
------------

A contract restricts allowed values of a named type.

Example:

    type Percent int range 0..100

Valid:
    0
    50
    100

Invalid:
    -1
    101

For range contracts:
- inclusive range: 0..100 includes both 0 and 100
- exclusive upper range: 0..<100 includes 0 but excludes 100
- open lower: ..100
- open upper: 0..

Compile-time constants must be checked at compile time.

Runtime values must generate runtime checks when explicitly converted into a constrained type.


7. Units
--------

A decimal unit creates a dimension.

Examples:

    type Money decimal<SEK>
    type Meter decimal<m>
    type Second decimal<s>
    type Speed decimal<m/s>

Units are part of type semantics.

Addition and subtraction:
- allowed only between identical units/dimensions.
- normally also require same named type.

Invalid:

    let mut mo: Money := 5.90
    let sp: Speed := 50
    mo += sp

Error:

    cannot add Speed to Money


8. Unit algebra
---------------

For multiplication and division, units are algebraic.

Rules:
- multiplication adds unit exponents
- division subtracts unit exponents
- equal numerator and denominator cancel out

Examples:

    Meter / Second -> decimal<m/s>
    Speed * Second -> decimal<m>
    Meter / Meter -> decimal
    Money / Money -> decimal
    Money * int -> Money
    Money / int -> Money

Recommended invalid:

    Money * Money

unless there is an explicit use case for SEK^2.


9. Matching derived unit types
------------------------------

If an operation produces a unit dimension and a named type exists for that dimension, sema may resolve to that named type.

Example:

    type Meter decimal<m>
    type Second decimal<s>
    type Speed decimal<m/s>

    let d: Meter := 100
    let t: Second := 10
    let v := d / t

v may infer as Speed.

If no named type exists, v may infer as anonymous decimal<m/s>.


10. Type inference
------------------

Untyped integer literal:

    let x := 10

Recommended inferred type:

    int

Untyped decimal literal:

    let x := 3.14

Recommended inferred type:

    decimal

Not float64.

Reason:
Sec decimal is exact and should be the safe default. Float must be explicit:

    let f: float64 := 3.14


11. Implementation structure
----------------------------

Suggested sema structures:

    Type

Checks should be implemented in this order:

1. Register builtin types.
2. Register named type declarations.
3. Resolve type references.
4. Check let declarations.
5. Check assignments.
6. Check expression types.
7. Check contracts.
8. Check unit algebra.

=====================================================================
Appendix - Unit Algebra and Semantic Units
=====================================================================

The unit system is designed around semantic correctness rather than
simple numeric conversions.

Units are semantic numeric types. They remain nominal types even when
their representation type is the same.

Unit categories:

    physical
        Units that participate in physical dimensional analysis.

    other
        Domain-specific units outside the physical SI system.

Physical and other units both participate in unit algebra. The
difference is that physical units may have known canonical dimensions
and fixed conversion scales, while other units preserve domain meaning
and require explicit configured or runtime conversions when no fixed
relationship exists.

Unit conversions are explicit. Numeric casts and unit conversions are
different operations:

    numeric cast
        Changes numeric representation.

    unit conversion
        Changes semantic unit while preserving correctness.

Automatic conversion is only permitted when the relationship is known
by the compiler to be fixed and exact. Runtime or configured
relationships require explicit conversion functions.

=====================================================================
Appendix - Unit Definitions
===========================

A unit declaration defines a semantic numeric type.

Example:

```
unit m decimal physical
```

The unit implementation defines how the unit participates in the unit
system.

Example:

```
impl m {
    dimension: [length * 1]
    scale: 1
    system: SI
}
```

The declaration:

```
unit m decimal physical
```

defines:

```
m
    The name and symbol of the unit.

decimal
    The numeric base type used to represent values of the unit.

physical
    The unit category. Physical units participate in physical
    dimensional algebra.
```

The implementation:

```
impl m {
    dimension: [length * 1]
    scale: 1
    system: SI
}
```

defines:

```
dimension
    The unit's dimensional vector.

scale
    The scale relative to the canonical unit of the dimension.

system
    The unit system to which the declaration belongs.
```

The metre is the canonical SI unit for length. Its scale is therefore 1.

A millimetre may be defined as:

```
unit mm decimal physical

impl mm {
    dimension: [length * 1]
    scale: 0.001
    system: SI
}
```

An inch may be defined as:

```
unit inch decimal physical

impl inch {
    dimension: [length * 1]
    scale: 0.0254
    system: Imperial
}
```

Both units have the same dimensional vector. The compiler can therefore
verify that conversions between them are dimensionally valid.

The scale values provide an exact conversion path through the canonical
unit:

```
inch -> m -> mm
```

The compiler may simplify this path to the exact relationship:

```
1 inch = 25.4 mm
```

Derived physical units use multiple dimensions and integer exponents.

Example:

```
unit mps decimal physical

impl mps {
    dimension: [length * 1, time * -1]
    scale: 1
    system: SI
}
```

A derived unit may also be matched against an algebraic expression.

Example:

```
distance / duration
```

If distance has the dimension:

```
[length * 1]
```

and duration has the dimension:

```
[time * 1]
```

the resulting dimension is:

```
[length * 1, time * -1]
```

The result may therefore be represented as mps when the numeric scale
also matches.

A frequency unit may be defined as:

```
unit Hz decimal physical

impl Hz {
    dimension: [time * -1]
    scale: 1
    system: SI
}
```

A rotational frequency may be defined as:

```
unit RPM decimal physical

impl RPM {
    dimension: [time * -1]
    scale: 1 / 60
    system: Other
}
```

Because Hz and RPM have the same dimensional vector and a fixed scale
relationship, the compiler can verify and perform an explicit unit
conversion between them.

The unit system name does not determine dimensional compatibility.

For example, SI, Imperial and older physical systems may contain units
with compatible dimensions. Compatibility is determined by the
dimensional vector, while conversion is determined by scale and, where
applicable, offset.

=====================================================================
Other and Domain Units
======================

Units that do not describe physical dimensions may use the other
category.

Example:

```
unit Item uint other

impl Item {
    dimension: [item * 1]
    system: Domain
}
```

Another semantic count must use a separate dimension.

Example:

```
unit Packet uint other

impl Packet {
    dimension: [packet * 1]
    system: Domain
}
```

Item and Packet may use the same numeric base type, but they are not
compatible units.

This is invalid:

```
let total := items + packets
```

Domain units may still participate in algebra.

Example:

```
Packet / second
```

produces the dimension:

```
[packet * 1, time * -1]
```

Similarly:

```
Item / Batch
```

produces:

```
[item * 1, batch * -1]
```

A domain relationship is not necessarily known at compile time.

For example:

```
1 batch = 24 items
```

may be configured by the program, while:

```
measured events / elapsed time
```

produces a runtime cadence.

The compiler verifies the dimensional algebra but does not invent
configured or measured values.

Currencies are domain units without a physical SI relationship.

Example:

```
unit SEK decimal other

impl SEK {
    dimension: [currency.SEK * 1]
    system: Currency
}

unit EUR decimal other

impl EUR {
    dimension: [currency.EUR * 1]
    system: Currency
}
```

SEK and EUR are separate semantic dimensions.

A conversion between them requires an explicit runtime or configured
exchange rate. The compiler must not treat it as a fixed unit
conversion.

=====================================================================
Current Implementation Status
=====================================================================

Implemented in the front-end and semantic analyzer:

```
unit Name int
unit Name uint
unit Name float
unit Name decimal
unit Name decimal<unit-expression>
unit Name decimal physical
unit Name uint other
unit Name physical
unit Name currency
unit Name other
```

The optional storage/default numeric type must be numeric. If it is omitted,
the unit defaults to `decimal`.

`unit Name physical` therefore means:

```
unit Name decimal physical
```

Units are unit symbols, not inherently decimal types. Numeric unit types are
written explicitly as `decimal<m>`, `float<m>`, `int<m>` and so on. The shorthand
`<m>` is allowed in type position and expands to the unit's default numeric
representation, unless a clearly typed numeric initializer such as `5f` selects
another numeric representation.

Implemented built-in physical aliases:

```
m, metre, meter
mm, millimetre, millimeter
inch
s, second
kg, kilogram
Hz, Hertz, hertz
rpm
```

Known built-in physical aliases map to internal dimensional vectors.
Unknown units keep their own semantic base dimension and default to the
other category unless explicitly declared as physical.

Implemented unit algebra is still compile-time semantic checking only.
No new LLVM lowering has been added for this appendix. Future lowering
will go through MLIR before LLVM IR.

Still not implemented:

```
fixed conversion factors such as inch -> millimetre
explicit unit conversion validation through impl constructors
configured or runtime conversion relationships
offset units
exponent syntax in unit expressions
MLIR representation of dimensional vectors
```

=====================================================================
Advanced Unit Metadata and Explicit Conversions
=====================================================================

This section refines the intended unit-definition model and supersedes older
examples that wrote dimension entries as `length * 1`.

Canonical dimension-vector syntax
---------------------------------

A unit implementation uses integer exponents:

```
impl m {
    Dimension: [length^1]
    Scale: 1
    System: SI
}

impl mps {
    Dimension: [length^1, time^-1]
    Scale: 1
    System: SI
}
```

Unit impl blocks may also contain static functions. A static function has no
`self` parameter and can be used as a named constructor or explicit conversion
helper.

Example:

```sec
/** Coulomb */
unit C decimal physical

impl C {
    // A * s.
    dimension: [electric_current^1, time^1]
    scale: 1
    system: SI

    fn C(Amp: decimal<A>, Second: decimal<s>) decimal<C> {
        return decimal<C>(decimal(Amp) * decimal(Second))
    }
}
```

Parameter syntax is always `Name: Type`. The form `Second decimal<s>` is not
valid.

Unit names and other Sec identifiers may use Unicode letters. This allows
domain-standard symbols such as `Ω`, `μ`, `Σ` or `π` when they improve clarity.

Guidance:

- prefer ASCII for ordinary API names and long-lived public functions
- use Unicode unit symbols when the symbol is the established scientific name
- avoid visually confusable names in the same scope
- formatter and tooling must preserve Unicode identifiers exactly

The canonical formatter keeps explicit exponents, including `^1`, because this
makes every dimension entry structurally uniform.

Dimension rules:

- every exponent is a non-zero compile-time integer
- dimension order is not semantically significant
- each dimension axis may occur only once in a declaration
- the compiler normalizes dimensions before comparing them
- multiplication adds exponents
- division subtracts exponents
- axes whose resulting exponent is zero are removed
- an empty normalized vector is dimensionless

The seven SI base dimensions use these canonical axis names:

```
length
mass
time
electric_current
thermodynamic_temperature
amount_of_substance
luminous_intensity
```

Additional physical axes may be standardized later. Domain units may use
qualified semantic axes such as `currency.SEK`, `item`, `packet` and `batch`.

Unit declaration shorthand
--------------------------

The complete declaration remains:

```
unit rpm decimal physical
```

The following shorthand is also part of the intended design:

```
unit rpm physical
```

When the numeric storage type is omitted and the next token is `physical` or
`other`, the storage type defaults to `decimal`.

Current implementation:

- this shorthand is implemented in the parser and semantic analyzer.

Integer-backed domain quantities should still declare their storage explicitly:

```
unit Item uint other
```

Required metadata
-----------------

A physical unit implementation declares:

```
dimension
Dimension
Scale
System
LongName
Symbol
BaseUnit
Status
```

The scale is relative to the canonical unit for the normalized dimension.
It must be a positive, non-zero compile-time numeric expression.
`Scale: 1` is valid and preferred for canonical units. Suffixes such as `1d`
are allowed but not required for exact integer scale values.

Metadata names are case-tolerant and underscore-tolerant in source. The
recommended stdlib spelling is PascalCase (`BaseUnit`, `LongName`, `Scale`),
while older forms such as `base_unit`, `baseunit`, `long_name` and `scale`
remain accepted.

`Status` accepts:

```
active
deprecated
obsolete
```

The default status is `active`.

Examples:

```
impl mm {
    Dimension: [length^1]
    Scale: 1 / 1000
    System: SI
}

impl inch {
    Dimension: [length^1]
    Scale: 127 / 5000
    System: Imperial
}

impl rpm {
    Dimension: [time^-1]
    Scale: 1 / 60
    System: Other
}
```

The unit system does not determine compatibility. Dimension determines physical
compatibility; scale determines the fixed numeric relationship; the unit name
preserves nominal meaning.

Domain units must declare at least `dimension` and `system`. A scale of `1` may
be explicit for canonical domain quantities. Currencies use separate dimensions
unless a future currency model explicitly defines otherwise.

Derived units
-------------

Derived units declare their complete normalized dimension and canonical scale.
Examples:

```
impl N {
    dimension: [mass^1, length^1, time^-2]
    scale: 1
    system: SI
}

impl Pa {
    dimension: [mass^1, length^-1, time^-2]
    scale: 1
    system: SI
}

impl V {
    dimension: [mass^1, length^2, time^-3, electric_current^-1]
    scale: 1
    system: SI
}
```

The compiler validates algebra using both the normalized dimension and the
resulting scale. For example:

```
km / h -> kph
m / s  -> mps
kN * m -> kJ
```

`km / h` must not silently become `mps`; the dimensions match but the scales and
nominal units differ. An explicit conversion is required.

Multiple units with the same dimension
--------------------------------------

Multiple units may share one dimension:

```
Hz   [time^-1]
Bq   [time^-1]
rpm  [time^-1]
```

They remain distinct nominal units.

When an algebraic expression has a target type, the compiler may validate the
expression against that target's dimension and scale.

When no target type exists and several named units match the same normalized
dimension and scale, the compiler must preserve an anonymous dimensional result
rather than selecting an arbitrary named unit.

Explicit unit conversion functions
----------------------------------

A unit may declare explicit conversion functions in its ordinary `impl` block.
A conversion function whose name equals the target unit participates in
constructor-style conversion resolution.

Example:

```
unit rpm physical

impl rpm {
    dimension: [time^-1]
    scale: 1 / 60
    system: Other

    fn rpm(hz: Hz) rpm {
        return hz * 60
    }
}
```

Usage:

```
let frequency: Hz := 2
let rotational: rpm := rpm(frequency)
```

Conversion-function rules:

- the function is static and has no `self` receiver
- the function name equals its containing target unit
- the return type equals its containing target unit
- the first and initially only parameter is a different unit type
- source and target must have compatible normalized dimensions
- conversion is invoked explicitly; assignment alone never calls it
- conversion overloads are resolved by the exact source unit type
- duplicate conversions from the same source unit are invalid
- ordinary numeric construction remains available for numeric literals and
  explicitly permitted numeric expressions

The conversion-function body defines the numeric conversion formula. Inside
such a function, the return expression may retain the source unit while the
function boundary explicitly retags the dimension-compatible numeric result as
the target unit. This special permission applies only to the return path of a
validated unit conversion function.

This permits fixed-scale conversions:

```
fn rpm(hz: Hz) rpm {
    return hz * 60
}
```

and also permits future affine conversions, such as temperature scales, without
making ordinary arithmetic silently discard unit identity.

Scale paths and direct conversion functions
-------------------------------------------

A fixed scale path and a direct conversion function may both exist.

- scale metadata allows the compiler to verify dimensional relationships
- a direct function is used only when the programmer explicitly calls it
- the compiler does not silently replace assignment with a conversion call
- an explicitly called conversion function is authoritative for that call

The compiler may warn when an explicit conversion formula appears redundant,
but it must not reject it merely because a scale path exists.

Current implementation boundary
-------------------------------

The first advanced unit pass is implemented:

- `unit Name physical` and `unit Name other` shorthand is accepted.
- `unit Name currency` shorthand is accepted.
- `unit Name` defaults to `decimal`.
- `<unit>` shorthand is accepted in type position.
- `impl Unit { dimension/scale/system }` metadata is accepted by the parser.
- `dimension: [...]` is registered in sema and used by unit algebra.
- `system: Name` is stored as unit metadata.
- `scale: ...` is accepted as metadata syntax, but scale arithmetic is not yet
  used when selecting algebra results.
- `LongName`, `Symbol`, `BaseUnit`, and `Status` metadata is stored in sema.
- `Status: deprecated` and `Status: obsolete` produce warnings when the unit is
  used.
- integer-backed domain units such as `Item uint other` can participate in
  dimensional algebra and produce decimal derived units.
- conversion functions declared as `fn TargetUnit(source: SourceUnit) TargetUnit`
  inside `impl TargetUnit` are visible as explicit constructor-style calls.
- return expressions from those conversion functions may retag compatible
  dimensional results through the declared return type.

Still not implemented:

- compile-time validation that every physical unit has complete metadata.
- duplicate dimension-axis diagnostics beyond the current parser helper.
- scale-sensitive algebra and fixed scale-path conversion.
- validation that conversion functions have compatible source/target dimensions.
- offset units.
- MLIR representation of dimensional vectors.
