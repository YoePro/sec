Lifetime Analysis
=================

Lifetime analysis proves that every non-owning access remains valid for its complete use.

The compiler determines:

* when a value becomes initialized
* when its semantic lifetime begins
* when its semantic lifetime ends
* how long each reference may remain valid
* whether a reference escapes its valid region
* whether destruction invalidates an active reference
* whether storage remains valid after ownership transfer
* whether control-flow paths produce compatible lifetime states
* whether a returned reference is backed by sufficiently long-lived storage
* whether an FFI or unsafe operation requires stronger lifetime obligations

Sec should infer lifetimes whenever possible.

Ordinary Sec code should not require programmer-written lifetime parameters or lifetime annotations.

## Design goals

Lifetime analysis must be:

* compile-time
* flow-sensitive
* path-sensitive
* ownership-aware
* borrow-aware
* destruction-aware
* storage-aware
* compatible with deterministic cleanup
* compatible with separate compilation
* independent of garbage collection
* independent of runtime borrow tracking
* independent of hidden heap promotion
* explicit enough for compiler verification
* mostly invisible in source code

The compiler carries the complexity.

The programmer expresses:

* ownership
* references
* mutability
* control flow
* explicit storage and allocation choices

The compiler derives valid lifetimes from those semantics.

## Core rule

A reference must never be usable beyond the valid lifetime of the referenced value and storage.

Conceptually:

```text
reference lifetime <= referent lifetime
```

This relation is semantic.

It does not require a source-level lifetime name.

## Terminology

The following concepts are distinct:

* value lifetime
* storage lifetime
* reference lifetime
* borrow lifetime
* lexical scope
* ownership duration
* temporary lifetime
* resource lifetime
* allocation lifetime
* physical stack-frame lifetime
* LLVM storage-lifetime metadata

These terms must not be treated as interchangeable.

## Value lifetime

A value lifetime begins when a value becomes fully or validly initialized.

A value lifetime ends when the value is:

* destroyed
* moved without remaining ownership at the source
* replaced
* transferred out of its storage
* invalidated through a defined operation
* no longer represented by an active variant
* no longer initialized on a particular control-flow path

For partially initialized aggregates, fields may have separate value lifetimes.

For partially moved aggregates, remaining fields continue to have valid lifetimes.

## Storage lifetime

Storage lifetime is the period during which a storage location may physically hold values.

Examples include:

* stack-slot lifetime
* global storage lifetime
* static data lifetime
* heap allocation lifetime
* caller-provided return storage lifetime
* aggregate field storage lifetime
* array-element storage lifetime
* fixed-address external storage lifetime
* register-mapped storage lifetime

Storage may outlive a particular value stored within it.

Example:

```sec
mut value := CreateFirst()
value = CreateSecond()
```

The binding's storage may remain the same.

The first value lifetime ends before the second value lifetime begins.

## Reference lifetime

A reference lifetime is the region in which a reference may be validly used.

A reference lifetime:

* cannot begin before the referent is initialized
* cannot extend beyond referent validity
* cannot extend beyond required storage validity
* cannot survive an incompatible move
* cannot survive destruction
* cannot survive replacement of the referenced value
* cannot survive deallocation of referenced storage
* must respect mutable and shared borrow conflicts

The physical bits of a reference may remain in a register or memory after its semantic lifetime ends.

Safe Sec code must not use it after that point.

## Borrow lifetime

A borrow lifetime is the period during which borrow restrictions apply to the referent.

A borrow lifetime is usually determined by actual use rather than lexical scope alone.

The compiler should use the narrowest provably correct borrow lifetime.

Example:

```sec
let value := CreateValue()
let view := ref value

Inspect(view)
Consume(value)
```

The shared borrow may end after `Inspect(view)` if `view` is not used again.

The borrow need not remain active until the end of the lexical scope.

## Resource lifetime

A resource lifetime may differ from the lifetime of a wrapper's storage.

Example:

```sec
mut file := OpenFile()
file.Close()
```

If `Close` consumes or transitions the resource:

* the operating-system handle lifetime may end
* the wrapper storage may remain
* the wrapper may become unavailable or enter a closed state
* later destruction must not release the handle again

Lifetime analysis must use the type's semantic state, not assume storage validity implies resource validity.

## Allocation lifetime

An allocation lifetime begins when allocation succeeds.

It ends when the allocation is deallocated.

Owned values within the allocation may have narrower lifetimes.

Deallocation is invalid while safe references into the allocation remain active.

A RawPtr does not extend allocation lifetime.

## Lexical scope

Lexical scope is a source-code region.

Lexical scope provides an upper bound for many local bindings, but it does not fully define semantic lifetime.

A value may end before scope exit because it is:

* moved
* destroyed
* replaced
* no longer initialized
* proven dead and safely destroyed early

A borrow may end before scope exit after its final use.

The compiler should not confuse lexical scope with exact lifetime.

## Non-lexical lifetime analysis

Sec should use non-lexical lifetime analysis.

Borrow validity is determined from:

* definition
* uses
* control-flow paths
* ownership operations
* destruction
* mutations
* escapes
* merge states

A borrow ends after its final relevant use when no path requires it later.

This permits ordinary code without artificial nested scopes.

Example:

```sec
mut buffer := Buffer.Create(1024)

let length := ref buffer.length
Print(length)

buffer.Clear()
```

The shared borrow of `buffer.length` may end after `Print(length)`.

The later mutable operation is valid when no reference use remains.

## No ordinary lifetime syntax

The initial Sec language should not require syntax resembling:

```text
'a
'b
lifetime T
where reference outlives X
```

The compiler should infer ordinary local, parameter and return-reference relationships.

Lifetime syntax should be introduced only if concrete APIs cannot be expressed safely and clearly without it.

The absence of source-level lifetime names does not weaken compiler analysis.

## Lifetime sources

The compiler derives lifetime facts from:

* binding declarations
* initialization
* ownership
* moves
* copies
* references
* borrows
* function signatures
* control-flow structure
* destruction points
* cleanup regions
* storage classes
* aggregate structure
* FFI contracts
* unsafe operations
* target and platform constraints

No lower compiler stage may invent lifetime semantics independently.

## Lifetime regions

The compiler may represent inferred lifetimes as regions.

A region is a set of program points where a value, storage location or reference is valid.

Regions need not be contiguous in source text.

A region may be represented by:

* control-flow graph nodes
* program points
* dominance relationships
* post-dominance relationships
* data-flow facts
* region variables
* constraints
* use ranges
* ownership tokens

Region representation is an implementation choice.

Source semantics remain independent of the chosen algorithm.

## Program points

Lifetime analysis should reason at defined program points.

Useful program points include:

* before expression evaluation
* after expression evaluation
* after successful initialization
* before a move
* after a move
* before destruction
* after destruction
* before a branch
* on each control-flow edge
* at block entry
* at block exit
* before a call
* after a call
* during cleanup
* at return transfer

This avoids ambiguity about whether a reference remains valid across an operation.

## Lifetime beginning

A value's lifetime begins only after successful initialization.

Example:

```sec
let file := try OpenFile()
```

On the Ok path:

* file becomes initialized
* file lifetime begins
* file destruction responsibility begins

On the Err path:

* file never becomes initialized
* file has no value lifetime
* file is not destroyed

The binding's potential storage may exist before successful initialization, but no valid File value exists.

## Field lifetime beginning

Aggregate fields may begin their lifetimes separately during construction.

Example:

```sec
let session := Session {
    file: try OpenFile()
    buffer: try Buffer.Create(4096)
}
```

If file succeeds:

* session.file begins a field lifetime

If buffer fails:

* session.file is destroyed
* session as a complete value never begins its lifetime
* session.free is not invoked if custom free requires complete construction

Field-level lifetime tracking must agree with partial initialization rules.

## Lifetime ending through destruction

Destruction ends the value lifetime.

All safe references to the complete value or destroyed portion must have ended before destruction begins.

Example:

```sec
let resource := OpenResource()
let view := ref resource

DestroyingOperation(resource)
Use(view)
```

This is invalid if `DestroyingOperation` consumes resource.

The reference cannot remain valid after ownership ends.

## Lifetime ending through move

A move ends the source value lifetime.

The destination begins or continues the transferred value's ownership lifetime.

The underlying storage may or may not change.

Example:

```sec
let first := Buffer.Create(4096)
let second := first
```

After the move:

* first no longer has a valid Buffer value
* second owns the Buffer
* first cannot be referenced
* references created from first before the move must not remain active across an incompatible move

The semantic value continues through ownership transfer, but the source binding's valid value lifetime ends.

## Lifetime ending through replacement

Replacing a value ends the previous value lifetime.

Example:

```sec
mut current := CreateFirst()
current = CreateSecond()
```

The old value lifetime ends before the new value becomes observably installed.

Any reference to the old value must end before replacement.

The storage may remain valid and then contain the new value.

A reference to the old value must not silently become a reference to the replacement.

## Lifetime ending through variant change

Changing the active variant ends the lifetime of the old payload.

Example conceptual operation:

```sec
mut state := Some(resource)
state = None
```

The resource payload lifetime ends and it is destroyed or moved according to the operation.

A reference to the Some payload cannot survive transition to None.

## Lifetime ending through field replacement

Replacing one field ends that field's previous value lifetime.

References to:

* that field
* any subfield within it
* the complete aggregate when replacement violates shared access

must satisfy borrowing rules.

References to disjoint fields may remain valid where the compiler proves disjointness and mutation is permitted.

## Last use

The last syntactic use is not always the semantic end of a reference.

The compiler must also consider:

* defer captures
* closure captures
* branch uses
* loop backedges
* cleanup operations
* calls that retain references according to contract
* storage of the reference
* returned references
* interface conversion
* FFI escape

A reference may end after its last ordinary expression use only when no hidden or deferred use remains.

## Reference creation

Creating a safe reference requires proving:

* the referent is initialized
* the referent is available
* the storage is valid
* requested mutability is permitted
* no conflicting active borrow exists
* the reference cannot immediately outlive its referent
* alignment and representation requirements hold
* the operation is not prohibited for the storage class

The resulting reference carries an inferred lifetime constraint.

## Shared reference lifetime

A shared reference permits non-mutating access according to the referenced type's rules.

While a shared borrow is active, conflicting operations are prohibited.

These normally include:

* mutable borrow of overlapping storage
* replacement of overlapping value
* destruction of overlapping value
* move of overlapping value
* mutation requiring exclusive access
* deallocation of backing storage

Multiple compatible shared references may coexist.

Their lifetimes may differ.

## Mutable reference lifetime

A mutable reference requires exclusive access to the referenced region for its active lifetime.

While a mutable borrow is active, the compiler normally prohibits:

* another mutable reference to overlapping storage
* a shared reference to overlapping storage
* direct access through the original owner
* move of the referenced value
* destruction of the referenced value
* replacement through another alias
* deallocation of backing storage

The mutable reference's own binding may be move-only.

Reborrowing may temporarily narrow access further.

## Reference lifetime and mutability

Mutability does not extend lifetime.

A mutable reference is not valid merely because it has exclusive access.

It must still remain within:

* referent value lifetime
* storage lifetime
* ownership validity
* allocation lifetime
* platform validity

Likewise, a shared reference does not keep an owner alive automatically.

## References do not own

A reference does not extend the lifetime of its referent through ownership.

Example:

```sec
fn Invalid() ref Value {
    let value := CreateValue()
    return ref value
}
```

This is invalid.

Returning the reference does not move or extend ownership of `value`.

The compiler must not silently allocate value on the heap.

## RawPtr does not extend lifetime

A RawPtr is non-owning and does not participate in safe lifetime extension.

Converting a reference to RawPtr may erase compiler-enforced reference guarantees at the unsafe boundary.

It does not:

* extend allocation lifetime
* preserve borrow validity
* keep the owner alive
* prevent destruction
* prevent relocation
* create automatic cleanup

Using a RawPtr after the underlying lifetime ends is unsafe program behavior.

## Reference-to-RawPtr conversion

When a safe reference is converted to RawPtr:

* the resulting RawPtr has no safe lifetime guarantee beyond the operation's validated region
* the compiler may still conservatively treat the pointer as an escape
* active borrow rules may remain until the last safe or unsafe use where required
* unsafe code assumes responsibility for validity

The compiler must not promise that arbitrary RawPtr uses can be statically tracked.

## RawPtr-to-reference conversion

Creating a safe reference from RawPtr requires unsafe validation.

The programmer or wrapper must uphold:

* non-null requirements
* alignment
* initialized valid representation
* accessible storage
* sufficient lifetime
* correct mutability
* aliasing legality
* provenance where applicable

The resulting safe reference re-enters lifetime and borrow analysis.

Unsafe creation does not exempt later safe uses from reference rules.

## Storage classes

Lifetime analysis must classify storage.

Relevant storage classes include:

* local stack storage
* function parameter storage
* caller-owned argument storage
* caller-provided return storage
* heap-owned storage
* global storage
* static immutable storage
* thread-local storage
* external addressed storage
* volatile storage
* hardware register storage
* foreign-owned storage
* arena storage
* embedded fixed storage

Each class defines different upper lifetime bounds.

## Local storage

A local binding cannot normally outlive the invocation that owns its storage.

A reference to a local cannot:

* be returned
* be stored in longer-lived global storage
* escape through FFI
* be captured by a longer-lived closure
* be transferred to an asynchronously executing context

unless the storage is explicitly moved into a valid longer-lived owner.

The compiler must not infer such storage promotion.

## Parameter storage

Parameter lifetime depends on parameter mode.

An owned parameter is owned by the callee.

A reference parameter borrows caller-backed storage.

A RawPtr parameter has only its declared unsafe contract.

A copied parameter has an independent value.

The compiler must distinguish parameter binding lifetime from backing referent lifetime.

## Owned parameters

An owned parameter's value lifetime begins when ownership transfers to the callee.

The callee may:

* borrow it
* move it
* return it
* store it in another owner
* destroy it

References to the parameter cannot outlive the resulting owner.

Returning a reference to an owned parameter is invalid unless ownership is also returned in a representation that guarantees the reference remains usable, which requires a dedicated self-referential design.

## Reference parameters

A function receiving `ref T` or `ref mut T` must not assume the reference is valid beyond the call unless its signature explicitly returns or stores a related reference under a safe inferred rule.

The caller guarantees validity for the call's required region.

The callee must not retain the reference beyond that contract.

## Parameter lifetime relationships

Some functions return a reference derived from a reference parameter.

Example:

```sec
fn First(value: ref Pair) ref Item {
    return ref value.first
}
```

The compiler may infer that the returned reference cannot outlive `value`.

No explicit lifetime name is required when the relationship is unambiguous.

The exported function interface must record the relationship in compiler metadata.

## Multiple reference parameters

A function returning a reference derived from one of multiple reference parameters may be ambiguous.

Example:

```sec
fn Select(
    condition: bool
    first: ref Value
    second: ref Value
) ref Value {
    if condition {
        return first
    }

    return second
}
```

The returned reference must be valid no longer than both possible input lifetimes.

Conceptually, its lifetime is bounded by the intersection of the relevant valid regions.

The compiler may infer this conservative relationship.

If the relationship cannot be represented safely in exported metadata, the compiler should reject the declaration or require a future explicit mechanism.

## Unambiguous return-source inference

The compiler may infer return-reference relationships when:

* every return reference derives from the same parameter
* every return reference derives from the same static storage
* every return path returns a reference with a compatible upper lifetime bound
* field projection preserves the base reference lifetime
* reborrowing does not extend lifetime
* no local storage participates

This should cover common accessor functions.

## Ambiguous return references

The compiler should reject a returned reference when its valid source cannot be proven.

Examples include references derived from:

* a local value
* a destroyed temporary
* one of several unrelated sources without a representable relationship
* an FFI pointer without sufficient contract
* a conditionally deallocated allocation
* an owner consumed before return

The diagnostic should explain the candidate sources and why no safe return lifetime exists.

## Static storage references

References to immutable static storage may have program-long validity where supported.

Examples include:

* string literals
* compile-time constant tables
* immutable static data
* target-provided immutable tables

A static reference still obeys:

* type validity
* mutability restrictions
* target availability
* module visibility
* concurrency rules

Static lifetime must not be inferred for ordinary globals requiring runtime initialization.

## Mutable global storage

Mutable global references introduce concurrency and initialization complexity.

The initial language should restrict them according to the global-storage rules.

A reference to mutable global storage may have long storage lifetime but still require:

* synchronization
* initialization proof
* exclusive mutable access
* target support

Long storage lifetime does not imply unrestricted safe borrowing.

## Heap storage

A heap allocation may outlive the scope where it was created when ownership transfers to a longer-lived owner.

References into heap storage remain valid only while:

* the allocation remains allocated
* the referenced value remains initialized
* the owner does not perform invalidating mutation or relocation
* borrow rules remain satisfied

The existence of heap storage does not by itself make references returnable.

The compiler must know who owns the allocation after return.

## Owning containers

References into an owning container are bounded by:

* container ownership lifetime
* element lifetime
* allocation lifetime
* operations that may relocate elements
* operations that may replace or remove elements
* mutable borrow conflicts

Example:

```sec
let item := ref vector[0]
vector.Append(newItem)
Use(item)
```

This may be invalid if Append may reallocate and relocate elements.

The type's operation contract must declare invalidation behavior.

## Container invalidation

Operations on containers must specify whether they may invalidate references.

Possible effects include:

* no invalidation
* invalidates references to modified element
* invalidates references at or after an index
* invalidates all element references
* invalidates only mutable references
* invalidates references when reallocation occurs
* preserves references through stable indirection

The compiler cannot infer all container-specific invalidation solely from layout.

The type or compiler-known operation must expose the effect.

## Stable-storage containers

A container may promise stable element addresses.

Such a promise becomes part of its semantic contract.

The compiler may then permit references across operations that preserve:

* element existence
* ownership
* borrow legality
* storage stability

Stable address does not permit references after element removal or container destruction.

## Array references

References to fixed-array elements are bounded by the array's lifetime.

A fixed array does not relocate its elements during ordinary lifetime.

References to disjoint elements may coexist under field- or index-sensitive borrow rules when disjointness is statically proven.

Runtime indices may require conservative overlap assumptions.

## Slice lifetime

A slice is a non-owning view.

Its lifetime is bounded by the backing storage.

Copying a slice does not extend the backing storage lifetime.

Returning a slice requires proving that the backing storage outlives the returned slice.

A slice into a local array cannot be returned.

## String-view lifetime

A borrowed string view is bounded by its backing string storage.

A string literal view may have static validity.

A view into an owned dynamic string may be invalidated by:

* destruction
* move that relocates storage where references cannot follow
* mutation
* reallocation
* replacement
* explicit clearing if storage is invalidated

The String API must define reference invalidation effects.

## Temporary lifetime

Temporary values have compiler-defined lifetimes.

The compiler should keep temporary lifetimes as narrow as possible while preserving:

* reference validity
* expression evaluation
* destruction order
* defer behavior
* call semantics
* observable side effects

A reference to a temporary cannot outlive the temporary.

## Full-expression lifetime

The initial rule may define that expression temporaries remain valid until completion of the enclosing full expression unless ownership moves earlier.

Example:

```sec
Inspect(ref CreateValue())
```

The temporary remains valid for the call.

It is destroyed after the call returns.

The reference cannot escape the call.

## Temporary lifetime extension

Sec should avoid broad implicit temporary lifetime extension.

Example:

```sec
let reference := ref CreateValue()
```

The compiler should initially reject this unless the language explicitly defines ownership storage for the temporary.

Silently extending the temporary to the binding scope complicates destruction and source reasoning.

A programmer needing a longer lifetime should bind the owned value:

```sec
let value := CreateValue()
let reference := ref value
```

## Temporaries returned by reference

Returning a reference to a temporary is invalid.

Example:

```sec
fn Invalid() ref Value {
    return ref CreateValue()
}
```

The temporary is destroyed before or during function exit.

The reference would be invalid.

No hidden allocation may repair the code.

## Expression chains

References produced in expression chains must remain valid for every downstream use.

Example:

```sec
let result := CreateContainer().First()
```

The meaning depends on `First`:

* if it returns a copied value, the temporary container may be destroyed after the call
* if it returns an owned moved value, ownership transfers
* if it returns a reference, the reference cannot survive destruction of the temporary container

The compiler should reject a resulting escaping reference.

## Method receiver lifetimes

A method borrowing `self` may return a reference into self.

The returned reference is bounded by the receiver's referent lifetime.

Example:

```sec
impl Buffer {
    fn Data(ref self) ref byte {
        return ref self.data[0]
    }
}
```

The compiler records that the result derives from self.

The reference cannot outlive the Buffer.

## Mutable receiver return

A method receiving `ref mut self` may return a mutable or shared reference into self.

The resulting borrow may continue after the method call.

During that returned-reference lifetime:

* the original owner remains restricted
* conflicting accesses remain prohibited
* another mutable borrow cannot overlap
* destruction and move remain prohibited

The borrow does not automatically end at call return.

## Consuming receiver

A method consuming self cannot safely return a reference into self unless ownership of backing storage is also transferred into a representation that outlives the reference.

The initial implementation should reject such methods.

Example:

```sec
fn Invalid(self) ref Item {
    return ref self.item
}
```

Self is destroyed or transferred when the method exits.

The returned reference has no owner.

## Reborrowing

Reborrowing creates a new reference derived from an existing reference.

The new reference cannot outlive the original reference.

Conceptually:

```text
reborrow lifetime <= original reference lifetime
```

A mutable reference may be reborrowed:

* as a shorter mutable reference
* as a shorter shared reference

During a mutable reborrow, use of the original mutable reference may be temporarily suspended.

## Mutable reborrow

Example:

```sec
fn Use(value: ref mut Value) void {
    Mutate(value)
    Inspect(ref value)
    MutateAgain(value)
}
```

The shared reborrow used by `Inspect` may temporarily suspend mutation through `value`.

After the shared reborrow ends, mutable access may resume.

The compiler should infer the narrow shared lifetime.

## Reference copies

Copying a shared reference produces another reference with a lifetime no longer than the original.

The compiler may infer a narrower lifetime for the copy.

Copying a mutable reference is prohibited under the initial copy/move rules.

Moving a mutable reference transfers that reference binding's access capability without extending referent lifetime.

## Reference assignment

Assigning a new reference into a mutable reference variable ends the previous reference value lifetime.

Example:

```sec
mut current := ref first
current = ref second
```

Any borrow restrictions associated solely with the previous reference may end after its last use, subject to other references.

The variable storage remains, but it contains a different reference value.

## Reference fields

A struct containing references requires lifetime relationships between the struct value and referenced storage.

Example:

```sec
type View struct {
    value: ref Value
}
```

The View cannot outlive the referenced Value.

Without explicit lifetime syntax, the compiler must determine whether such types are permitted and how relationships are represented.

## Initial reference-field policy

The initial implementation should support reference-containing aggregates only when their lifetime relationships remain local and inferable.

Recommended restrictions:

* a local reference-containing aggregate may be constructed and used within the proven referent lifetime
* it may be passed by value when the callee does not retain it beyond the relationship
* it may be returned only when the compiler can encode its lifetime dependency on input references
* it may not be stored in unrestricted global storage
* it may not be hidden behind erased ownership without lifetime metadata
* it may not outlive any referenced field source

This avoids requiring immediate source-level lifetime parameters.

## Nested reference fields

A type containing another reference-bearing type inherits its lifetime constraints.

The compiler must recursively collect lifetime dependencies.

Layout identity alone is insufficient.

Two values of the same reference-bearing type may have different inferred concrete valid regions.

## Owned and borrowed aggregate fields

An aggregate may contain both owned and borrowed fields.

Example:

```sec
type Parser struct {
    input: ref string
    state: ParserState
}
```

Parser owns state but borrows input.

Its destruction does not destroy input.

Parser's usable lifetime is bounded by input.

Moving Parser transfers its borrowed relationship but does not extend input lifetime.

## Self-referential aggregates

Safe self-referential aggregates require stable storage and special construction rules.

The initial implementation should reject ordinary safe construction where a field references another field in the same movable aggregate.

Example conceptual invalid pattern:

```sec
type SelfView struct {
    data: Buffer
    view: ref byte
}
```

if view points into data.

Moving the aggregate may invalidate the internal reference.

Such designs should use:

* stable indirection
* explicit ownership-separated view types
* unsafe RawPtr with programmer-maintained invariants
* a future pinned-storage feature

## Closure lifetimes

A closure may capture:

* copied values
* moved owned values
* shared references
* mutable references

A closure capturing a reference cannot outlive the referent or borrow.

A closure capturing an owned value by move owns that value and may outlive the original scope according to closure storage.

The compiler must determine capture mode before validating closure escape.

## Non-escaping closures

A closure known to execute only within the current call may safely borrow local values.

Example conceptual use:

```sec
WithLock(ref lock, fn() {
    Use(ref local)
})
```

The compiler may permit this when the closure contract guarantees it does not escape.

The function parameter must encode non-escaping behavior in compiler metadata.

## Escaping closures

A closure stored or returned beyond the current scope cannot capture references to shorter-lived locals.

It may instead capture owned values by move.

Example:

```sec
fn CreateTask() Task {
    let data := CreateData()

    return task fn() {
        Use(data)
    }
}
```

If the closure owns data by move, this may be valid.

The original local binding becomes moved.

The closure's destruction later destroys captured data.

## Asynchronous execution

Any callback or task that may execute after the current scope is an escaping context.

References to locals must not escape into it.

The compiler must not assume immediate execution unless the API contract explicitly guarantees it.

This applies to:

* threads
* task queues
* event loops
* interrupt registrations
* operating-system callbacks
* stored closures
* foreign callbacks

## Defer lifetimes

A defer captures values or references for later cleanup use.

Its captured lifetime extends until the defer executes.

Example:

```sec
let value := CreateValue()
defer Log(ref value)

Consume(value)
```

This is invalid if defer needs a reference to value after value is consumed.

The compiler must treat defer as a future use at its cleanup position.

## Defer capture modes

A defer may conceptually capture by:

* borrow
* copy
* move
* deferred expression evaluation

The language must define the capture semantics precisely.

Lifetime analysis must reflect the selected model.

A borrow capture extends the borrow until defer execution.

A copy capture creates an independent value.

A move capture transfers ownership into the deferred operation.

## Cleanup lifetimes

Values used during cleanup must remain valid until their cleanup entry executes.

Cleanup ordering affects lifetime validity.

With a unified reverse registration cleanup stack, a deferred operation may safely reference an earlier-owned value only when that value is destroyed after the defer.

Example:

```sec
let resource := CreateResource()
defer Inspect(ref resource)
```

If defer executes before resource destruction, the reference remains valid.

The compiler must verify cleanup ordering rather than merely lexical ordering.

## Free-operation lifetimes

During a custom free operation:

* self remains valid in a restricted destruction state
* initialized unmoved fields remain valid
* external references must not remain active
* self must not escape
* new references to self must not escape
* references created for internal cleanup end before free completes

After free completes, self's lifetime ends.

The compiler must not permit a free operation to store a reference to self globally or return it.

## Destruction-time references

A free implementation may temporarily borrow its fields for cleanup.

These references must remain internal to the free operation.

A field may not be destroyed while an internal reference to it remains active.

The compiler-generated field-destruction sequence occurs after the custom free body under the selected destruction model.

## Branch analysis

Lifetime state is path-sensitive.

Each branch may:

* create references
* end references
* move owners
* destroy values
* reinitialize storage
* return references

At a merge point, the compiler must compute a valid combined state.

A reference usable after the merge must be valid on every incoming reachable path.

## Conditional reference creation

Example:

```sec
let reference :=
    if condition {
        ref first
    } else {
        ref second
    }
```

The resulting reference is valid only within a region where both possible referents remain valid.

Its inferred upper lifetime bound is the conservative intersection of the two source lifetimes.

If one source ends earlier, the merged reference cannot be used afterward.

## Conditional owner destruction

A reference cannot merge into a region where its referent is destroyed on one path.

Example:

```sec
let reference := ref value

if condition {
    Consume(value)
}

Use(reference)
```

This is invalid.

The referent exists on only one incoming ownership path.

## Exhaustive branches

Mutually exclusive branch borrows may be valid even when they overlap conceptually because only one branch executes.

Each branch is analyzed independently.

At merge, only borrows still live on outgoing paths participate.

The compiler should not retain branch-local borrows after their last branch use.

## Early return

An early return ends lifetimes in exited scopes after:

* return expression evaluation
* return ownership or reference validation
* defer execution
* destruction cleanup

A returned reference must remain valid after callee cleanup.

A reference into a local destroyed during return cleanup is invalid.

## Returning reference before cleanup

The compiler must not validate a return reference solely at the return expression.

It must account for all cleanup executed before control reaches the caller.

Example:

```sec
fn Invalid() ref Value {
    let value := CreateValue()
    return ref value
}
```

Even though value is valid while evaluating the return expression, it is destroyed before the caller can use the reference.

## Error propagation

Error propagation exits scopes and executes cleanup.

References into cleaned-up values cannot be propagated or retained.

The compiler must validate both:

* Ok continuation lifetime
* Err exit cleanup

A reference created before a `try` expression may remain valid after successful continuation only when the try call does not invalidate its referent.

## Loops

Lifetime analysis for loops must reach a fixed point over loop-carried states.

The compiler must account for:

* zero iterations
* one iteration
* repeated iterations
* backedges
* break edges
* continue edges
* condition evaluation
* iteration-local destruction
* loop-carried references
* reinitialization
* moves

A reference used on a later iteration must remain valid across the backedge.

## Iteration-local borrows

A borrow created within one iteration normally ends before the next iteration unless carried explicitly.

Example:

```sec
for item in items {
    let view := ref item
    Inspect(view)
}
```

The borrow may end after Inspect during each iteration.

It does not automatically accumulate across iterations.

## Loop-carried references

A reference stored in a variable across iterations requires proving:

* the referent remains valid
* the referent is not moved or destroyed
* no conflicting mutation occurs
* container operations do not invalidate it
* every path initializes the reference before use

Loop-carried mutable references require especially strict exclusivity.

## Break

A reference returned or carried out through break must remain valid at the break destination.

Branch-local and iteration-local referents destroyed during break cleanup cannot back an escaping break reference.

If break values are supported, their lifetime relationships must be validated before cleanup.

## Continue

References needed after continue on the next iteration must survive iteration cleanup.

References to iteration-local values normally cannot cross continue.

The compiler must end iteration-local borrows before executing cleanup and taking the backedge.

## While conditions

References created during a while condition normally remain valid only for condition evaluation unless transferred into loop state explicitly.

The condition may be evaluated multiple times.

No condition temporary may be assumed to persist across iterations without explicit ownership.

## Match payload lifetimes

Matching a value may:

* borrow payloads
* copy payloads
* move payloads

A borrowed payload reference cannot outlive:

* the matched value
* the active variant
* the match borrow
* any mutation changing the variant

Moving the payload ends its lifetime in the original container.

## Match on temporary

Matching a temporary by reference permits use only while the temporary remains alive.

A reference bound from a temporary match cannot escape the match expression unless the temporary's ownership is transferred into a longer-lived owner.

## Switch lifetimes

Switch discriminant temporaries remain valid for the required comparison operations.

Branch-local values and references end according to branch control flow.

A switch over a borrowed value must not allow payload-like ownership extraction unless the type and operation permit it.

## Function-call lifetimes

Before a call, the compiler determines:

* argument evaluation lifetimes
* parameter borrow regions
* ownership transfers
* alias relationships
* possible invalidation effects
* returned reference relationships
* escape behavior

After the call, it updates lifetime state according to the function contract.

## Call-duration borrows

A reference argument may be borrowed only for the call duration when the callee does not retain or return it.

Example:

```sec
Inspect(ref value)
Consume(value)
```

This is valid when Inspect's contract is non-escaping.

The borrow ends at call return.

## Retained references

A function that stores a reference beyond return must declare a relationship through its result, output owner or explicit API contract.

Ordinary reference parameters should be non-retained by default.

The compiler should assume safe Sec functions do not retain references unless their types make the retention visible.

This default simplifies local inference.

## Returned-reference contracts

Exported function metadata must describe which input lifetime bounds a returned reference depends on.

Possible relationships include:

* result derives from parameter N
* result derives from receiver
* result derives from static storage
* result lifetime is bounded by multiple input references
* result is unsafe or externally contracted

This metadata is required for separate compilation.

## Multiple return values

Each returned reference may have separate lifetime dependencies.

Example conceptual return:

```sec
fn Split(value: ref Pair) (ref First, ref Second)
```

Both results derive from value.

Their borrow overlap and mutability must be valid.

Two mutable references to disjoint fields may be returned only when disjointness is proven.

## Function pointers

A function pointer signature must preserve lifetime-relevant parameter and result relationships.

Erasing these relationships would make indirect safe calls unsound.

The compiler may restrict function-pointer compatibility when lifetime relationships differ.

A raw foreign function pointer lacks safe lifetime metadata and requires an unsafe wrapper.

## Interfaces

Interface method declarations must preserve lifetime relationships.

Dynamic dispatch must not erase whether a returned reference derives from:

* receiver
* a parameter
* static storage

All implementations must satisfy the interface's declared relationship.

An implementation may return a shorter-lived reference only when it still satisfies the interface contract.

## Interface object lifetime

A borrowed interface view is bounded by the concrete value it references.

An owning interface value owns the concrete value.

A reference returned from an interface method into the concrete object is bounded by the interface object's underlying ownership lifetime.

Moving or destroying the owning interface invalidates such references.

## Generic lifetimes

Generic code must be valid for every permitted instantiation.

Generic type abstraction must not hide reference fields or lifetime requirements needed for safety.

Constraints may need to distinguish:

* owned values
* borrowed values
* static-backed values
* types containing references
* types permitted to escape

The initial implementation should infer generic lifetime requirements from operations in the generic body.

## Monomorphized lifetime analysis

Monomorphization may refine lifetime facts based on concrete types.

However, generic source validation should reject operations that are unsound for allowed instantiations.

Concrete instantiation may determine:

* whether a type contains references
* whether storage is address-stable
* whether an operation invalidates element references
* whether a value is move-only
* whether a type may safely escape

## Higher-order functions

Functions accepting callbacks must define callback lifetime and escape behavior.

Relevant distinctions include:

* callback invoked immediately and not retained
* callback may be called multiple times during the call
* callback stored after return
* callback transferred to another thread
* callback invoked during cleanup
* callback may outlive captured references

The compiler cannot safely assume non-escape for an opaque callback-taking function without metadata.

## FFI lifetimes

Foreign interfaces require explicit lifetime contracts.

The compiler must distinguish:

* pointer borrowed for call duration
* pointer retained by foreign code
* pointer returned and owned by caller
* pointer returned and borrowed from an input
* pointer valid until next foreign call
* pointer valid until explicit foreign release
* callback retained by foreign code
* thread-local foreign pointer
* static foreign storage

C types alone do not encode these relationships.

## Safe FFI wrappers

A safe Sec wrapper must convert foreign lifetime contracts into Sec ownership and reference semantics.

Example:

* a borrowed foreign pointer may become a call-local ref
* a foreign-owned object may be wrapped in a move-only owning type
* a pointer valid until next call may be copied into owned Sec storage
* a retained callback may require moved owned captures rather than borrowed locals

Unsafe complexity should remain inside the wrapper.

## Foreign-retained pointers

Passing a reference to foreign code that retains it is invalid unless the backing storage is guaranteed to outlive the foreign use.

Local stack references normally cannot be retained.

Valid strategies may include:

* explicit heap allocation with transferred ownership
* static storage
* caller-managed long-lived storage
* pinned storage if later supported
* opaque owning callback context

The contract must define release and completion.

## Foreign callbacks

A callback registered with foreign code may outlive the registration call.

The callback closure cannot borrow short-lived locals.

The wrapper must know:

* whether callback invocation is synchronous
* whether it is retained
* which thread invokes it
* when registration ends
* how captured ownership is released

Ambiguous callback lifetime requires unsafe code.

## Syscalls

Raw syscalls involving pointers borrow or access memory according to syscall semantics.

The wrapper must define whether the kernel:

* accesses memory only during the call
* retains the pointer
* writes asynchronously
* maps shared memory
* transfers ownership
* pins pages

The raw syscall ABI does not provide a Sec lifetime guarantee.

Safe platform APIs must encode the correct model.

## Asynchronous operating-system I/O

An asynchronous operation that retains a buffer cannot accept an ordinary short-lived reference unless completion is guaranteed before the borrow ends.

Possible safe designs include:

* consuming an owned buffer
* borrowing through a scoped completion API
* returning an operation object that owns the buffer
* requiring static or externally managed storage
* using pinned storage where required

The compiler must not treat asynchronous use as call-duration borrowing.

## Interrupt handlers

References captured by interrupt handlers may outlive the registration call and execute unpredictably.

Safe registration requires storage with sufficient lifetime and concurrency semantics.

Local references must not escape into interrupt handlers.

Bare-metal target rules may provide explicit static storage patterns.

## Volatile storage

Volatile or addressed storage has an external storage lifetime.

A variable bound with `@address(...)` is treated as volatile external storage.

Its storage is not owned or destroyed like an ordinary local value.

A reference or accessor to addressed storage is bounded by:

* target validity
* address mapping validity
* peripheral lifetime
* mutability declaration
* register rules
* execution context

The compiler must not infer that lexical scope controls the hardware storage lifetime.

## Addressed instances

For an addressed instance:

```sec
let device := @address(...)
```

the binding provides access to external storage.

Without `mut`, access is read-only but remains volatile.

With `mut`, reads and writes are permitted according to register and field rules.

Lifetime analysis must treat the external storage as available according to the platform declaration, not as stack-owned storage.

References derived from it must not be used where the target mapping is invalid.

## Register fields

A reference to a register field may be prohibited entirely if a normal reference would imply non-volatile memory semantics.

Register access should normally use register-aware operations rather than general references.

If register references are supported, their lifetime and access effects must remain tied to the addressed register object.

The compiler must not cache or extend volatile accesses through lifetime optimization.

## Thread-local storage

A reference to thread-local storage is bounded by the current thread's storage lifetime.

It must not be transferred to another thread unless the type and platform explicitly support such behavior.

Thread exit invalidates thread-local storage references.

## Concurrency

Lifetime validity across threads requires both:

* storage lifetime
* legal ownership or sharing

A reference cannot be sent to another thread merely because the storage physically remains alive.

The type must satisfy concurrency and synchronization constraints.

Thread transfer may extend execution beyond the current scope, making local borrows invalid.

## Scoped threads

A future scoped-thread API may permit borrowed references when all spawned threads are guaranteed to complete before the scope exits.

The API must make this join relationship compiler-visible.

Without such a contract, spawned execution is treated as escaping.

## Locks and guards

A lock guard may control the lifetime of a protected reference.

A reference into protected data must not outlive the guard unless another ownership rule provides safety.

Moving or destroying the guard may end access permission even if the underlying data remains allocated.

Lifetime analysis may need to track capability lifetime, not only storage lifetime.

## Capability lifetimes

Some values grant temporary authority rather than physical storage.

Examples include:

* lock guards
* transaction handles
* interrupt-disable guards
* mapped-access tokens
* borrow tokens
* device-session guards

References or operations depending on a capability must not outlive it.

The type system may represent this through borrowed relationships or compiler-known capability effects.

## Unsafe blocks

Unsafe code may perform operations the compiler cannot fully prove.

Unsafe does not disable lifetime analysis for surrounding safe references.

Inside unsafe, the programmer may assume responsibility for:

* RawPtr validity
* external storage lifetime
* aliasing
* foreign retention
* manual allocation
* self-referential structures

When values return to safe code, all safe reference invariants must again hold.

## Unsafe lifetime promises

The compiler may eventually support an internal unsafe mechanism for asserting a lifetime relationship.

Such a mechanism must:

* be narrow
* be explicit
* remain inside unsafe code
* not fabricate ownership
* not extend actual storage lifetime
* make the programmer responsible for proof
* preserve diagnostics and IR metadata where possible

The initial implementation should avoid general-purpose lifetime casting.

## Lifetime extension casts

Safe lifetime extension is prohibited.

An unsafe cast that claims a longer lifetime does not make storage remain alive.

It is valid only when the programmer separately guarantees the backing storage.

The compiler should treat such casts as highly unsafe and unnecessary for ordinary Sec code.

## Allocation and deallocation

Deallocation ends the allocation storage lifetime.

Before deallocation:

* all contained initialized values must be destroyed or transferred
* all safe references into the allocation must have ended
* no retained foreign access may remain
* allocator requirements must be satisfied

The compiler must not move deallocation before the final valid reference use.

## Arena lifetimes

An arena may own many allocations with a shared upper lifetime bound.

References into an arena cannot outlive the arena.

Individual objects may become logically invalid earlier.

Destroying or resetting the arena invalidates all contained references.

A future arena type must expose these invalidation effects.

## Region allocation

If Sec later supports region allocation, region lifetime remains an explicit ownership concept.

The compiler must not introduce hidden regions to repair invalid references.

Region-backed values should be represented through owning region objects or compiler-visible scopes.

## Static analysis inputs

Lifetime analysis consumes facts from:

* resolved symbols
* canonical types
* ownership analysis
* copy and move classification
* borrow analysis
* control-flow graph
* definite initialization analysis
* destruction analysis
* escape analysis
* function-effect metadata
* storage classification
* target validation
* FFI contracts

The exact scheduling may require iterative coordination.

## Analysis ordering

Recommended analysis order:

1. symbol and type resolution
2. storage and value classification
3. control-flow graph construction
4. initialization analysis
5. ownership and move-state analysis
6. preliminary borrow creation analysis
7. lifetime region inference
8. borrow conflict validation
9. escape analysis
10. destruction and cleanup placement
11. final lifetime validation
12. Semantic IR verification

Some phases may iterate to a fixed point.

Lifetime analysis and escape analysis are closely related but distinct.

## Lifetime and escape analysis

Lifetime analysis asks:

* where is a value or reference valid?

Escape analysis asks:

* does a value, reference or address leave a permitted region?

An escaping reference is invalid when its destination lifetime exceeds the inferred reference lifetime.

Escape analysis must not repair invalidity through hidden heap allocation.

## Lifetime and destruction analysis

Lifetime analysis constrains destruction placement.

Destruction must not occur before:

* final valid value use
* final valid reference use
* final required cleanup use
* completion of foreign borrowing
* completion of asynchronous scoped use

Destruction analysis provides the definitive end point for owned values.

The analyses must agree.

## Lifetime and move analysis

A move:

* ends source-binding validity
* transfers ownership
* may preserve physical storage
* may invalidate references
* may be prohibited by active borrows

Lifetime analysis must use semantic move state, not machine relocation.

A non-relocating move may still be prohibited under conservative initial rules.

## Lifetime and mutability analysis

Mutability analysis determines which accesses may modify storage.

Lifetime analysis determines when such access is valid.

A mutable binding is not valid outside its value lifetime.

A mutable reference cannot remain active after referent replacement or destruction.

Both analyses are required.

## Lifetime and definite initialization

A reference may be created only to initialized values or fields.

Conditional initialization creates path-specific lifetime beginnings.

At merges, a reference cannot be assumed valid when its referent is uninitialized on any reachable incoming path.

## Lifetime constraints

The compiler may formulate constraints such as:

```text
reference R begins after initialization of V
R ends before destruction of V
R ends before move of V
R ends before replacement of V
R is contained within storage lifetime S
returned reference R is bounded by parameter P
reborrow R2 is bounded by R1
closure capture C is bounded by captured reference R
```

The solver must either prove all constraints or issue diagnostics.

## Constraint solving

Lifetime constraints may be solved through:

* data-flow analysis
* region inference
* graph reachability
* dominance and post-dominance
* strongly connected components
* fixed-point iteration
* use-def analysis
* borrow-state propagation

The algorithm should prioritize:

* predictable diagnostics
* compiler performance
* correctness
* incremental recomputation
* debuggability

The source language must not depend on one specific solver implementation.

## Minimal inferred lifetime

The compiler should infer the smallest lifetime satisfying all valid uses.

Benefits include:

* fewer artificial borrow conflicts
* earlier permitted mutation
* earlier permitted move
* earlier destruction where semantically valid
* better stack-slot reuse
* clearer diagnostics
* improved optimization

The inferred lifetime must never be shortened below a hidden or deferred use.

## Conservative inference

When the compiler cannot prove a narrower lifetime safely, it may use a conservative longer borrow within the referent's valid region.

Conservatism may reject otherwise safe code.

It must never accept unsafe code.

Compiler improvements may later accept more programs without changing language semantics.

## Lifetime equivalence

Two references may have equivalent inferred lifetimes without sharing identity.

Lifetime equivalence does not imply:

* pointer equality
* same referent
* aliasing
* same mutability
* interchangeable ownership

The compiler tracks lifetime and alias facts separately.

## Subtyping-like lifetime relationships

A reference valid for a longer region may be used where a shorter region is required.

The reverse is not valid.

The compiler may shorten a borrow through reborrowing or argument passing.

It must not extend a shorter reference to satisfy a longer requirement.

This relation can remain internal without source-level lifetime notation.

## Mutability conversion

A mutable reference may be reborrowed as a shared reference for a shorter region.

A shared reference cannot become mutable.

The shared reborrow must not outlive the mutable reference.

During the shared reborrow, mutation through the original mutable reference is suspended.

## Variance

The initial implementation should avoid exposing general variance rules in source syntax.

Internally, reference-containing generic types may require variance-like reasoning.

The compiler should use conservative invariance where necessary until a complete rule exists.

Safety is more important than accepting every theoretically valid conversion.

## Lifetime erasure boundary

Lifetime metadata may be progressively erased only after all dependent semantic decisions are fixed.

Before erasure, the compiler must have resolved:

* borrow legality
* ownership transfer
* escape validity
* destruction placement
* storage allocation
* reference invalidation
* ABI lifetime contracts
* FFI requirements

LLVM IR does not need to encode Sec lifetime regions directly.

## Semantic IR representation

Semantic IR should preserve lifetime-relevant operations explicitly.

Expected concepts include:

* begin value lifetime
* initialize storage
* create shared borrow
* create mutable borrow
* reborrow
* end borrow
* move value
* replace value
* destroy value
* deallocate storage
* project field reference
* project element reference
* return reference
* escape reference
* call with borrow effect
* call with retained-reference effect

Not every concept must become a runtime instruction.

They may be verifier-visible semantic operations.

## Reference provenance in Semantic IR

Each safe reference should retain provenance to:

* source value or storage
* projection path
* mutability
* inferred lifetime region
* ownership root
* source location
* FFI or unsafe origin where applicable

Projection paths may include:

* struct fields
* array indices
* slice ranges
* union payloads
* dereference layers

Provenance supports diagnostics and overlap analysis.

## Lifetime IDs

The compiler may assign internal lifetime or region IDs.

These IDs:

* are not source-visible
* need not be stable across builds
* support analysis and diagnostics
* may be emitted in debug IR dumps
* must not affect symbol ABI unless representing exported relationships

Exported relationships should use stable structural metadata rather than arbitrary local IDs.

## Cleanup regions

Semantic IR cleanup regions must preserve reference validity.

A reference used by a cleanup entry remains live until that entry executes.

A value cannot be destroyed before cleanup entries that borrow it.

Cleanup ordering must be reflected in lifetime dependencies.

## Block arguments

References crossing block edges should be explicit block arguments or otherwise explicit data-flow values.

Their lifetime region must include:

* source edge
* destination block
* all downstream uses

A block must not receive a reference on an edge where its referent is invalid.

## Phi-like reference merges

A merged reference must carry compatible:

* type
* mutability
* lifetime upper bound
* provenance class where required
* alias restrictions

The resulting lifetime is no longer than any possible incoming referent validity.

Mutable-reference merges require exclusive access on every incoming path.

## Semantic IR verifier

The Semantic IR lifetime verifier must check:

* every reference derives from valid storage
* every reference use lies within its inferred region
* no reference use occurs after referent move
* no reference use occurs after destruction
* no reference use occurs after replacement
* no reference use occurs after deallocation
* no returned reference derives from invalid local storage
* reborrows do not outlive source references
* mutable and shared overlaps are legal
* cleanup uses occur before referent destruction
* block arguments have valid incoming references
* function-result relationships match signatures
* FFI lifetime effects are explicit
* unsafe-created safe references carry required validation boundaries

Verifier failure after semantic analysis is an internal compiler error.

## MLIR lowering

MLIR lowering may erase high-level lifetime regions gradually.

It must preserve:

* storage validity
* destruction order
* borrow-dependent alias assumptions
* noalias conditions
* volatile semantics
* deallocation ordering
* returned-reference validity
* address stability

Lifetime facts may inform optimization but must not be translated into overly strong LLVM assumptions.

## MLIR lifetime operations

A Sec MLIR dialect may include operations such as:

```text
sec.lifetime.begin
sec.lifetime.end
sec.borrow.shared
sec.borrow.mutable
sec.borrow.end
sec.reference.project
sec.reference.reborrow
```

These operations may be compile-time-only.

They provide verification boundaries before lowering to ordinary memory and control-flow operations.

## Alias analysis

Lifetime facts support alias analysis but do not completely determine aliasing.

A mutable reference may justify exclusive-access assumptions during its valid region.

A shared reference may justify read-only access according to type rules.

RawPtr must not inherit safe-reference noalias guarantees.

The compiler must avoid unsound alias metadata when unsafe code may violate assumptions.

## LLVM lifetime intrinsics

LLVM lifetime intrinsics may describe stack storage availability.

They are optimization hints.

They do not implement Sec lifetime semantics.

An LLVM lifetime end may be emitted only after:

* value destruction
* final reference use
* final cleanup use
* any required secure clearing
* any required volatile operation

The compiler must not emit a lifetime end merely because a lexical scope ended.

## LLVM noalias

LLVM `noalias`-like attributes or metadata may be emitted only when proven by Sec's ownership and borrow model for the complete relevant region.

Potentially valid sources include:

* exclusive mutable reference parameters
* unique owned allocations
* non-overlapping field projections
* compiler-generated return storage

Unsafe RawPtr conversion or foreign calls may weaken assumptions.

Incorrect noalias is a miscompilation bug.

## LLVM dereferenceability

Dereferenceability metadata requires proof that:

* storage is valid
* the byte range is accessible
* alignment holds
* lifetime covers the operation
* no deallocation may intervene

The compiler should be conservative, especially for external, volatile and foreign storage.

## LLVM invariant metadata

Immutable data may permit invariant-load-like optimization only when:

* the underlying storage cannot change during the relevant lifetime
* no foreign or unsafe alias may mutate it
* volatility is absent
* target rules permit it

A shared reference alone does not always prove global immutability of externally aliased storage.

## Stack-slot reuse

Lifetime analysis may permit reuse of stack storage after:

* value lifetime ends
* destruction completes
* references end
* debug constraints allow reuse
* no escaping address remains

Stack-slot reuse must not make stale safe references valid.

RawPtr use after reuse remains unsafe.

## Early destruction

The compiler may destroy a value before lexical scope exit when:

* no later value use exists
* no active reference exists
* no defer or cleanup entry requires it
* destruction ordering relative to other observable operations is preserved
* source semantics permit lifetime shortening
* debugging policy permits it

Early destruction is an optimization unless the language defines last-use destruction semantically.

The initial compiler should preserve deterministic source-derived destruction order unless a proof confirms no observable change.

## Observable destruction

If free or field destruction has observable effects, lifetime shortening may change behavior.

The compiler must not move destruction earlier across:

* function calls
* volatile operations
* foreign calls
* logging
* synchronization
* other destruction
* defer execution

unless effect analysis proves equivalence.

## Lifetime extension for optimization

The compiler may keep storage physically alive longer than the semantic value lifetime.

It must not permit semantic access after lifetime end.

Physical lifetime extension may occur through:

* delayed stack reuse
* unremoved stack slots
* register live ranges
* debug information
* backend limitations

This does not extend safe reference validity.

## Debug information

Debug builds should expose useful lifetime information where practical.

A debugger may show:

* initialized
* borrowed
* mutably borrowed
* moved
* destroyed
* out of scope
* storage reused

The debugger must not imply that physically present stale bytes form a valid Sec value.

## Diagnostics

Lifetime diagnostics should explain:

* which reference is invalid
* where the reference was created
* which value or storage it refers to
* where the referent lifetime ends
* which operation causes invalidation
* where the invalid use occurs
* whether a borrow, move, replacement, destruction or escape caused the conflict

Diagnostics should use source terminology rather than solver terminology.

## Diagnostic examples

Preferred style:

```text
error: reference to local value escapes the function
  reference created here
  local value is destroyed when the function returns
  returned here
```

Preferred style:

```text
error: cannot move `buffer` while it is borrowed
  shared borrow starts here
  borrow is later used here
  move attempted here
```

Preferred style:

```text
error: returned reference may refer to either `first` or `second`
  the compiler cannot establish a valid result lifetime for this signature
```

Avoid exposing internal region-variable names unless requested through compiler debug output.

## Diagnostic suggestions

The compiler may suggest:

* bind the temporary to an owned local
* pass a reference instead of returning it
* return an owned value
* shorten the borrow by moving its final use
* clone or duplicate explicitly
* transfer ownership into the closure
* use a non-escaping callback
* wrap foreign storage in an owning type
* restructure control flow
* avoid retaining a pointer in foreign code

Suggestions must not propose hidden heap allocation.

## Lifetime debug mode

The compiler may provide a diagnostic mode such as:

sec check --explain-lifetimes

* inferred reference regions
* final uses
* borrow beginnings and ends
* ownership transfers
* destruction points
* escape paths
* invalidation effects

* loop-carried reference
* unsafe-origin metadata

MLIR tests should verify:

* lifetime operation lowering
* correct stack lifetime markers
* no premature lifetime end
* no invalid noalias
* deallocation after final use
* stack-slot reuse only after valid end
* volatile storage unaffected
* return storage relationships

## Performance

Lifetime analysis must scale to large functions and modules.

The compiler should avoid:

* exponential path enumeration
* unrestricted symbolic execution
* repeated whole-module analysis
* excessive per-use graph rebuilding

Potential implementation strategies include:

* bitset data-flow states
* sparse use-def propagation
* incremental region constraints
* strongly connected component handling for loops
* cached function summaries
* field masks for aggregate state
* stable symbol and value IDs

Correctness takes priority over optimization of the compiler itself.

## Incremental compilation

Lifetime analysis should support incremental recomputation.

Changes should invalidate:

* affected function bodies
* dependent generic instantiations
* callers when exported lifetime summaries change
* interface implementations when method relationships change
* FFI wrappers when contracts change

Internal changes that do not alter exported lifetime summaries should not require unrelated module reanalysis.

## Function lifetime summaries

Each compiled function should produce a summary containing relevant facts such as:

* parameter ownership modes
* parameter borrow modes
* whether references escape
* returned-reference dependencies
* callback escape behavior
* receiver relationship
* storage invalidation effects
* container reference invalidation
* foreign retention behavior
* concurrency transfer behavior

Summaries support separate compilation and incremental analysis.

## Stable exported metadata

Exported lifetime metadata must be:

* deterministic
* versioned
* target-aware where necessary
* independent of local solver IDs
* compatible with module hashing
* validated when imported
* rejected when compiler versions cannot interpret it safely

Imported metadata is part of semantic validation, not merely optimization hints.

## Compiler implementation responsibilities

The lifetime-analysis implementation should have clear boundaries.

The type system determines:

* whether a type contains references
* projection types
* mutability
* storage-related type properties
* reference-bearing generic conditions

Ownership analysis determines:

* current owners
* moves
* destruction responsibility
* partial moves
* reinitialization

Borrow analysis determines:

* borrow creation
* overlap
* exclusivity
* reborrowing
* access conflicts

Lifetime analysis determines:

* valid regions
* final uses
* result dependencies
* escape compatibility
* referent end constraints

Escape analysis determines:

* where references and addresses travel
* whether destination outlives source
* whether callbacks or foreign code retain access

Destruction analysis determines:

* final cleanup positions
* invalidation through destruction
* cleanup ordering

These analyses may cooperate but should not be collapsed into an unmaintainable single pass.

## Failure policy

When lifetime safety cannot be proven:

* safe code is rejected
* the compiler does not guess
* the compiler does not silently allocate
* the compiler does not silently copy
* the compiler does not silently leak
* the compiler does not extend storage lifetime invisibly
* the compiler does not weaken mutability rules

Unsafe code may express externally guaranteed behavior only through explicit unsafe operations.

## Initial implementation restrictions

The initial Sec implementation should use conservative rules:

* no ordinary source-level lifetime parameters
* non-lexical local borrow inference
* references do not own or extend referent lifetime
* RawPtr does not extend lifetime
* no hidden stack-to-heap promotion
* no returned references to locals or temporaries
* returned references may derive from parameters, receiver or static storage when unambiguous
* ambiguous returned-reference relationships are rejected
* reference-containing aggregates are supported only when dependencies remain inferable
* safe self-referential aggregates are rejected
* moving a value with an active reference is rejected
* moving out through references is rejected
* escaping closures cannot borrow local storage
* callbacks are assumed escaping unless marked non-escaping
* foreign code is assumed not to retain references only when the contract says so
* container mutations use declared invalidation effects
* lifetime extension casts are not provided in safe code
* no runtime borrow checking
* no garbage collection
* no automatic reference counting

These restrictions may be relaxed through stronger analysis or explicit later language features.

* no reference crosses destruction
* no reference crosses incompatible replacement
* no reference outlives backing storage
* every returned reference has a representable valid source relationship
* every closure capture is lifetime-valid
* every callback escape is classified
* every FFI lifetime effect is known or unsafe
* every branch and loop has a consistent lifetime state
* every borrow conflict is resolved
* every cleanup dependency is valid

Before lifetime-aware MLIR lowering completes:

* cleanup ordering preserves borrowed values
* function summaries are emitted

Before LLVM translation:

## Design rule

Ownership determines who is responsible for a value.

Lifetime analysis determines when that value and its references remain valid.

Borrow analysis determines which accesses may coexist.

Escape analysis determines whether access leaves its valid region.

Destruction ends ownership and invalidates dependent references.

The programmer should not describe ordinary lifetimes manually.

The compiler must infer them, prove them and preserve them through every lowering stage.

=============================================================================================================================================================================================
Lifetime Analysis
=================

Lifetime analysis determines when values, storage locations, references, borrows and ownership responsibilities are valid.

The Sec compiler must infer lifetimes from:

* lexical scopes
* control flow
* ownership
* moves
* borrows
* reference creation
* reference use
* storage duration
* function boundaries
* destruction points
* platform and FFI contracts

Programmers should not normally write explicit lifetime names or lifetime parameters.

Lifetime complexity belongs in the compiler.

## Design goals

Sec lifetime analysis must be:

* static
* flow-sensitive
* path-sensitive
* ownership-aware
* borrow-aware
* field-sensitive where required
* compatible with deterministic destruction
* compatible with move-only types
* compatible with references
* compatible with generics
* compatible with separate compilation
* independent of garbage collection
* independent of runtime borrow tracking
* usable without a runtime

The analysis must prevent:

* dangling references
* use after destruction
* use after move
* returning references to dead storage
* references surviving replacement of their target
* incompatible mutable aliasing
* references escaping beyond storage validity
* destruction while references remain valid
* storage reuse while references remain valid

## Terminology

The following concepts are distinct:

* value lifetime
* storage lifetime
* ownership lifetime
* reference lifetime
* borrow lifetime
* lexical scope
* live range
* storage duration
* destruction point
* validity region

A value lifetime is the period during which a particular initialized value exists.

A storage lifetime is the period during which a storage location may contain a value.

An ownership lifetime is the period during which an owner is responsible for a value.

A reference lifetime is the period during which a reference may be validly used.

A borrow lifetime is the period during which access restrictions caused by a borrow apply.

A lexical scope is a source-language region.

A live range is a compiler data-flow concept.

Storage duration describes how long storage exists physically.

A destruction point ends an owned value's lifetime.

These concepts often overlap but must not be treated as identical.

## Core lifetime rule

A reference may be used only while all of the following remain true:

* its target storage exists
* its target value remains initialized
* its target value has not been moved
* its target value has not been destroyed
* its target storage has not been invalidly replaced
* the reference's borrow permissions remain valid
* no conflicting borrow invalidates access
* the reference has not itself been moved or expired
* target-specific restrictions remain satisfied

The compiler must prove these conditions statically.

## Inferred lifetimes

Sec should infer lifetimes automatically.

Example:

```sec id="v4sk19"
fn ReadLength(buffer: ref Buffer) uint {
    return buffer.Length()
}
```

The compiler knows that:

* buffer is borrowed for the call
* the borrow cannot outlive the caller-provided value
* the reference is valid only during the function invocation unless otherwise expressed in the signature
* the function does not own Buffer

No explicit lifetime name should be required.

Sec should avoid source syntax equivalent to:

```text id="k4br2h"
fn ReadLength['a](buffer: ref['a] Buffer) uint
```

unless future language requirements prove explicit lifetime parameters unavoidable.

## Lifetime sources

A lifetime may be constrained by:

* local declaration
* parameter validity
* caller-provided storage
* return value ownership
* reference source
* aggregate field ownership
* variant payload activation
* loop iteration
* temporary expression
* static storage
* explicitly allocated storage
* external storage
* fixed-address storage
* addressed register storage
* FFI declaration
* closure capture
* interface object ownership

The compiler must derive the narrowest lifetime consistent with valid use.

## Lexical lifetime versus actual lifetime

A value's actual semantic lifetime may be shorter than its lexical scope.

Example:

```sec id="ex5mko"
fn Example() void {
    let buffer := Buffer.Create(4096)

    Use(buffer)
}
```

If Use consumes buffer:

* buffer's ownership lifetime ends at the call
* buffer remains lexically declared until the end of the scope
* the binding is unavailable after the move
* no destruction occurs at lexical scope exit

Lexical scope does not imply continued value validity.

## Non-lexical lifetimes

Borrow lifetimes should end after their final valid use rather than always extending to the end of the lexical scope.

Example:

```sec id="rdh31e"
fn Example(mut value: Value) void {
    let view := ref value

    Inspect(view)

    value.Update()
}
```

The shared borrow may end after `Inspect(view)` if `view` is not used again.

The later mutation may therefore be valid.

The compiler should implement non-lexical lifetime analysis where practical.

The source programmer should not need to create artificial blocks solely to end ordinary borrows.

## Conservative inference

When the compiler cannot prove that a borrow has ended, it must keep the borrow active.

The compiler must reject ambiguous code rather than assume a shorter lifetime.

Diagnostics should explain:

* where the borrow begins
* the last use extending the borrow
* the conflicting operation
* how control flow keeps the borrow active

## Lifetime of local values

A successfully initialized local value begins its lifetime after initialization completes.

Its lifetime ends when one of the following occurs:

* the value is moved
* the value is destroyed
* the owning scope exits
* the binding is replaced
* the value is transferred into another owner
* the active variant containing it changes
* the program follows a non-returning operation that bypasses cleanup

The binding may remain lexically visible after the value lifetime ends.

Such a binding is unavailable until valid reinitialization where permitted.

## Initialization boundary

A value lifetime does not begin before successful initialization.

Example:

```sec id="orjvlw"
let resource := try OpenResource()
```

If OpenResource returns Err:

* resource never becomes initialized
* resource has no value lifetime
* resource requires no destruction
* no reference may be formed to resource

The compiler must not emit lifetime start before successful construction.

## Partial initialization

An aggregate may contain fields whose lifetimes begin at different times during construction.

Example:

```sec id="gr2ppc"
let session := Session {
    connection: try OpenConnection()
    buffer: try CreateBuffer()
}
```

If connection succeeds but buffer fails:

* connection's field value lifetime began
* buffer's field value lifetime never began
* Session's complete-value lifetime never began
* connection must be destroyed
* Session's complete free operation must not run unless partial-construction rules explicitly allow it

Lifetime analysis must cooperate with initialization analysis at field granularity.

## Reinitialization

A moved or otherwise uninitialized mutable binding may begin a new value lifetime through valid reinitialization.

Example:

```sec id="t1zoyc"
mut buffer := Buffer.Create(1024)

Consume(buffer)

buffer = Buffer.Create(2048)

Use(buffer)
```

There are two distinct Buffer value lifetimes associated with the same source binding and possibly the same storage.

The compiler must not merge them into one continuous lifetime.

References to the first value cannot become references to the second value automatically.

## Replacement

Replacing an initialized value ends the old value lifetime and begins a new value lifetime.

Example:

```sec id="xwibmz"
mut current := CreateFirst()
current = CreateSecond()
```

Conceptually:

1. CreateSecond is evaluated
2. the old current value is destroyed
3. the old value lifetime ends
4. the new value is installed
5. the new value lifetime begins

Exact physical storage may be reused.

Semantic value identity changes.

## References across replacement

A reference to a value normally prevents replacement while the reference remains active.

Example:

```sec id="ik3ttg"
mut current := CreateValue()
let view := ref current

current = CreateReplacement()

Inspect(view)
```

This is invalid.

The replacement would end the lifetime of the referenced value.

The compiler must not reinterpret `view` as referring to the replacement value.

## Storage lifetime

Storage may outlive the value currently stored in it.

Examples include:

* mutable local storage after a move
* stack slot reused for another value
* heap allocation containing changing elements
* static storage with changing state
* union storage with different active variants

The compiler must distinguish:

* valid storage address
* valid initialized value

A reference to uninitialized storage is not a valid reference to T.

## Stack storage lifetime

Stack storage normally exists from stack-frame allocation until frame exit.

Individual values stored in stack storage may begin and end at narrower points.

The compiler may emit stack lifetime markers reflecting these narrower ranges.

Stack storage existence does not make a destroyed or moved value accessible.

## Heap storage lifetime

Heap storage exists from successful allocation until deallocation.

Values inside the allocation may have narrower lifetimes.

Example:

* allocation succeeds
* element 0 is initialized
* element 1 is initialized
* element 0 is destroyed
* new element 0 is constructed
* allocation remains the same throughout

The compiler must track both allocation lifetime and contained value lifetimes.

## Static storage lifetime

Static storage may exist for the entire program or target-defined execution period.

A reference to static storage may have a program-long storage lifetime.

This does not imply that mutable contents remain one unchanged value forever.

Static storage containing replaceable values requires explicit synchronization and initialization rules.

The initial language should restrict static owned mutable values until global initialization and destruction are fully specified.

## External storage lifetime

Storage declared with `@address(...)` is external or volatile storage.

Its physical storage lifetime is not controlled by Sec.

The compiler must not assume that scope exit deallocates such storage.

A typed addressed binding describes access to external storage, not ownership of the physical address.

Example:

```sec id="5bhkt3"
let status: DeviceStatus @address(0x40001000)
```

The binding is volatile external access even without a `volatile` keyword.

The lifetime of the binding is lexical.

The physical storage may exist independently of the Sec program.

The compiler must follow register and addressed-storage rules for each access.

## Mutability of addressed storage

An addressed instance without `mut` is read-only from Sec source.

An addressed instance with `mut` permits writes subject to field and register rules.

Mutability does not determine physical lifetime.

Destruction of the binding must not:

* deallocate the address
* clear the register
* write a default value
* invoke ordinary owned-field destruction against hardware storage

External storage semantics are defined by registers.txt and volatile rules.

## Reference lifetime

A reference lifetime begins when the reference is created.

It ends after its final valid use or an earlier explicit control-flow boundary where the reference becomes unavailable.

A reference lifetime cannot exceed:

* target storage lifetime
* target value lifetime
* source borrow permission
* containing reference lifetime
* FFI contract lifetime
* closure or aggregate lifetime containing the reference

The compiler must compute the intersection of all applicable constraints.

## Reference value lifetime

The reference itself is also a value.

A reference value may:

* be copied if shared
* be moved if mutable or otherwise move-only
* be stored
* be passed
* be returned when valid
* cease to be available

The lifetime of the reference value is distinct from the lifetime of its target.

Destroying or moving the reference value does not destroy the target.

## Shared reference lifetime

A shared reference grants read access.

While active, it prevents incompatible mutation or destruction of the target.

Multiple shared references may coexist.

The compiler may shorten each shared borrow to its final use.

Example:

```sec id="l3pjm1"
let first := ref value
let second := ref value

Inspect(first)
Inspect(second)
```

Both shared references may be valid simultaneously.

## Mutable reference lifetime

A mutable reference grants exclusive mutable access.

While active:

* no other mutable reference to overlapping storage may exist
* no shared reference to overlapping mutable storage may be actively used
* the owner cannot access the borrowed region incompatibly
* the target cannot be moved
* the target cannot be destroyed
* the target cannot be replaced

The mutable borrow may end after the mutable reference's final use.

## Lifetime of reborrows

A reference may be reborrowed.

A reborrow lifetime cannot exceed the source reference lifetime.

Example:

```sec id="sb9z51"
fn InspectPart(value: ref Value) void {
    let field := ref value.field
    Inspect(field)
}
```

The field reference is constrained by:

* value's reference lifetime
* field's presence in the containing value
* field's initialization state
* active variant state where applicable

A reborrow must not extend access beyond the original reference.

## Reborrowing mutable references

A mutable reference may be reborrowed for a shorter lifetime.

Example:

```sec id="40lzv0"
fn Update(value: ref mut Value) void {
    UpdateField(ref mut value.field)

    value.Validate()
}
```

The field reborrow may end after UpdateField returns.

The original mutable reference may then become usable again.

The compiler must model temporary suspension rather than permanent consumption where the reference rules permit it.

## Borrow lifetime versus reference storage

A stored reference may cause its borrow to remain active as long as the stored reference may still be used.

Example:

```sec id="2h62wb"
let view := ref value
container.Store(view)
```

If Store retains the reference, the borrow may outlive the call.

The function signature must express retention in a compiler-readable way.

A simple borrowed parameter should default to call-duration access unless its result or destination explicitly preserves the lifetime relationship.

## Call-duration borrows

A reference passed to an ordinary non-retaining parameter is borrowed for the call and any lifetime required by the returned value.

Example:

```sec id="vd633v"
Inspect(ref value)
```

If Inspect returns no reference tied to value and does not store the reference, the borrow may end after the call.

Separate compilation requires the function signature or interface metadata to guarantee non-retention.

The compiler must not rely on inspecting another module's implementation.

## Lifetime relationships in signatures

A function signature may need to express that a returned reference is derived from one of its inputs.

Example:

```sec id="mi0ahg"
fn First(values: ref [Value]) ref Value
```

The compiler must understand that the returned reference cannot outlive `values`.

The preferred design is for this relationship to be inferred from the function body and exported in module metadata without requiring explicit source lifetime names when unambiguous.

For declarations without bodies, such as interfaces or extern declarations, an explicit relation syntax or annotation may eventually be required.

## Single-source returned references

When a function has exactly one reference input capable of producing the returned reference, the lifetime relationship may be inferred unambiguously.

Example:

```sec id="47pt46"
fn Data(buffer: ref Buffer) ref [byte] {
    return ref buffer.data
}
```

The returned reference is bounded by buffer's input lifetime.

The compiler should encode this relationship in the exported function signature metadata.

## Multiple-source returned references

When a returned reference may derive from multiple inputs, lifetime inference may become ambiguous.

Example:

```sec id="n4fqle"
fn Select(
    condition: bool
    first: ref Value
    second: ref Value
) ref Value {
    if condition {
        return first
    }

    return second
}
```

The returned reference is valid only for a lifetime common to both possible sources.

The compiler may derive the intersection of the input lifetimes.

The caller must treat the returned reference as no longer-lived than either relevant input.

## Source-specific branch returns

The compiler may internally preserve path-dependent source relationships.

However, a function's exported signature must provide a stable caller-visible rule.

Possible strategies include:

* lifetime intersection
* explicit variant return carrying the source relation
* rejection of relationships that cannot be represented safely
* future explicit lifetime relation syntax

The initial implementation should prefer conservative lifetime intersection.

## Returning local references

Returning a reference to ordinary local storage is invalid.

Example:

```sec id="bu82q2"
fn Invalid() ref Value {
    let value := CreateValue()
    return ref value
}
```

The local value and stack storage do not outlive the function call.

The compiler must reject the return even if optimization could physically keep the storage alive.

No hidden heap promotion may be introduced.

## Returning references to owned parameters

Returning a reference into a by-value owned parameter is invalid unless ownership of the containing value also returns in a representation preserving the reference.

Example:

```sec id="zrhdsm"
fn Invalid(value: Value) ref Field {
    return ref value.field
}
```

The owned parameter is destroyed when the function exits unless transferred elsewhere.

A bare returned reference would dangle.

## Returning references to borrowed parameters

Returning a reference derived from a borrowed parameter may be valid.

Example:

```sec id="3uwbsl"
fn FieldOf(value: ref Value) ref Field {
    return ref value.field
}
```

The returned reference lifetime is bounded by the caller's source value and the original input borrow.

The compiler must export that relation.

## Returning mutable references

Returning a mutable reference derived from a mutable input may be valid.

Example:

```sec id="cp1a3n"
fn FieldOf(value: ref mut Value) ref mut Field {
    return ref mut value.field
}
```

The returned mutable reference continues the exclusive borrow.

The original mutable reference cannot be used by the caller while the returned mutable reference remains active.

The compiler must preserve exclusivity across the call boundary.

## Returned reference from shared input

A mutable reference must not be returned from a shared reference input.

Example:

```sec id="ai2inf"
fn Invalid(value: ref Value) ref mut Field {
    return ref mut value.field
}
```

This violates mutability and borrow permissions.

Lifetime compatibility alone is insufficient.

## References in structs

A struct containing references carries lifetime dependencies.

Example:

```sec id="pcigb1"
type View struct {
    data: ref [byte]
}
```

A View value cannot outlive the target of `data`.

The compiler must attach lifetime constraints to View instances.

The programmer should not normally need to parameterize View with an explicit source lifetime.

## Aggregate lifetime constraints

The lifetime of an aggregate containing references cannot exceed the shortest required referenced lifetime.

For:

```sec id="r7ze76"
type PairView struct {
    first: ref First
    second: ref Second
}
```

PairView's valid lifetime is bounded by both first and second.

The compiler must compute their intersection.

Moving PairView transfers the references but does not extend their target lifetimes.

## Replacing reference fields

Replacing a reference field ends the old stored reference value lifetime.

It may release the old borrow if no other reference derived from it remains active.

The new reference establishes a new borrow and new lifetime constraints.

The containing aggregate may remain in the same storage while its lifetime dependencies change.

## Self-referential aggregates

An aggregate containing a reference to itself or one of its own fields is self-referential.

Safe ordinary construction of such a value is problematic because relocation can invalidate the reference.

The initial implementation should reject self-referential safe aggregates unless a later stable-address mechanism explicitly supports them.

RawPtr-based self-reference remains unsafe.

## References in enums and unions

A tagged union containing a reference carries the lifetime dependency only while that variant is active.

Changing the active variant ends the old reference value lifetime.

The compiler must ensure that:

* the old borrow ends or remains represented elsewhere
* the old payload is destroyed correctly
* no reference to inactive payload storage remains usable

An untagged union cannot safely infer active reference lifetime without explicit unsafe control.

## Option of reference

An Option containing a reference is bounded by the target lifetime when Some.

None contains no active reference.

Moving Option transfers the stored reference value.

Copying Option[ref T] may copy the shared reference.

Option[ref mut T] is move-only under mutable-reference rules.

## Result containing references

Result may carry references in either variant.

The resulting value's lifetime constraints depend on the active payload.

For exported signatures, the compiler must represent all possible source relationships conservatively.

Pattern matching preserves the payload's reference lifetime.

## Pattern-bound references

Pattern matching may create references to payloads.

Example:

```sec id="iyd61q"
match result {
case Ok(ref value):
    Inspect(value)
case Err(ref error):
    Report(error)
}
```

Each branch borrow is bounded by:

* result's value lifetime
* active variant lifetime
* branch scope
* final reference use

The borrow ends before the variant may change or result may be destroyed.

## Temporary lifetimes

Temporary values require deterministic lifetime boundaries.

A temporary normally lives until:

* ownership is transferred
* its final reference use ends
* the enclosing full expression completes
* an explicitly defined extension applies
* cleanup executes on an early exit

The compiler should choose narrow lifetimes while preserving evaluation and reference validity.

## Borrowing a temporary

Borrowing a temporary requires extending the temporary's value lifetime long enough for the reference's valid use.

Example:

```sec id="xv6kw1"
Inspect(ref CreateValue())
```

The temporary may remain valid through the call.

It is destroyed after the call when the reference does not escape.

The compiler must reject storing or returning the reference beyond the temporary's lifetime.

## Binding a reference to a temporary

Example:

```sec id="jb0swk"
let view := ref CreateValue()
Inspect(view)
```

The language must define whether binding the reference extends the temporary's lifetime.

Recommended initial rule:

* allow temporary lifetime extension to the lifetime of the local reference binding only when the temporary is owned by compiler-generated local storage
* prohibit extension beyond the current lexical scope
* prohibit return or escape
* preserve deterministic destruction

A simpler initial implementation may reject reference bindings to non-named temporaries except call-duration borrows.

## Temporary lifetime extension

If supported, temporary lifetime extension must be explicit in Semantic IR.

The compiler must create hidden owned storage for the temporary.

This storage is not heap allocation.

The temporary is destroyed when the extended local lifetime ends.

The compiler must not extend temporaries silently across function boundaries.

## Expression lifetime

Operands created during an expression remain valid according to Sec's evaluation order.

References created from earlier operands must remain valid through later uses in the same expression.

The compiler must not destroy a temporary before all expression uses complete.

## Full-expression boundary

A full-expression boundary should be defined for:

* initializer completion
* assignment completion
* return expression completion
* expression statement completion
* condition evaluation
* loop clause evaluation
* argument-list completion
* match subject evaluation

Temporaries not transferred or extended are destroyed at the appropriate boundary in reverse creation order.

## Short-circuit expressions

Lifetime analysis must account for short-circuit evaluation.

Example:

```sec id="wp2mt6"
condition && Check(ref value)
```

The borrow exists only on paths where Check executes.

Temporary and borrow states after the expression must merge correctly.

No borrow may be assumed to occur on the short-circuited path.

## If expressions

An if expression may produce a value or reference.

Example:

```sec id="7028br"
let selected :=
    if condition {
        ref first
    } else {
        ref second
    }
```

The selected reference is bounded by both possible target lifetimes.

Branch-local temporaries cannot be referenced after the branch unless their lifetime is validly extended into storage that outlives the branch.

## Branch-local references

A reference to a branch-local value cannot escape the branch.

Example:

```sec id="rjr50v"
let view :=
    if condition {
        let local := CreateValue()
        ref local
    } else {
        ref external
    }
```

The first branch would return a reference to dead local storage.

The entire expression is invalid.

The compiler must diagnose the exact branch causing the invalid lifetime.

## Switch and match lifetimes

Each switch or match arm has independent lifetime state.

At the merge point, the compiler must combine:

* value availability
* reference sources
* borrow states
* active variants
* destruction requirements

A reference result from different branches receives a lifetime no longer than all possible sources.

## Loop iteration lifetimes

Values declared inside a loop iteration normally end before the next iteration begins.

Example:

```sec id="3wfhlo"
for item in items {
    let temporary := Prepare(item)
    Use(temporary)
}
```

Each iteration creates a distinct temporary value lifetime.

A reference to one iteration's temporary cannot be used in a later iteration.

## Loop-carried references

A reference may be carried across iterations only if its target outlives every iteration where it is used.

The compiler must analyze back edges.

Example:

```sec id="s9py9t"
let view := ref persistent

while condition {
    Inspect(view)
}
```

This may be valid when persistent remains valid for the loop duration.

## Borrow created inside loop

A borrow created inside an iteration should normally end after its final use in that iteration.

Example:

```sec id="u4g4mt"
for item in items {
    let view := ref item
    Inspect(view)
}
```

The borrow must not accidentally remain active across the loop back edge.

Non-lexical analysis should end it before the next iteration when no use continues.

## Mutable loop borrows

A mutable borrow must not overlap the next iteration unless explicitly loop-carried.

Example:

```sec id="dzcz8y"
for index in 0..<count {
    let item := ref mut values[index]
    Update(item)
}
```

The borrow may end after Update.

The next iteration may then borrow another element.

If item is stored outside the loop body, the compiler must analyze possible overlap.

## References escaping loops

A reference to an iteration-local value cannot escape through:

* return
* outer variable assignment
* aggregate storage outside the iteration
* closure capture outliving the iteration
* FFI retention

A reference to an element of an outer container may escape only when the container and element storage remain stable.

## Break values

If loops later support break values, a reference carried by break must be valid at the destination.

A reference to iteration-local storage is invalid after the loop body exits.

An owned value may be moved out through break.

The compiler must distinguish ownership transfer from reference escape.

## Continue

continue ends lifetimes of iteration-local values and borrows for scopes it exits.

Loop-carried state remains active.

The compiler must insert cleanup and borrow termination before the continuation edge.

## Function parameter lifetimes

An owned by-value parameter has a value lifetime within the callee.

The caller's ownership ends at successful transfer for move-only types.

The callee's parameter lifetime begins when argument transfer completes.

A copied parameter creates an independent callee value lifetime.

## Reference parameter lifetimes

A reference parameter's target lifetime is provided by the caller.

The callee may use the reference only within the declared or inferred contract.

The callee cannot extend the target lifetime.

The reference parameter value itself normally exists for the call duration unless moved into an allowed returned or stored value.

## Parameter aliasing

Multiple reference parameters may refer to overlapping storage.

Shared references may overlap.

Mutable references must satisfy exclusivity.

Example:

```sec id="67tu5j"
fn UpdateBoth(
    first: ref mut Value
    second: ref mut Value
) void
```

The caller must not pass overlapping targets unless the language explicitly allows and the function contract handles it.

The compiler should detect statically provable overlap.

Unknown unsafe overlap remains prohibited for safe references.

## Lifetime covariance and variance

If Sec later formalizes variance, lifetime relationships must account for it.

The initial implementation should avoid exposing variance terminology in source syntax.

Compiler rules should conservatively ensure that:

* a reference valid for a longer period may be used where a shorter one is required
* a shorter-lived reference cannot be extended into a longer-lived requirement
* mutable references do not gain unsafe substitution through type relationships

A full variance model may be documented separately if generics require it.

## Closure captures

Closures may capture:

* owned values
* shared references
* mutable references
* copyable values
* move-only values

The closure's lifetime is constrained by all borrowed captures.

An owning capture transfers or copies the captured value into the closure.

A borrowed capture does not extend the target lifetime.

## Escaping closures

A closure that escapes its creation scope cannot borrow local values that end with that scope.

Example:

```sec id="7bvx7m"
fn Invalid() fn() void {
    let value := CreateValue()

    return fn() void {
        Use(ref value)
    }
}
```

This is invalid if the closure borrows value.

It may be valid if the closure owns value through capture-by-move and the closure representation owns the captured storage.

Capture mode must be explicit in Semantic IR.

## Non-escaping closures

A closure passed to a function that guarantees non-retention may borrow local values for the call.

Separate compilation requires the callee signature to express whether the closure may escape.

The compiler must not infer non-escape from implementation availability alone for public contracts.

## Interface values containing references

A borrowed interface value must carry lifetime constraints from the concrete referenced object.

Dynamic dispatch must not erase lifetime information.

An owning interface value may own its concrete object independently.

The interface representation must distinguish owning and borrowed forms.

## Generic lifetime inference

Generic code must preserve lifetime relationships without requiring runtime information.

Example:

```sec id="jzo4pt"
fn IdentityRef[T](value: ref T) ref T {
    return value
}
```

The returned reference is tied to the input reference for every T.

Monomorphization preserves the relationship.

The compiler should infer the lifetime relation once at generic definition validation.

## Generic aggregates with references

A generic type may contain references through its type arguments.

Example:

```sec id="s4cx2b"
type Box[T] struct {
    value: T
}
```

If T is `ref Value`, Box carries that reference lifetime.

If T is owned Value, it does not.

Type metadata must preserve lifetime-bearing properties through instantiation.

## Generic constraints involving lifetimes

Most lifetime requirements should be inferred structurally.

A generic function may require that one input outlive another operation or returned value.

The initial implementation should support common structural cases without explicit lifetime syntax.

Complex relationships that cannot be represented safely may be rejected until a deliberate syntax exists.

## Separate compilation

Public module interfaces must contain lifetime relationship metadata.

Required information may include:

* which returned references derive from which parameters
* whether a parameter reference may be retained
* whether a closure may escape
* whether an interface value borrows or owns
* whether an FFI call retains pointers
* whether a returned aggregate contains input-derived references
* receiver lifetime relationships
* mutable exclusivity continuation

This metadata is compiler-facing and need not appear in ordinary source syntax.

## Interface declarations

An interface method without a body cannot rely on body inference.

The method signature must still define sufficient lifetime relationships.

Common relationships may be implicit by structural rule.

Example:

```sec id="pbksvz"
interface Container[T] {
    fn Get(ref self, index: uint) ref T
}
```

The returned reference naturally derives from self.

The compiler may define this as a standard inference rule.

Ambiguous methods may require future explicit relation annotations.

## Extern declarations

Extern functions have no Sec body for lifetime inference.

Their pointer and reference behavior must be declared explicitly.

The compiler must know whether an extern function:

* borrows only for call duration
* retains a pointer
* returns a reference into an input
* returns static storage
* transfers ownership
* invalidates a buffer
* may reallocate storage
* stores a callback

Safe wrappers should encode these rules.

Raw ambiguous extern access belongs in unsafe code.

## FFI pointer retention

Passing RawPtr to foreign code does not create a checked reference lifetime.

The programmer or safe wrapper must uphold validity.

When a safe wrapper passes an address derived from a Sec reference, it must ensure that foreign retention does not exceed the target lifetime.

The compiler may enforce this only when retention metadata exists.

Otherwise, the operation must be unsafe.

## FFI callbacks

A callback passed to foreign code may outlive the call.

The declaration must specify retention.

A callback that captures borrowed local values cannot be retained beyond those values.

A retained callback normally requires owned captured state in explicitly allocated or externally valid storage.

The compiler must not silently heap-promote captures.

## Asynchronous retention

Even without a language-level async runtime, platform APIs may retain pointers or callbacks asynchronously.

Such APIs require explicit lifetime contracts.

A safe Sec wrapper must ensure that:

* target storage remains alive
* ownership remains available
* cancellation and completion end retention
* destruction waits until retention ends
* foreign completion cannot access freed storage

This may require an owning handle type rather than a raw reference.

## Threads

A reference sent to another thread must remain valid for the remote use duration.

Ordinary stack-local references cannot outlive the creating thread's scope.

Thread creation APIs must define whether arguments are:

* copied
* moved
* borrowed for joined lifetime
* required to have static lifetime

The compiler must not assume a thread completes before scope exit unless the API contract guarantees joining.

## Scoped threads

A future scoped-thread API may permit borrowing local values if the compiler knows all threads join before the scope exits.

The scope object would define a lifetime boundary.

Such behavior must be explicit in the API contract and compiler metadata.

No general runtime borrow tracking is required.

## Static lifetime

A value in immutable static storage may support references valid for the program lifetime.

The compiler may classify such references as static.

Programmers should not need to write an explicit `'static`-like lifetime name.

The compiler can represent static lifetime internally.

Mutable static references require synchronization and aliasing rules.

## String literals

String literals normally reside in immutable static storage.

A reference or view to a string literal may be valid for the entire program execution.

The string view value itself may have a shorter lifetime.

Copying the view does not affect literal storage.

## Constant data

Compile-time constant data embedded in the binary may have static storage duration.

The compiler must distinguish:

* pure compile-time value requiring no address
* materialized immutable static data
* local temporary generated from a constant
* target-specific constant storage

Taking a reference may force materialization.

## Address-taken locals

Taking a reference to a local may require materialized storage.

The compiler may no longer keep the value only as a transient SSA scalar if an address is needed.

This affects physical storage, not semantic lifetime.

The compiler must ensure the storage remains valid for every reference use.

## Escape analysis relationship

Lifetime analysis asks whether a reference or value can remain valid for its required uses.

Escape analysis asks whether values or references leave a region or function.

The analyses cooperate but remain distinct.

Escape analysis must not repair an invalid lifetime through hidden allocation.

Lifetime rules remain authoritative.

## Destruction relationship

A value cannot be destroyed while a valid reference to it remains active.

Lifetime analysis constrains destruction placement.

Destruction analysis determines where ownership ends.

The analyses must agree.

If a borrow extends beyond an otherwise possible early destruction point, destruction must be delayed.

If ownership must end earlier, the borrow is invalid.

The compiler must not choose whichever result is easier after the fact.

## Move relationship

Moving a value ends the source value lifetime.

References to the moved source cannot remain valid unless the physical storage and semantic relationship are explicitly preserved by a supported non-relocating ownership-transfer rule.

The initial implementation should reject moves while references remain active.

Moving an aggregate may end complete-value access while references to unmoved disjoint fields remain theoretically possible.

The initial compiler may use conservative whole-value invalidation until field-sensitive support is complete.

## Partial move relationship

A partial move ends the moved field's value lifetime in the source aggregate.

Unmoved fields retain their lifetimes.

References to the moved field become invalid.

References to disjoint unmoved fields may remain valid if:

* the compiler tracks fields independently
* the aggregate has no conflicting invariant
* no whole-value operation occurs
* destruction remains field-sensitive

The initial implementation may reject partial moves while any borrow of the aggregate is active.

## Field-sensitive lifetimes

The compiler should support field-sensitive lifetime analysis for ordinary structs.

This allows:

* borrowing one field while mutating a disjoint field
* moving one field while retaining others
* independently ending field borrows
* precise destruction

Example:

```sec id="tzowbw"
mut pair := Pair.Create()
let first := ref pair.first

pair.second.Update()
Inspect(first)
```

This may be valid when first and second are disjoint fields.

The compiler must remain conservative for unions, packed layouts, volatile registers and opaque foreign layouts.

## Indexed elements

Field sensitivity for dynamically indexed arrays and slices is harder.

The compiler may prove disjointness for:

* distinct compile-time constant indexes
* non-overlapping split operations
* trusted iterator semantics
* range partitioning with proven bounds

Otherwise, mutable borrows of elements may be treated as overlapping.

The initial implementation should prefer safe conservative rejection over complex unsound alias inference.

## Slice lifetimes

A slice is a view over contiguous storage.

Its lifetime cannot exceed:

* backing storage lifetime
* contained element validity
* source borrow lifetime
* any mutation that reallocates or invalidates storage

A slice does not own backing storage unless a distinct owning slice type is defined.

## Container reallocation

A reference or slice into a container becomes invalid if the container reallocates its storage.

Methods that may reallocate must require exclusive access and must not run while element references remain active.

Example:

```sec id="8jb67p"
mut values := Vector.Create()
let first := ref values[0]

values.Append(newValue)

Use(first)
```

This is invalid when Append may reallocate.

The method contract should mark storage invalidation behavior.

## Non-reallocating operations

A container operation known not to relocate storage may preserve element references.

Examples might include:

* reading length
* modifying an element in place
* changing metadata without storage relocation
* appending within guaranteed reserved capacity if the contract expresses this safely

The compiler needs method effect metadata to reason across separate compilation.

The initial implementation may conservatively invalidate mutable-container borrows for any mutating call.

## Method lifetime effects

Method signatures or inferred metadata may describe:

* borrows receiver for call only
* returns reference tied to receiver
* retains reference
* may invalidate element references
* may replace receiver value
* consumes receiver
* preserves address stability
* mutates only specified fields

These effects support precise lifetime analysis.

They must not require runtime tracking.

## Destruction of reference-containing values

Destroying an aggregate containing references ends those reference value lifetimes.

It does not destroy referenced targets.

Any borrow restrictions caused only by those references may end after their final possible use.

The compiler may end them before aggregate destruction when references are dead.

## Copying reference-containing values

Copying shared references may extend the borrow to the last use of any copy.

The compiler must track aliasing reference values.

Moving a mutable reference transfers the exclusive borrow continuation.

The moved source reference becomes unavailable.

## Reference assignment

Assigning one reference value to another binding follows reference copy/move rules.

For shared references:

```sec id="5kxr6g"
let first := ref value
let second := first
```

Both reference values may remain valid.

The underlying shared borrow remains active until the final use of all copies.

For mutable references, assignment moves the reference.

## Lifetime joins

At a control-flow merge, lifetime state must be joined conservatively.

For each reference or borrow, the compiler must determine:

* whether it exists on all paths
* possible target origins
* latest required use
* active permissions
* conflicting states
* whether the target remains valid on all paths

A reference available on only some paths cannot be used after the merge without an explicit variant or optional representation.

## Lifetime intersections

When a reference may target one of several values, its valid lifetime is the intersection of all possible target lifetimes.

The compiler may preserve more precise path-dependent information internally.

Caller-visible or merge-visible safety must use a lifetime valid for every reachable possibility.

## Lifetime widening

The compiler must never widen a reference lifetime beyond what all constraints permit.

Optimizations may keep storage physically alive longer.

That does not permit additional source-level reference uses.

Semantic validity is determined before storage optimization.

## Lifetime shortening

The compiler may shorten internal storage or borrow lifetimes after final use.

Shortening is valid only when it does not change:

* destructor order
* defer behavior
* address observability
* volatile operations
* debug guarantees explicitly promised by the language
* FFI retention
* concurrency visibility

## Dead references

A reference value with no future use does not require runtime destruction.

Its borrow restrictions may end after its final semantic use.

The compiler need not wait for lexical scope exit.

This is central to non-lexical lifetime behavior.

## Last-use analysis

Last-use analysis identifies the final reachable use of a reference or borrowed value on each path.

The compiler must account for:

* branches
* loops
* calls
* closures
* stores
* returns
* interface conversion
* aggregate construction
* potential retention
* defer captures

A syntactically later statement does not extend a borrow unless it can use the reference.

## Defer and lifetimes

A reference captured by defer remains potentially used until the defer executes.

Example:

```sec id="ab21o4"
let view := ref value

defer Inspect(view)

value.Update()
```

The shared borrow remains active until scope cleanup.

The mutation is invalid.

The compiler must treat defer captures as delayed uses.

## Owned defer captures

If defer captures an owned copy or moved value, lifetime behavior follows capture semantics.

A copied scalar capture does not keep the original borrowed.

A moved owned capture transfers destruction responsibility into the defer closure or cleanup entry.

Capture mode must be explicit in Semantic IR.

## Defer ordering

Borrow lifetimes used by defer must remain valid through the defer's execution point.

Destruction ordering must ensure that referenced values are not destroyed before deferred uses.

The unified cleanup-stack model naturally supports this when registration order is correct.

The compiler verifier must still confirm validity.

## Error propagation

A reference used only on the success path may end before error cleanup.

A reference captured into an error value or returned error payload must remain valid according to the returned type.

Error propagation cannot return references to locals that are destroyed during cleanup.

The compiler must analyze both Ok and Err paths.

## Result propagation and borrows

Example:

```sec id="9snlwa"
let view := ref value
let parsed := try Parse(view)
value.Update()
```

If Parse does not retain or return a reference tied to view, the borrow may end after the call succeeds.

On Err, cleanup occurs and the function returns.

If parsed contains a reference derived from view, the borrow remains active according to parsed's lifetime.

## Panic and abnormal termination

Until unwinding semantics are defined, lifetime analysis applies to normal reachable execution paths.

An aborting panic does not create a valid continuation after termination.

The compiler must not rely on cleanup after abnormal termination unless guaranteed.

References need not remain valid after program termination.

## Unreachable code

Unreachable code does not extend lifetimes.

The compiler may diagnose unreachable uses separately.

Ownership and borrow analysis should operate on reachable control flow.

However, the parser and type checker may still validate unreachable syntax according to language policy.

## Unsafe and lifetimes

Unsafe code may perform operations the compiler cannot fully verify.

Unsafe does not suspend all lifetime rules automatically.

Safe references created or returned from unsafe code must still satisfy safe lifetime guarantees.

An unsafe block may use RawPtr beyond checked reference lifetimes only when the programmer upholds validity.

The compiler must not infer safety from unsafe intent.

## RawPtr lifetimes

RawPtr does not carry a checked lifetime.

The compiler does not guarantee:

* target existence
* target initialization
* alignment
* aliasing
* ownership
* mutability
* provenance
* thread safety

Copying RawPtr does not extend target lifetime.

Converting a reference to RawPtr may allow the pointer bits to outlive the reference value, but dereferencing later is safe only if the target remains valid under unsafe rules.

## RawPtr to reference conversion

Creating a safe reference from RawPtr requires proving or asserting:

* target storage exists
* target value is initialized
* alignment is correct
* lifetime is sufficient
* mutability permissions are valid
* aliasing rules are satisfied
* target type representation is valid

This conversion must occur in unsafe context unless a trusted abstraction establishes the proof.

The resulting safe reference receives a compiler-tracked lifetime bounded by the proof context.

## Pointer provenance

Lifetime analysis should preserve pointer origin information where available.

Provenance may help determine:

* allocation identity
* target storage
* bounds
* deallocation compatibility
* reference conversion validity

LLVM pointer representation must not erase Sec-level safety assumptions before relevant checks and transformations complete.

A full provenance model may be defined separately.

## Volatile references

A volatile or addressed reference remains constrained by its lexical access declaration and platform rules.

External hardware storage may physically persist indefinitely.

This does not permit safe references to be retained arbitrarily when:

* register access semantics require immediate volatile operations
* the device may change state asynchronously
* the address mapping is target-specific
* the binding goes out of scope
* the reference abstraction forbids retention

Register access should generally use dedicated typed access operations rather than ordinary long-lived references.

## Interrupt contexts

References used across interrupt boundaries require concurrency and interrupt safety rules.

A local stack reference cannot be assumed valid in an interrupt handler unless the handler executes within a guaranteed stack and lifetime arrangement.

Global or static storage may be referenced with appropriate synchronization and volatile semantics.

This requires a separate interrupts specification.

## Atomics and lifetimes

Atomic access can synchronize memory visibility.

It does not by itself extend object lifetime.

A reference loaded atomically is not automatically safe if the target may be destroyed concurrently.

Shared ownership or hazard management requires explicit types and protocols.

The compiler must not equate atomic pointer validity with lifetime validity.

## Concurrency memory model

The concurrency memory model determines when operations are ordered across threads.

Lifetime analysis determines whether referenced storage still exists.

Both are required.

A data race rule does not prevent use-after-free.

A lifetime guarantee does not prevent a data race.

The analyses must remain conceptually separate.

## Region inference

The compiler may internally model lifetimes as regions.

A region may correspond to:

* function invocation
* lexical scope
* expression
* loop iteration
* borrow live range
* temporary storage
* static storage
* caller-provided input lifetime
* owned allocation

Region names need not appear in source code.

## Region constraints

Lifetime inference may generate constraints such as:

* reference R must not outlive target T
* returned reference R derives from parameter P
* borrow B covers all uses of reference R
* mutable borrow B excludes overlapping borrow C
* value V must outlive deferred use D
* closure C must not outlive captured reference R
* aggregate A must not outlive reference field F

The solver must find a valid conservative assignment or reject the program.

## Constraint direction

The compiler must carefully distinguish:

* target outlives reference
* source reference outlives reborrow
* aggregate lifetime is bounded by fields
* returned reference is bounded by input
* owner must remain alive through active borrow

Incorrect constraint direction can make the analysis unsound.

Compiler tests should include focused constraint cases.

## Lifetime graph

The compiler may represent lifetime relationships as a directed constraint graph.

Nodes may include:

* value lifetimes
* storage lifetimes
* reference lifetimes
* borrow regions
* function parameters
* return relationships
* aggregate instances
* closure captures
* static regions

Edges encode outlives or containment constraints.

Cycles are not automatically invalid.

They may represent equivalent regions.

Cycles implying impossible local escape must be rejected.

## Constraint solving

A possible solving strategy is:

1. build control-flow graph
2. determine initialization and move states
3. collect reference origins
4. collect borrow uses
5. compute live ranges
6. create lifetime constraints
7. propagate outlives requirements
8. detect conflicts
9. shorten borrows to final use
10. validate destruction and escape
11. record resolved lifetime metadata

The exact implementation may differ.

The result must be deterministic.

## Dominance

Reference creation normally dominates every use of that reference.

A use reachable without reference initialization is invalid.

Dominance alone does not prove target lifetime.

The compiler must also prove:

* target remains initialized
* target was not moved
* borrow permissions remain active
* target storage remains alive

## Post-dominance

Destruction or move may post-dominate some uses.

Post-dominance can help identify lifetime ends and cleanup.

It does not replace path-sensitive ownership analysis.

A value may end through different terminal actions on different branches.

## Data-flow state

Lifetime analysis may maintain per-program-point state including:

* initialized values
* active owners
* moved values
* partially moved fields
* active shared borrows
* active mutable borrows
* reference origins
* reference live uses
* storage validity
* active variants
* pending defer uses

The state must merge conservatively at control-flow joins.

## Fixed-point analysis

Loops require iterative data-flow analysis until lifetime and borrow states reach a fixed point.

The compiler must detect invalid states such as:

* a borrow carried indefinitely without a valid target
* repeated mutable borrow overlap
* move on one iteration and use on the next
* reference to iteration-local storage crossing the back edge

The analysis must terminate deterministically.

## Field paths

Borrow and lifetime analysis should identify places using field paths.

Examples:

```text id="kr749c"
value
value.first
value.second
value.items[constant-index]
value.variant.payload
```

Disjoint field paths may permit independent borrows.

Unknown index paths should conservatively overlap.

Union payload paths overlap by representation unless active-variant rules prove exclusivity.

## Place lifetime

A place is an addressable storage path.

The lifetime of a place depends on:

* root storage
* active aggregate value
* initialized field chain
* active union variant
* index validity
* storage relocation
* ownership state

A reference to a place is valid only while the entire path remains valid.

## Dereference paths

A place may include dereference through:

* shared reference
* mutable reference
* RawPtr
* owning pointer abstraction

Each dereference contributes different lifetime and safety guarantees.

RawPtr dereference enters unsafe semantics.

Reference dereference inherits the reference's tracked lifetime.

Owning pointer dereference is bounded by owner lifetime and borrow rules.

## Method calls and place validity

A method call may invalidate places within its receiver.

Example:

```sec id="h5dbj8"
let view := ref vector[0]
vector.Reserve(100)
Use(view)
```

If Reserve may reallocate, view becomes invalid.

The compiler needs effect metadata or a conservative rule.

A method receiving only `ref self` cannot mutate or reallocate through safe semantics.

A method receiving `ref mut self` may invalidate internal places unless declared more precisely.

## Lifetime-preserving methods

The compiler may infer or record that a method preserves certain references.

Examples:

* read-only methods
* mutation of disjoint metadata
* in-place updates without relocation
* methods returning references tied to self

Such precision is an optimization and usability improvement.

The default safe rule for unknown mutable effects should remain conservative.

## Destructive methods

A consuming receiver method ends the source ownership lifetime.

References into the receiver must not remain active across the call.

A method that consumes and returns the same type creates a new ownership continuation.

The compiler must not assume internal addresses remain stable unless guaranteed.

## ABI lifetime behavior

The physical ABI may copy addresses and values.

It does not define reference validity.

Lifetime metadata must remain available across Sec module boundaries.

C ABI calls require wrapper declarations because C types alone do not express Sec lifetimes.

## Sec ABI metadata

Sec ABI metadata may encode:

* borrowed parameter
* consuming parameter
* mutable exclusive parameter
* returned reference source
* retention behavior
* receiver relation
* reference-containing return
* storage invalidation effect
* callback escape behavior

This metadata supports safe separate compilation.

It may be omitted from final machine ABI where not needed at runtime.

## MLIR representation

MLIR lowering must preserve resolved lifetime information long enough to support:

* stack allocation placement
* lifetime markers
* alias analysis
* borrow-aware optimization
* destruction placement
* address-stability constraints
* cleanup lowering
* reference verification

Lifetime semantics may be represented through:

* Sec dialect operations
* attributes
* region arguments
* memory effects
* ownership interfaces
* custom verification

## Lifetime erasure

Lifetime metadata may be erased only after all operations depending on it are resolved.

Before erasure, the compiler must have fixed:

* destruction points
* storage allocation
* reference validity
* escape legality
* address stability
* cleanup control flow
* ABI ownership relationships

LLVM must not be expected to reconstruct Sec lifetimes.

## LLVM lifetime intrinsics

LLVM lifetime intrinsics are optional optimization hints.

They may represent storage availability intervals.

They must not be treated as enforcement of Sec lifetime safety.

The compiler must ensure:

* lifetime.start follows storage allocation and precedes valid access
* lifetime.end follows semantic destruction
* no valid reference use occurs after lifetime.end
* storage reuse does not violate address observability
* multiple value lifetimes in one slot use correct markers

## Stack-slot reuse

The compiler may reuse one stack slot for non-overlapping value lifetimes.

This requires proof that:

* the previous value lifetime ended
* destruction completed
* no valid references remain
* no RawPtr observable through safe guarantees requires the old bytes
* alignment and size are sufficient
* debug information remains coherent enough for the selected profile

Stack-slot reuse does not merge semantic values.

## Alias analysis

Resolved reference and borrow information may improve alias analysis.

The compiler may infer:

* mutable reference has exclusive access within its active region
* disjoint field references do not alias
* shared references do not permit mutation through safe paths
* RawPtr may alias conservatively
* FFI calls may invalidate assumptions unless annotated

Alias optimization must not outlive the lifetime proof supporting it.

## noalias lowering

A mutable reference may sometimes lower with noalias-like assumptions.

This is valid only if Sec's exclusivity guarantee matches the backend attribute semantics for the entire attributed region.

The compiler must be cautious with:

* reborrows
* FFI
* callbacks
* interior mutable abstractions
* unsafe aliases
* address escape
* partial fields

Incorrect noalias is a miscompilation.

The initial implementation should emit such attributes only when strongly proven.

## Dereferenceable attributes

Backend attributes indicating pointer validity or dereferenceable size must match the proven reference lifetime and bounds.

They must not claim validity beyond:

* actual target size
* active value lifetime
* reference lifetime
* allocation lifetime

Attributes attached too broadly may allow invalid optimization.

## Readonly attributes

A shared reference may support readonly assumptions through safe code.

However, mutation may still occur through:

* explicitly defined interior mutability
* volatile hardware
* foreign code
* concurrent synchronization
* unsafe RawPtr aliases

The compiler must account for the type and call contract before emitting readonly attributes.

## Runtime-free lifetime enforcement

Sec lifetime correctness should require no runtime tracking.

The compiler generates no default:

* borrow counters
* lifetime tables
* reference registry
* tracing garbage collector
* hidden reference counting
* runtime dangling checks

Optional debug instrumentation may be added separately.

Static validation remains the language guarantee.

## Debug instrumentation

A debug profile may optionally add checks for:

* use after free through unsafe paths
* stack reference escape
* generation counters in selected containers
* invalidated slice use
* double release
* stale external handles

Such checks are supplementary.

They must not become required for correct safe Sec programs.

* return the owner together with the reference-bearing value
* move the value into an owning closure
* shorten the reference use
* create a narrower scope
* borrow a field instead of the entire aggregate
* use an owning container instead of a slice
* make an FFI wrapper own retained storage

Suggestions must not recommend hidden heap allocation as an automatic fix.

## Compiler verification

The lifetime verifier must check:

* every reference has a valid origin
* every reference use occurs within target validity
* every reborrow is bounded by its source
* returned references have representable source relationships
* no local reference escapes invalidly
* no destruction occurs during an active incompatible borrow
* no move occurs during an active incompatible borrow
* no replacement invalidates a live reference
* branch merges preserve valid lifetime state
* loop back edges preserve target validity
* closure captures do not outlive borrowed targets
* aggregate reference fields do not outlive targets
* FFI lifetime contracts are satisfied at safe call sites
* lifetime metadata remains consistent with ownership and destruction
* backend lifetime attributes do not exceed semantic proof

Verifier failure after successful semantic analysis is an internal compiler error.

## Analysis ordering

Recommended ordering is:

1. symbol resolution
2. type resolution
3. place and storage classification
4. control-flow graph construction
5. definite initialization
6. ownership and move analysis
7. preliminary borrow analysis
8. reference-origin analysis
9. escape analysis
10. lifetime constraint generation
11. lifetime solving
12. final borrow conflict validation
13. destruction placement
14. lifetime verification
15. Semantic IR emission or finalization

Some implementations may iterate between:

* borrow analysis
* lifetime analysis
* escape analysis
* destruction analysis

The final result must reach a stable verified state.

## Relationship to ownership analysis

Ownership analysis answers:

* who owns the value
* whether ownership moved
* who must destroy it

Lifetime analysis answers:

* how long the value and its references remain valid
* whether target validity covers every reference use

A valid owner does not automatically imply every reference is valid.

A valid reference does not imply ownership.

The analyses must exchange information but remain conceptually distinct.

## Relationship to borrow analysis

Borrow analysis answers:

* what access permissions exist
* which accesses conflict
* whether exclusivity is preserved

Lifetime analysis determines how long those borrow restrictions remain active.

A borrow conflict depends on overlapping lifetime regions.

The compiler should therefore derive borrow durations from actual reference use.

## Relationship to escape analysis

Escape analysis identifies values and references leaving their defining region.

Lifetime analysis determines whether such escape is legal.

An escape may be:

* valid ownership transfer
* valid returned borrow
* invalid dangling reference
* unsafe RawPtr escape
* FFI retention
* closure capture

Escape analysis must classify the form, not merely mark a boolean.

## Relationship to layout

Layout usually does not determine semantic lifetime.

However, layout affects:

* field overlap
* union storage
* packed fields
* addressability
* alignment
* stable storage
* container relocation

Lifetime analysis must consult layout-related facts where disjointness or address stability depends on them.

## Relationship to target profiles

Target profiles may restrict:

* heap allocation
* recursion
* threads
* callbacks
* global constructors
* dynamic interfaces
* debug instrumentation

Core lifetime validity must remain target-independent.

A program valid on one target must not become lifetime-unsafe on another.

A target may reject unsupported storage mechanisms.

## Incremental compilation

Lifetime analysis results may be cached for unchanged functions and types.

Cache keys must include relevant semantic dependencies:

* function body
* parameter modes
* returned reference relationships
* type ownership classification
* field layout affecting disjointness
* method invalidation effects
* generic constraints
* imported interface metadata
* target-independent semantic options

Target-specific lowering metadata should be cached separately.

## Determinism

Lifetime inference must be deterministic.

The same source and compiler configuration must produce:

* the same accepted or rejected result
* the same lifetime relationships
* the same diagnostics ordering
* equivalent Semantic IR

Hash-map iteration order or parallel analysis must not affect results.

## MLIR tests

* lifetime information retained through cleanup lowering
* no invalid relocation of borrowed storage

## Initial implementation policy

The initial compiler should use conservative rules:

* lifetimes are inferred
* no user-written lifetime names
* references to locals cannot escape their valid scope
* no hidden stack-to-heap promotion
* returned references must derive from reference inputs or static storage
* ambiguous returned-reference relationships use conservative intersection or are rejected
* call parameters borrow only for the call unless signature metadata states retention
* mutable methods may invalidate internal references conservatively
* moves and replacement are prohibited while any overlapping borrow remains active
* partial-move and field-borrow interaction is conservative
* dynamic-index mutable borrows are assumed overlapping unless proven disjoint
* self-referential safe values are rejected
* RawPtr has no checked lifetime
* FFI retention requires explicit metadata or unsafe code
* non-lexical borrow ending is supported
* lifetime correctness requires no runtime

## Future extensions

Possible future extensions include:

* explicit lifetime relation annotations for ambiguous declarations
* stable-address types
* pinning
* scoped threads
* async task lifetimes
* generators
* richer container invalidation effects
* precise index-disjointness analysis
* region-based allocation
* arena-bounded references
* retained callback abstractions
* safe self-referential structures
* lifetime-aware interface existential types

These features must extend the inferred model rather than forcing routine code to declare explicit lifetimes.

## Final invariants

Before Semantic IR verification completes:

* every reference has a known origin
* every reference has a bounded validity region
* every reference use lies within that region
* every borrow has a known active region
* every mutable borrow is exclusive
* every returned reference has a caller-visible lifetime relation
* every escaped reference is valid at its destination
* every move and destruction respects active borrows
* every replacement ends old references correctly
* every reference-containing aggregate is lifetime-bounded
* every closure capture is valid for the closure lifetime
* every FFI reference contract is satisfied
* every control-flow merge has compatible lifetime state

Before MLIR lifetime metadata is erased:

* destruction placement is fixed
* stack and heap storage decisions are valid
* no hidden lifetime repair occurred
* address-stable values are not relocated
* all lifetime-dependent alias assumptions are recorded safely
* all cleanup paths preserve reference validity
* backend attributes do not exceed semantic guarantees

Before LLVM translation:

* no unresolved lifetime constraint remains
* all reference validity has been statically verified
* all borrow conflicts are resolved
* all lifetime-sensitive control flow is explicit
* storage lifetime markers are correct
* ownership and destruction agree with lifetime results
* no LLVM optimization is expected to enforce Sec safety

## Design rule

Storage may exist without a live value.

A value may exist without being borrowed.

A reference may exist only while its target remains valid.

A borrow remains active only as long as required by valid reference use.

Ownership determines responsibility.

Lifetime analysis determines validity.

The compiler infers both and rejects any program it cannot prove safe.
