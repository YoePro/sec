Enums

An enum defines a finite set of named values. Each enum is a distinct type, even when two enums use the same underlying integer representation. 

enum Direction {
    north,
    east,
    south,
    west,
}

The variants Direction.north, Direction.east, Direction.south and Direction.west are values of type Direction. They are not plain integers.

Rule: Enum values retain their enum type. Conversion to or from an integer is always explicit.

Basic Syntax

An enum declaration contains a name and a comma-separated list of variants:

enum Color {
    red,
    green,
    blue,
}

A trailing comma is allowed and is the normal formatted style.

An enum may specify an underlying integer type after its name:

enum Status int {
    unknown = 0,
    active = 10,
    paused,
    disabled = 99,
}

If no underlying type is written, int is used.

Variants and Values

The enum name identifies the type:

Direction

A qualified variant name identifies a value of that type:

Direction.north
let direction: Direction := Direction.north

The numeric representation belongs to the variant, but it does not replace the enum type. For example, Direction.north may store the numeric value 0, while still having type Direction.

Underlying Types

The optional underlying type must be an integer type.

enum SmallCode uint8 {
    none,
    ready,
    failed,
}

Examples of valid underlying types include:

int
int8
int16
int32
int64
int128
int256
uint
uint8
uint16
uint32
uint64

The exact set depends on the integer types supported by the target compiler version.

A non-integer underlying type is invalid:

enum Invalid string {
    value,
}
The underlying type of an enum must be an integer type.

Every resolved variant value must be representable by the underlying type. Overflow is a compile-time error.

Automatic Values

The first variant receives numeric value 0 unless it has an explicit initializer.

Each later variant without an initializer receives the previous numeric value plus one.

enum Direction {
    north,
    east,
    south,
    west,
}
Variant Numeric value
Direction.north0
Direction.east1
Direction.south2
Direction.west3

Automatic continuation follows the previous resolved numeric value:

enum Status int {
    unknown = 0,
    active = 10,
    paused,
    disabled = 99,
}
Variant Numeric value
Status.unknown0
Status.active10
Status.paused11
Status.disabled99

Explicit Values

A variant initializer must be an integer constant expression.

enum HttpStatus int {
    ok = 200,
    notFound = 404,
    internalError = 500,
}

Integer constant operators may be used:

enum Flags uint {
    read = 1 << 0,
    write = 1 << 1,
    execute = 1 << 2,
}

A runtime value, function call or non-integer expression cannot initialize an enum variant.

enum InvalidInitializer {
    value = "invalid",
}

iota

iota is a compile-time integer constant available only while an enum variant initializer is being evaluated.

It starts at 0 for the first variant and increases by one for each following variant. Its value is based on the variant position, not on the previous numeric result. 

enum Permission uint {
    none = 0,
    read = 1 << iota,
    write = 1 << iota,
    execute = 1 << iota,
}
Variant iota Numeric value
Permission.none00
Permission.read12
Permission.write24
Permission.execute38

Explicit values do not reset iota:

enum Example int {
    first = iota,
    second = 100,
    third = iota,
    fourth,
}
Variant iota Numeric value
Example.first00
Example.second1100
Example.third22
Example.fourth33

Example.fourth uses the previous numeric value plus one. It does not repeat the initializer expression from Example.third.

Each enum has its own independent iota sequence.

let value: int := iota
iota is not visible outside enum variant initializers.

Aliases

Several variant names may have the same numeric value.

enum ResultCode int {
    success = 0,
    ok = 0,
    notFound = 404,
    missing = 404,
}

ResultCode.success and ResultCode.ok are both valid variant names. They have the same numeric representation. The same applies to notFound and missing.

Rule: Duplicate numeric values are valid aliases. Duplicate variant names are invalid.

Alias names remain separate declared variants even though equality compares their shared enum value.

Enum Namespaces

Module-level enum variants are accessed through the enum name:

let color := Color.red
let state := Status.active

Variant names do not enter the surrounding module namespace directly. This allows different enums to reuse clear names such as none, ready or failed. 

enum ConnectionState {
    ready,
    failed,
}

enum JobState {
    ready,
    failed,
}
let connection := ConnectionState.ready
let job := JobState.ready

Nested Enums

An enum may be declared inside an impl block:

type Vehicle struct {
    Name: string
}

impl Vehicle {
    enum FuelType {
        petrol,
        diesel,
        electric,
    }

    fn IsElectric(fuel: FuelType) bool {
        return fuel == FuelType.electric
    }
}

Outside the owner implementation, the enum type is qualified with the owner type:

let fuel: Vehicle.FuelType := Vehicle.FuelType.electric

Inside impl Vehicle, the owner prefix is omitted:

FuelType
FuelType.petrol

The full nested enum name is Vehicle.FuelType. Its variants belong to the nested enum namespace.

Variant Name Uniqueness

Variant names must be unique inside one enum.

enum Invalid {
    value,
    value,
}
Duplicate enum variant name value.

Different enums may use the same variant names because each enum provides its own namespace.

Enums Are Distinct Types

An enum does not implicitly become its underlying integer type.

enum Color int {
    red,
    green,
    blue,
}

let color: Color := Color.red

The following assignments are invalid:

let color: Color := 1
let number: int := Color.red

Different enum types are also incompatible:

enum Color {
    red,
}

enum Status {
    active,
}

let color: Color := Status.active
Rule: Sharing an underlying type does not make two enum types compatible.

Explicit Conversions

Enum to Integer

Use normal cast syntax to obtain the numeric representation:

let numeric: int := int(Color.red)

Integer to Enum

Use the enum type as an explicit conversion:

let color: Color := Color(2)

Integer-to-enum conversion may produce a valid value of the enum type even when no declared variant has that numeric representation. 

let unknownColor: Color := Color(200)

This value has type Color, but it does not correspond to a declared variant name. Code that accepts externally supplied enum numbers should account for this possibility.

Between Enum Types

Conversion between different enum types must go through an integer type:

let status: Status := Status(int(color))

There is no implicit conversion between enums.

Comparisons

Equality and inequality are supported between values of the same enum type:

if color == Color.red {
    PaintRed()
}

if color != Color.blue {
    Continue()
}

Comparing different enum types is invalid:

let same := Color.red == Status.active

Comparing an enum directly with an integer is also invalid:

let same := Color.red == 0

Use an explicit conversion when comparing numeric representations:

let same := int(Color.red) == 0

Arithmetic

Arithmetic is not directly defined for enum values.

let next := Color.red + 1
Arithmetic operands cannot use an enum value directly.

Convert explicitly when numeric arithmetic is intended:

let nextNumeric: int := int(Color.red) + 1
let nextColor: Color := Color(nextNumeric)

The explicit conversions make it clear that the program is operating on the numeric representation rather than on enum variants as semantic values.

Variables and Assignment

An immutable enum variable must be initialized when declared:

let color: Color := Color.red

An immutable declaration without an initializer is invalid:

let color: Color

A mutable variable may be declared before it is assigned:

let mut color: Color
color = Color.green

Every assignment must still use a compatible value of the same enum type.

Definite Assignment

An uninitialized mutable enum variable may not be read before every continuing path assigns it.

fn SelectColor(useRed: bool) Color {
    let mut color: Color

    if useRed {
        color = Color.red
    } else {
        color = Color.blue
    }

    return color
}

This is valid because both branches assign color.

fn Invalid(useRed: bool) Color {
    let mut color: Color

    if useRed {
        color = Color.red
    }

    return color
}

This is invalid because the condition may be false and leave color unassigned.

switch and match

Enums may be used with both switch and match.

switch

Use switch for ordinary value-based branching:

switch direction {
case Direction.north:
    MoveNorth()

case Direction.south:
    MoveSouth()

default:
    Stop()
}

An enum switch is not required to be exhaustive.

match

Use match when every declared enum variant must be handled:

match direction {
    Direction.north => MoveNorth()
    Direction.east => MoveEast()
    Direction.south => MoveSouth()
    Direction.west => MoveWest()
}

An enum match is exhaustive unless an allowed catch-all branch handles the remaining variants.

Common Errors

Duplicate Variant Name

enum Invalid {
    value,
    value,
}

Invalid Underlying Type

enum Invalid string {
    value,
}

Invalid Initializer

enum Invalid {
    value = "invalid",
}

Implicit Integer Conversion

let color: Color := 1
let number: int := Color.red

Mixed Enum Types

let color: Color := Status.active
let same := Color.red == Status.active

Arithmetic Without Conversion

let value := Color.red + 1

iota Outside an Enum

let value: int := iota

Summary of Rules

  • An enum defines a distinct named type and a finite set of named variants.
  • Variants are comma-separated and may use a trailing comma.
  • The default underlying type is int.
  • An explicit underlying type must be an integer type.
  • The first automatic value is 0.
  • Later automatic values use the previous numeric value plus one.
  • Explicit initializers must be representable integer constant expressions.
  • iota is local to enum initializers and follows variant position.
  • Variant names must be unique inside an enum.
  • Duplicate numeric values are allowed as aliases.
  • Enum variants have the enum type, not the underlying integer type.
  • Integer and enum conversions are explicit in both directions.
  • Different enum types are never implicitly compatible.
  • Equality and inequality require the same enum type.
  • Arithmetic requires explicit conversion to an integer type.
  • Nested enums use the owner-qualified name outside their impl.
  • Normal initialization and definite-assignment rules apply to enum variables.
